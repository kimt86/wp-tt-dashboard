클라우드 용량 산정 — CPU · 메모리

플랫폼(대시보드·실시간 수집·KPI)과 배차 로직(1·2단계)을 클라우드에 띄울 때 필요한 vCPU·메모리·스토리지 추정. ML 학습은 다른 서버에서 돌리므로 별도로 산정했습니다. 핵심 결론부터: 서빙·배차는 가볍고, 무거운 건 학습입니다.

사이징 원칙 — 의도된 여유아래 수치는 의도적으로 보수적(넉넉)하게 잡았고 범위(최소 → 권장 → 넉넉)로 제시합니다. 권장값에 이미 충분한 헤드룸이 들어 있으며, 실측 부하 테스트·모니터링 이후 더 작은 사양으로 다운사이징(right-size down)할 수 있습니다 — 즉 이 버퍼는 의도된 것입니다. 불확실할 때는 권장~넉넉 사이에서 시작하고, 안정화되면 비용을 줄이세요.

핵심실시간 서빙·배차의 부하는 websocket 수집(~90 msg/s)과 대시보드 폴링이 대부분인데 둘 다 가볍습니다(코어 1개 미만). 배차 최적화(헝가리안)·예측 추론도 결정당 수~수십 ms라 가볍습니다. 그럼에도 운영 안정성을 위해 권장값을 넉넉히 잡았습니다. 진짜 자원이 드는 건 ML "학습"이고, 그건 상시가 아니라 배치라서 별도 서버에서 필요할 때만 띄우면 됩니다.

Part 1

구성요소 — 무엇이 자원을 쓰나

① 플랫폼 (상시)

API(axum, websocket 수집 + 메모리 라이브맵 + 대시보드 서빙) · PostgreSQL(서빙 저장소) · 추출기(타이머: 야간·교대·작업풀 90s) · 웹(정적) · SSH 터널.

② 배차 엔진 (상시)

차량 풀·후보 작업 풀을 읽어 매칭/배정(헝가리안 등). 1단계=규칙. 2단계=결정 시 예측 추론(이동시간·ETW·핸드오버) 호출 후 최적화.

③ ML 학습 (배치·별도)

누적 데이터로 2단계 모형을 학습. 상시 아님 — 야간/주간. 다른 서버(CPU 다코어, 필요 시 GPU). 추론과 분리.

①②는 클라우드 상시 가동, ③은 별도 서버에서 배치. 데이터 흐름·KPI 산출은 KPI 산출, 배차 풀은 차량·작업 풀 갱신 참조.

왜 가벼운가장비가 450대뿐이고 메시지도 초당 ~90건입니다. 배차 최적화도 "유휴 트럭 × 후보 작업"의 작은 행렬(수백×수백)이라 헝가리안이 ms 단위로 끝납니다. 대규모 트래픽 서비스가 아니라 한 터미널 규모의 실시간 제어라 코어·메모리 요구가 작습니다.

Part 3

1단계 사이징 — 플랫폼 + 규칙 배차

예측 없이, 차량 풀과 후보 작업 풀을 규칙/최적화로 매칭합니다. 컴포넌트별 (평상시 측정 → 최소 → 권장 → 넉넉, 헤드룸 포함):

컴포넌트vCPU 평상→권장→넉넉RAM 평상→권장→넉넉메모
API (axum + WS 수집 + 라이브맵)0.3 → 1.5 → 2.5150 MB → 1 → 2 GBWS 파싱 + 폴링 서빙 + TT 분류
PostgreSQL0.5 → 2 → 41 → 4 → 8 GB소규모 집계·90s 쓰기 (관리형 분리 권장)

배차 엔진 (규칙·헝가리안)0.3 → 1.5 → 2.5256 MB → 1 → 2 GB결정당 수~수십 ms
웹(정적)·터널·OS 오버헤드~0.1 → 0.5 → 164 MB → 1 → 2 GBAPI가 정적 서빙 + 커널/모니터링 여유

1단계 권장최소 4 vCPU·8 GB → 권장 8 vCPU·16 GB → 넉넉 8 vCPU·16–32 GB 단일 노드로 전부 수용(API+DB+추출기+배차). 운영 안정성을 원하면 PostgreSQL은 관리형(2–4 vCPU·8–16 GB)으로 분리하고 앱 노드는 8 vCPU·16 GB. 예: AWS m6i.2xlarge / GCP e2-standard-8 / Azure D8s_v5 (넉넉히 가려면 m6i.4xlarge / e2-standard-16). 권장값엔 이미 충분한 여유가 있어, 실측 후 4 vCPU·8 GB로 줄일 여지가 큽니다.

Part 4

2단계 사이징 — + 예측 추론

2단계는 배차 결정 때 모형으로 예측(이동시간·크레인 준비 ETW·블록 핸드오버)을 만들어 더 정확히 매칭합니다. 추론은 1단계 위에 얇게 얹힙니다(학습은 별도, §5).

추가분 (배차/추론 노드)vCPU 평상→권장→넉넉RAM 평상→권장→넉넉메모
예측 추론 (배차 엔진 내)+0.5 → 1.5 → 2.5+0.3 → 1 → 2 GB트리모형(LightGBM 등) 추론 µs/건, 결정당 수 ms
피처 조립 (라이브+캐시)+0.3 → 1 → 2+0.2 → 1 → 2 GB쌍별 피처 계산이 추론보다 비쌈
학습데이터 적재 (record-only)+0.2 → 0.5 → 1+0.2 → 0.5 → 1 GB추출기가 매일 스냅샷 누적(§6)
OS·모니터링 여유~0.1 → 1 → 2.5+0.3 → 5.5 → 11 GB버스트·격리 헤드룸
배차/추론 노드 합~1.1 → 4 → 8 vCPU~1 → 8 → 16 GB플랫폼과 분리 권장

왜 추론이 가벼운가2단계 예측은 표 형식(tabular) — 이동시간·ETW 같은 값을 그래디언트 부스팅 트리(LightGBM/XGBoost)로 맞히는 게 표준입니다. 트리 추론은 건당 µs, 모형은 RAM 수십 MB. 결정당 수천 건 예측해도 한 자릿수 ms. GPU 불필요(학습 때만 선택). 노드 사양은 추론 자체보다 운영 격리·버스트 여유를 위해 넉넉히 잡은 것입니다.

2단계 권장플랫폼 8 vCPU·16 GB + 배차/추론 최소 4 vCPU·8 GB → 권장 8 vCPU·16 GB (격리). 합치는 경우 1× 16 vCPU·32 GB로 넉넉히. 예: m6i.2xlarge ×2 → m6i.4xlarge ×1 (16 vCPU·32 GB) · e2-standard-8 ×2 / e2-standard-16 · Azure D8s_v5 ×2 / D16s_v5. 권장값은 여유를 포함하므로 안정화 후 배차 노드를 4 vCPU·8 GB로 다운사이징 가능.

Part 5

ML 학습 — 별도 서버 (배치)

2단계 모형을 누적 데이터로 학습하는 작업. 상시가 아니라 야간/주간 배치라, 서빙 클러스터와 분리된 서버에서 필요할 때만 띄웁니다. 데이터 규모로 사이징합니다.

데이터 규모 (추정)

  • 완료 작업 ~15K move/일 → 6개월 ≈ 2.7M 행 × 피처 30–50개.
  • 학습 행렬 메모리 ≈ 2.7M×50×8B ≈ ~1 GB; 교차검증·피처가공 중간물 포함 4–16 GB. 넉넉히 잡아 64–128 GB면 여러 모형·하이퍼파라미터 탐색을 병렬로 돌릴 수 있음.
  • GPS 원시까지 학습에 쓰면 훨씬 큼(§6) — 보통 move 단위 피처로 집약.
접근vCPU 최소→권장RAM 최소→권장GPU학습시간(1회)
트리모형 (LightGBM/XGBoost) — 권장16 → 3264 → 128 GB—수분 ~ 1시간
딥러닝 (선택: 시퀀스/그래프 모형)8 → 1632 → 64 GB1× T4(16G)/A10(24G)/L4(24G)수 시간
대형 딥러닝 (선택·여유)16 → 3264 → 128 GB1× A100(40–80G)수 시간~

학습 서버 권장ETW·이동시간 예측은 트리모형으로 충분 → 최소 16 vCPU·64 GB → 권장 32 vCPU·128 GB (GPU 불필요). 예: c6i.4xlarge / e2-highmem-16 → c6i.8xlarge / e2-highmem-32 / Azure E16s–E32s. 배치라 spot/preemptible로 비용 절감, 학습 끝나면 종료. 야드 레이아웃을 쓰는 시퀀스/그래프 모형으로 확장하면 GPU 1장(T4 16G / A10 24G / L4 24G — 예: AWS g4dn / g5, GCP는 L4) 추가, 무거운 딥러닝이면 A100 40–80 GB까지 고려. 학습 산출물(모형 파일, 수~수십 MB)만 서빙 노드로 배포합니다. 사양을 넉넉히 잡아도 배치·spot이라 비용 부담이 작습니다.

학습은 데이터 누적이 선행2단계 학습엔 수개월치가 필요합니다. TOS 보존은 ~15/35일뿐이라(레퍼런스 §2), 1단계 운영과 동시에 매일 스냅샷을 적재해 두어야 6–12개월 뒤 학습이 가능합니다(record-only 파이프라인, §6).

스토리지 주의 — GPS 원시가장 큰 스토리지 항목은 학습용 GPS 원시 보관(연 ~200–300 GB)입니다. 대부분의 예측은 move 단위 집약 피처로 충분하므로, 원시는 샘플링/집약하거나 오브젝트 스토리지 콜드 계층(넉넉히 1–2 TB 예산, 수명주기 정책으로 자동 만료)에 두는 걸 권장합니다. 서빙 SSD와 학습 데이터는 모두 자동 확장 가능한 볼륨으로 잡아 두면 안전합니다. 실시간 서빙은 GPS를 메모리에만 두고 DB에 안 쌓습니다(websocket 문서).

Part 7

권장 · 주의

  • 넉넉히 시작, 실측 후 다운사이징. 1단계는 권장 8 vCPU·16 GB 단일 노드(최소 4 vCPU·8 GB)로 출발해 모니터링 후 조정. 본 산정은 의도적으로 보수적(여유 포함)이라, 안정화되면 더 작은 사양으로 right-size down해 비용을 줄이는 방향을 권장.
  • 가용성(HA). 배차가 운영에 직접 관여하면 앱·배차 노드를 이중화(2 AZ)하고 PostgreSQL은 관리형(자동 failover) 권장. 상태는 대부분 Postgres + 메모리 라이브맵(재구성 가능)이라 무상태에 가깝게 운영 가능.
  • 분리 원칙. ① 서빙 ② 배차/추론 ③ 학습을 분리하면 장애·부하가 서로 안 번짐. 특히 학습은 절대 서빙 클러스터에서 돌리지 않음(이 문서의 전제).
  • 확장. 사용자가 수백 명으로 늘면 API를 수평 확장(라이브맵 상태는 공유 캐시/단일 인제스트 + read replica). 현재 규모(내부 수십 명)에선 불필요.
  • 비용 절감. 학습 노드는 spot/preemptible + 배치 종료, GPS 원시는 콜드 스토리지.

전제·한계위 수치는 한 터미널(Westports) 규모·내부 사용자·측정된 메시지율 기준의 추정이며 의도적으로 보수적인(넉넉한) 헤드룸을 포함한 범위로 제시했습니다. 실제 값은 (가) 동시 사용자 수, (나) 배차 결정 주기, (다) 2단계 모형 복잡도, (라) GPS 원시 보관 정책에 따라 달라집니다. 권장값엔 버퍼가 들어 있으니 부하 테스트와 모니터링으로 검증 후 필요하면 다운사이징해 확정하세요.
