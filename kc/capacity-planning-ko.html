클라우드 용량 산정 — CPU · 메모리

플랫폼(대시보드·실시간 수집·KPI)과 배차 로직(1·2단계)을 클라우드에 띄울 때 필요한 vCPU·메모리·스토리지 추정. ML 학습은 다른 서버에서 돌리므로 별도로 산정했습니다. 핵심 결론부터: 서빙·배차는 가볍고, 무거운 건 학습입니다.

기준 2026-06-09 · 사이트 = Westports(장비 ~450대) · 산정 = 측정값 + 헤드룸 · 단위 vCPU / RAM · 학습 분리
한눈에

1단계 플랫폼+규칙배차4 vCPU · 8 GB ×1 (+ 관리형 Postgres 2 vCPU·8GB 선택)m6i.xlarge · e2-standard-4 · D4s_v5단일 노드로 충분
2단계 +예측 추론플랫폼 4 vCPU·8 GB + 배차/추론 4 vCPU·8 GB (또는 합쳐 8 vCPU·16 GB)m6i.xlarge ×2 · e2-standard-8추론은 가벼움(트리모형)

핵심실시간 서빙·배차의 부하는 websocket 수집(~90 msg/s)과 대시보드 폴링이 대부분인데 둘 다 가볍습니다(코어 1개 미만). 배차 최적화(헝가리안)·예측 추론도 결정당 수~수십 ms라 가볍습니다. 진짜 자원이 드는 건 ML "학습"이고, 그건 상시가 아니라 배치라서 별도 서버에서 필요할 때만 띄우면 됩니다.

Part 1

구성요소 — 무엇이 자원을 쓰나

① 플랫폼 (상시)

API(axum, websocket 수집 + 메모리 라이브맵 + 대시보드 서빙) · PostgreSQL(서빙 저장소) · 추출기(타이머: 야간·교대·작업풀 90s) · 웹(정적) · SSH 터널.

② 배차 엔진 (상시)

차량 풀·후보 작업 풀을 읽어 매칭/배정(헝가리안 등). 1단계=규칙. 2단계=결정 시 예측 추론(이동시간·ETW·핸드오버) 호출 후 최적화.

③ ML 학습 (배치·별도)

누적 데이터로 2단계 모형을 학습. 상시 아님 — 야간/주간. 다른 서버(CPU 다코어, 필요 시 GPU). 추론과 분리.

①②는 클라우드 상시 가동, ③은 별도 서버에서 배치. 데이터 흐름·KPI 산출은 KPI 산출, 배차 풀은 차량·작업 풀 갱신 참조.

Part 2

왜 가벼운가장비가 450대뿐이고 메시지도 초당 ~90건입니다. 배차 최적화도 "유휴 트럭 × 후보 작업"의 작은 행렬(수백×수백)이라 헝가리안이 ms 단위로 끝납니다. 대규모 트래픽 서비스가 아니라 한 터미널 규모의 실시간 제어라 코어·메모리 요구가 작습니다.

Part 3

1단계 사이징 — 플랫폼 + 규칙 배차

예측 없이, 차량 풀과 후보 작업 풀을 규칙/최적화로 매칭합니다. 컴포넌트별 (평상시 → 권장 헤드룸):

컴포넌트vCPU 평상/권장RAM 평상/권장메모

PostgreSQL0.5 / 1.01 GB / 2 GB소규모 집계·90s 쓰기
추출기 (타이머)~0.05 / 1.0*100 MB / 256 MB*실행 순간만 1코어, 평균 무시

1단계 권장1× (4 vCPU · 8 GB) 단일 노드로 전부 수용(API+DB+추출기+배차). 운영 안정성을 원하면 PostgreSQL은 관리형(2 vCPU·8 GB)으로 분리하고 앱 노드는 2–4 vCPU·4–8 GB. 예: AWS m6i.xlarge / GCP e2-standard-4 / Azure D4s_v5.

Part 4

2단계 사이징 — + 예측 추론

2단계는 배차 결정 때 모형으로 예측(이동시간·크레인 준비 ETW·블록 핸드오버)을 만들어 더 정확히 매칭합니다. 추론은 1단계 위에 얇게 얹힙니다(학습은 별도, §5).

추가분vCPU 추가RAM 추가메모
예측 추론 (배차 엔진 내)+0.5 ~ 1.0+100 ~ 300 MB트리모형(LightGBM 등) 추론 µs/건, 결정당 수 ms
피처 조립 (라이브+캐시)+0.3+200 MB쌍별 피처 계산이 추론보다 비쌈
학습데이터 적재 (record-only)+0.2+200 MB추출기가 매일 스냅샷 누적(§6)
배차/추론 노드 합~2 / 4 vCPU~2 / 4 GB플랫폼과 분리 권장

왜 추론이 가벼운가2단계 예측은 표 형식(tabular) — 이동시간·ETW 같은 값을 그래디언트 부스팅 트리(LightGBM/XGBoost)로 맞히는 게 표준입니다. 트리 추론은 건당 µs, 모형은 RAM 수십 MB. 결정당 수천 건 예측해도 한 자릿수 ms. GPU 불필요(학습 때만 선택).

2단계 권장플랫폼 4 vCPU·8 GB + 배차/추론 4 vCPU·8 GB (격리). 소규모면 합쳐 1× 8 vCPU·16 GB도 가능. 예: m6i.xlarge ×2 또는 m6i.2xlarge ×1.

Part 5

ML 학습 — 별도 서버 (배치)

2단계 모형을 누적 데이터로 학습하는 작업. 상시가 아니라 야간/주간 배치라, 서빙 클러스터와 분리된 서버에서 필요할 때만 띄웁니다. 데이터 규모로 사이징합니다.

데이터 규모 (추정)

  • 완료 작업 ~15K move/일 → 6개월 ≈ 2.7M 행 × 피처 30–50개.
  • 학습 행렬 메모리 ≈ 2.7M×50×8B ≈ ~1 GB; 교차검증·피처가공 중간물 포함 4–16 GB.
  • GPS 원시까지 학습에 쓰면 훨씬 큼(§6) — 보통 move 단위 피처로 집약.
접근vCPURAMGPU학습시간(1회)
트리모형 (LightGBM/XGBoost) — 권장1664 GB—수분 ~ 1시간
딥러닝 (선택: 시퀀스/그래프 모형)832 GB1× T4(16G)/A10(24G)수 시간

학습 서버 권장ETW·이동시간 예측은 트리모형으로 충분 → 16 vCPU · 64 GB (GPU 불필요). 예: c6i.4xlarge / e2-highmem-16 / Azure E16s. 배치라 spot/preemptible로 비용 절감, 학습 끝나면 종료. 야드 레이아웃을 쓰는 시퀀스/그래프 모형으로 확장하면 GPU 1장 추가. 학습 산출물(모형 파일, 수~수십 MB)만 서빙 노드로 배포합니다.

학습은 데이터 누적이 선행2단계 학습엔 수개월치가 필요합니다. TOS 보존은 ~15/35일뿐이라(레퍼런스 §2), 1단계 운영과 동시에 매일 스냅샷을 적재해 두어야 6–12개월 뒤 학습이 가능합니다(record-only 파이프라인, §6).

Part 7

권장 · 주의

  • 시작은 작게. 1단계는 4 vCPU·8 GB 단일 노드로 시작해 실측 후 조정. 과다 산정보다 모니터링 기반 증설이 효율적.
  • 가용성(HA). 배차가 운영에 직접 관여하면 앱·배차 노드를 이중화(2 AZ)하고 PostgreSQL은 관리형(자동 failover) 권장. 상태는 대부분 Postgres + 메모리 라이브맵(재구성 가능)이라 무상태에 가깝게 운영 가능.
  • 분리 원칙. ① 서빙 ② 배차/추론 ③ 학습을 분리하면 장애·부하가 서로 안 번짐. 특히 학습은 절대 서빙 클러스터에서 돌리지 않음(이 문서의 전제).
  • 확장. 사용자가 수백 명으로 늘면 API를 수평 확장(라이브맵 상태는 공유 캐시/단일 인제스트 + read replica). 현재 규모(내부 수십 명)에선 불필요.
  • 비용 절감. 학습 노드는 spot/preemptible + 배치 종료, GPS 원시는 콜드 스토리지.

전제·한계위 수치는 한 터미널(Westports) 규모·내부 사용자·측정된 메시지율 기준의 추정이며 헤드룸을 포함했습니다. 실제 값은 (가) 동시 사용자 수, (나) 배차 결정 주기, (다) 2단계 모형 복잡도, (라) GPS 원시 보관 정책에 따라 달라집니다. 부하 테스트와 모니터링으로 검증 후 확정하세요.
