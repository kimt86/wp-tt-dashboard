사이클 감지 v2 설계 — 레그 기반 상태머신

topos1 전이(=작업 할당)를 1급 신호로 승격한 차세대 단계 감지 설계. 실측 검증으로 확정한 신호 의미론 위에 세워졌고, 그림자(병렬) 모드로 가동하며 승격 게이트를 통과해야 본 로직이 됩니다. 현행 v1 로직은 사이클 감지 로직 참고.

1목표와 비목표

  1. 4단계(공차이동·받기·부하이동·주기) 포착률·시각 정확도 최대화 — 특히 약한 크레인 측 경계 (v1 실측: DS 픽업 29%, LD 드롭 42%).
  2. 무회귀: 사이클 완료 판정(이동필터)·라이브 KPI·dispatch 분류는 불변. v2는 사이클 내부 단계만 재설계.
  3. 그림자 우선: 별도 테이블에 병렬 기록 → 같은 사이클에서 v1과 페어드 비교 → 게이트 통과 시에만 승격. (핫패스 즉흥 수정 실패의 교훈 — 13분 워밍업 구간을 회귀로 오판해 성급히 리버트했던 사례가 연구 일지에 있다.)
  4. 관측 못 한 단계는 NULL (추정 금지 원칙 유지).

3레그(Leg) 상태머신

트럭의 삶 = 레그의 연속. 레그 = "한 목적지(topos1 타깃)로의 이동 + 체류" 단위.

레그 할당→ 이동→ 도착→ 출발→ 다음 레그
경계판정
레그 시작latch된 topos1이 새 비-빈 값으로 전이 (= 그 구간 할당)
레그 도착우선순위 합성, 출처(arr_src) 기록: ① arr_dtime(소급, 단 레그 할당 이후 값만) ② ARRIVED 상승엣지 ③ 정차 + cur_loc 매치(블록=prefix, 크레인=WHARF_*) ④ 정차 + 크레인 GPS ≤60m. 크레인측 조기 cur_loc latch는 이후 arr_dtime으로 업그레이드(v2.3), 레그가 못 잡은 블록 도착은 v1 신호로 백필(v2.4).
레그 출발도착 후 속도 ≥ 3km/h 회복 (또는 다음 레그 할당이 닫음)
핸드오버 시작은 레그 경계 아님크레인 핸드오버 *시작 시각*은 웹소켓으로 신뢰도 있게 관측 불가(QC PLC에 트럭 식별자 없음) → 수집 포기. 받기·주기(픽업·드롭에서의 체류)는 도착~출발 간격으로 충분히 표현된다(5장).

완료(v1 이동필터, 불변) ──→ 오픈 = 직전 완료 후 첫 할당 신호 (topos1 전이 ∪ container1 세팅)

작업레그1 = 공차이동·받기레그2 = 부하이동·주기

5사이클 6-이벤트 모델 (우선순위 정렬)

수집 목표를 우선순위로 재정렬: ① 사이클 열림/닫힘 ② 각 trip의 시작·도착 위치+시각. 크레인 핸드오버 시작 시각은 웹소켓으로 신뢰도 있게 관측 불가 → 수집 포기. 받기·주기는 체류 간격으로 대체.

사이클시작 ─[공차이동]─ 공차완료 ─[받기]─ 부하시작 ─[부하이동]─ 부하완료 ─[주기]─ 사이클종료
           (공차이동시작)
이벤트컬럼무엇으로 관측
① 사이클 시작opened_at직전 완료 후 첫 배정 신호(topos1/container1)
② 공차이동 시작empty_travel_start_at오픈 후 첫 실제 출발(speed≥idle, 픽업 도달 전). ①→② = 배차 후 출발 대기. 선배치면 NULL
③ 공차이동 완료empty_arrived_at픽업 도착 (arr_dtime>ARRIVED>cur_loc>GPS; container1-set이 픽업 레그 보장; v1 백필이 시각 보정)
④ 부하이동 시작pickup_left_at픽업 출발(적재 후) = topos1 플립
⑤ 부하이동 완료laden_arrived_at드롭 도착
⑥ 사이클 종료dropped_atcontainer1 클리어(이동필터, v1과 공유)
받기 / 주기 = 체류 간격받기 = ④−③ (픽업에서 머문 시간), 주기 = ⑥−⑤ (드롭에서 머문 시간). 핸드오버를 따로 안 쪼개도 두 체류 시간이 도착~출발 간격으로 자동으로 떨어진다 — 핸드오버 시작 시각이 필요 없음.
왜 핸드오버 시작을 포기했나QC PLC는 6키뿐이고 트럭·컨테이너 식별자가 없다(전수 확인). 플립-앵커 페어링으로 근사했으나 실측 충실도 낮고(픽업 ~26%, 확신 plc_paired는 극소수) 대부분 추정(plc_active)이라, "확신 없으면 빼라" 원칙에 따라 수집 자체를 중단했다.

7검증 계획과 승격 게이트

  • 페어드 비교: 같은 사이클 행에서 v1 vs v2 — 운영 변동(블록 ARRIVED율이 시간대별 78↔63% 출렁임 실측)이 양쪽에 동일 작용.
  • 워밍업 필터: opened_at > 배포 + 15분 사이클만 평가. 재시작 직후 측정 금지.
  • 관찰 24시간 이상 후 게이트 평가.
게이트기준합격선
G1약했던 경계 포착률DS 픽업도착 ≥ 60% (v1 29%) · LD 드롭도착 ≥ 70% (v1 42%)
G2강했던 경계 유지LD 픽업도착·DS 드롭도착 v1 동등 이상 (−5%p 허용)
G3일치성v1·v2 모두 있는 경계의 |Δ| 중앙값 ≤ 30초
G4건전성순서 위반 0 · 단계합 ≤ cycle_s(+5%) · 단일 신호 의존 없음(arr_src 분포)

전부 충족 시에만 메인 단계 컬럼 산출을 v2로 교체. 불합격 시 arr_src·legs로 신호별 원인 분해 → 수정 → 재검증 (그림자라 안전).

8리스크와 한계 (정직)

  • topos1도 이벤트성 — 할당 메시지 유실 시 레그 시작이 늦게 잡힘. arr_dtime 소급이 일부 만회. NULL 허용.
  • cur_loc은 지역 해상도 — WHARF_12는 "안벽 어딘가". 어느 크레인인지는 topos1(타깃)이 제공.
  • PLC 폴백(plc_active)은 추정 — 같은 크레인 다중 트럭 시 모호. 출처 태그로 구분, 정밀은 plc_paired만.
  • arr_dtime — 날짜 미포함(자정 롤오버 처리), 6시간 초과 체류는 기각.
  • 재시작 워밍업 — v2도 동일(~15분). 검증 필터로 통제.

9반복 개선과 실측 결과 (v2.2 → v2.4)

그림자 가동 후 페어드 평가(같은 (ytno, dropped_at) 행에서 v1 vs v2)로 격차를 분해하고, arr_src·legs JSONB로 신호별 원인을 짚어 네 차례 수정했다. 모두 v1 핫패스 무변경 — 그림자라 가역적.

9.1 v2.2 진단 (n=6,836 페어드 · 약 10.5시간)

두 곳이 무너져 있었다.

  • 크레인측 |Δ| ~200초 — 크레인 도착의 gps·arrived 출처가 실측 거의 0 (QC GPS가 디바이스맵에 유효 진입하지 않음). 그래서 cur_loc=WHARF가 안벽 진입 첫 프레임에 ~200초 조기 latch되어, 트럭이 실제 크레인에 도달했을 때의 정확한 arr_dtime을 잠가버렸다. → 당초 설계의 "크레인 GPS ≤40m로 조이기" 처방은 폐기(GPS 자체가 없음).
  • LD 픽업 포착 45% (v1 62%) — 미포착의 55%가 블록 레그 자체 미형성(mode2): v1 이동필터 드롭 에지가 블록 c2c에서 조기 발화 → 한 LD 트립이 "픽업 전용 + 드롭 전용" 두 사이클로 분할되어 뒤 사이클이 픽업 레그를 잃음(트럭 트레이스로 확정). 나머지 45%는 도착 미감지(mode1).

남은 약점 = 블록 포착 하나블록 도착 포착이 v1보다 ~7%p 낮음(G2 강경계 −2%p 미달). 뿌리는 토포스 레그가 안 뜨는 픽업/드롭(잔존 mode2) — v1의 연속 추적기가 포착에 더 robust. → v2.4(D)로 해소.

9.4 v2.4 — 블록 도착 백필 (D)

레그 모델은 풍부한 단계 구조를 주지만 토포스 레그가 안 뜨면 도착을 놓친다. v1 연속 추적기는 반대로 포착에 강하다. 그래서 v2 도착값을 "레그 신호 우선, 없으면 같은 행 v1 신호"로 정의했다 — 레그 모델은 손대지 않고 플러시에서만 폴백, 단조성(pickup ≤ drop ≤ dropped)을 클램프로 보장. v2 포착이 구조적으로 v1 이상이 되어 G2가 닫히고, 같은 신호이므로 |Δ|≈0이 유지된다. (선배치 근사값도 v1 실측 도착이 있으면 그것으로 교정.)

v2.4 평가 (n=359): 블록 포착이 v1을 추월 — LD픽업 67% > v1 64% · DS드롭 84% > v1 80%(G2 닫힘), |Δ| 전 경계 1초 유지. 단 백필한 정확한 도착이 분할 사이클의 스테일 pickup_left보다 늦어 empty_arrived > pickup_left 18건 발생(G4 위반).

9.5 v2.4.1 — 단조 클램프 확장

백필 도착이 pickup_left를 추월하면, 신뢰 못 할 그 출발 시각을 NULL 처리(정확한 도착 포착은 유지). 단조 체인을 empty_arrived ≤ pickup_left ≤ laden_arrived 전체로 강제.

v2.4.1 검증 (n=204)empty_arrived > pickup_left 18 → 0, 주순서 위반 전부 0(G4 ✅). |Δ| 전 경계 1초 유지(G3 ✅). DS드롭 92% ≥ v1 90%, LD픽업 v1−4%p(G2 허용 ±5%p 내), LD드롭 75% ≥ 70%(G1 ✅).
남은 미세 격차(참고)G1 DS픽업 절대 60%는 크레인 신호 한계(QC GPS 부재)로 미달하나 v1을 +5~13%p 추월. LD픽업 포착은 G2 경계에서 표본별 −4~+3%p 출렁(분할 사이클 잔존분).

9.6 v2.5 — 6-이벤트 모델로 정리 (우선순위 재정렬)

수집 우선순위를 ① 사이클 열림/닫힘 ② trip별 시작·도착으로 재정렬하고, 관측 불가한 크레인 핸드오버 시작 시각 수집을 포기했다(5장). 마이그레이션 0025: 핸드오버 컬럼 제거 + empty_travel_start_at(공차이동시작) 추가. Fix A/B/C/D(업그레이드·선배치·이월·백필)는 ②(trip 도착)에 그대로 기여하므로 유지.

경계 정확도(P1) — v2가 v1보다 나음v2는 한 트립을 둘로 쪼개는 거짓 분할을 억제: 2분 미만 비정상 사이클 v1 7.5% → v2 0.9%, 사이클 수 18% 감소(더 완결), 중앙 길이 14→17.6분(물리적 타당). 닫힘 dropped_at = container1 클리어(≈인계 종료+TOS 갱신, 드롭 도착 후 중앙 5.5분) — 물리적 들어올림 순간은 아님(외부 정답지 없음).

6-이벤트 검증 (n=152): 공차이동시작 채움 51%(선배치·이월은 정당히 NULL), 배차 후 출발 대기 중앙 24초(p90 90초), P2 도착은 v1 동등 이상 유지(DS픽업 47/43·DS드롭 90/80·LD픽업 76/75·LD드롭 84/60). 단조 위반은 공차완료 < 공차이동시작 3건(분할·이월 잔존) → v2.5.1 클램프로 정리, 전 인접쌍 위반 0 확인(n=105).

9.7 v2.6 — 픽업 레그 보장 (P2 trip 커버리지)

"레그 1개" 사이클의 지배적 격차는 픽업 레그 누락(드롭전용 407건). 픽업은 container1이 빈값→박스번호로 바뀌는 순간(가장 확실한 픽업 신호)에 일어나는데, topos1이 픽업 타깃으로 전이하지 않으면 픽업 레그가 아예 안 생겼다. → container1-set 시점에 픽업 레그를 보장(없으면 합성, 미도착이면 도착 마킹; 종류는 jobtype으로 DS=크레인/LD=블록, 정확한 도착 시각은 v1 백필이 container1 임시값을 덮어씀).

v2.6 검증 (n=164)픽업 도착 채움 66% → 84%(v1 추월: LD 87/68 · DS 74/23). 드롭전용 1-레그 LD 211→6 · DS 196→8로 거의 소멸. 단조 위반 0 유지.
남은 잔존 = 대부분 관측 불가잔존 1-레그(~29%)는 주로 LD 픽업전용 — 크레인 드롭 도착이 v2·v1 모두 미관측(약경계). "관측 못 한 값은 NULL" 원칙대로 드롭 시각은 허위 생성하지 않고 비워 둔다.
