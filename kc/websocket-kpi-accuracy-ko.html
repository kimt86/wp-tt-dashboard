websocket로 KPI 정확도 향상

KPI의 권위적 원천은 TOS지만, TOS는 몇 분마다 당겨오고 시간 단위로 버킷팅되어 실시간 해상도가 거칠고 교대 중 잠정값이 과소합니다. websocket(초단위 GPS/PLC)을 활용하면 이 실시간 구간의 정확도를 끌어올릴 수 있습니다. 이 문서는 어떤 KPI가 향상 가능한지 선별하고, 방안·구현 계획을 세우고, 첫 구현(실시간 K_MPH)을 기록합니다.

기준 2026-06-09 · 원칙: websocket은 TOS를 대체하지 않고 교차검증/실시간 보완 · 야간 TOS 확정값이 최종 권위
한눈에

30초 요약

원칙

websocket = 실시간 추정·교차검증. TOS = 권위·최종 확정. 둘을 나란히 두어 "지금 이 순간"의 정확도를 올림.

선별

크레인 PLC/GPS로 향상 가능: K_MPH(처리량)·K_QC_Q(QC 대기)가 1순위. 공차·사이클은 GPS 노이즈로 효과 낮음.

구현됨

실시간 K_MPH — 크레인 PLC 적재→비움 전이를 세어 QC별 시간당 처리량. 화면에 TOS와 교차검증.

핵심TOS K_MPH는 완료 move를 3분마다 당기고 가동시간을 정시 단위(시간 버킷)로 세어, "지금 크레인이 얼마나 빠른가"가 거칩니다. PLC는 초당 1회 들어오고 적재 전이가 곧 move라, 그 자리에서 실시간 처리량을 잴 수 있습니다.

Part 1

TOS의 한계 — 실시간 구간이 부정확한 지점

TOS 값은 과거 확정일엔 정확하지만, 진행 중인 교대(오늘)에선 다음 이유로 거칠거나 과소합니다(KPI 산출 참조).

한계내용 · 영향 KPI
당기는 주기교대 틱이 3~15분마다 → 그 사이는 정지. "현재 처리량/대기"가 최대 수분 지연. (전 KPI)
가동시간 시간버킷K_MPH의 active_hours = COUNT(DISTINCT 시) — 한 시간에 10분만 일해도 1시간. 순간 처리율을 과소평가. (K_MPH)
유휴갭 임계K_QC_Q는 QC당 유휴갭 HAVING ≥ 10 필요 → 부분 교대에선 미달해 과소/누락. (K_QC_Q)
잠정(provisional)오늘 값은 야간 확정 전까지 잠정. (전 KPI)

반대로 websocket은장비당 GPS ~3초 · 크레인 PLC ~1초로 연속 관측. "지금 이 순간"의 처리량·대기·가동을 거의 실시간으로 잴 수 있습니다(websocket 데이터).

Part 2

KPI별 향상 가능성 — 선별

KPI가능성websocket 신호 · 방안
K_MPH QC 처리량높음크레인 PLC 적재→비움 전이 = 1 move. 롤링 1시간 카운트 → 실시간 move/hr. 정시버킷 문제 없음. 구현됨(§3).
K_QC_Q QC 대기높음크레인이 move를 마치고 다음 픽업까지 idle인데 크레인에 TT가 없음(GPS) → "트럭 대기"를 직접 측정. HAVING 과소 문제 회피. 계획(§4).
K_UTIL TT 가동률중간TT GPS 시동 ON + (적재 or 작업 향해 이동) = 가동. 실시간 가동률. TOS 세션도 비교적 권위적이라 보완 성격. 계획(§4).
K_CYCLE 사이클타임낮음GPS 사이클 위상으로 추정 가능하나 TOS가 정확한 이벤트 타임스탬프 보유 → 신선도만 이득.
K_EMPTY/_R 공차거리낮음GPS 경로적분은 정확도 ~10m·3초 샘플로 노이즈가 커 TOS 기록 거리보다 부정확. 권장 안 함.

결론크레인 PLC 기반의 K_MPH·K_QC_Q가 명확한 정확도 향상 대상입니다 — 신호가 깨끗(PLC ~1초·적재 전이가 곧 move/핸드오버)하고, TOS의 실시간 약점(정시버킷·HAVING)을 정확히 메웁니다. GPS 경로 기반(공차/사이클)은 노이즈로 역효과라 제외.

Part 3

구현 — 실시간 K_MPH (PLC 사이클)

크레인 PLC의 하중(load)은 컨테이너를 들면 ≥1t, 내려놓으면 ~0이 됩니다. 적재→비움 전이 한 번 = move 한 번입니다(양하: 선박→TT 드롭 / 적하: TT→선박 드롭).

// API 인제스트 (livemap.rs) — 크레인당
PLC load 수신 (~1초):
  laden = 이력관성(≥1.0t 적재 · <0.5t 비움)   // 경계 떨림 방지(hysteresis)
  if 직전 laden && 지금 비움:  moves.push(now)    // 드롭 = move 완료
  moves에서 1시간 지난 것 제거

mph(QC) = round( 최근1시간 moves 수 / 관측경과h )   // 재시작 직후 윈도우 미충전 보정
함대 = 일한 크레인들의 mph 평균   // 종일 논 크레인은 평균에서 제외

노출 · 화면

  • API /api/livemap/positions — 크레인 PLC에 mph(최근 1h move) + last_move_age_s, 그리고 함대 crane_mph_live · crane_moves_60m · cranes_working 추가.
  • 대시보드 TT 페이지 · QC 작업 시퀀스: QC 헤더에 ⚡N/h(실시간 처리량), 패널 헤더에 ⚡ 함대 평균 move/h를 TOS와 교차검증으로 표시.

콜드스타트 보정처리량은 "최근 1시간 move"인데 재시작 직후엔 1시간이 안 차 과소합니다. 그래서 실제 관측 경과시간(최대 1h)으로 나눠 제대로 된 시간당 비율로 환산합니다 — 가동 ~10분이면 그 비율을 1시간으로 외삽(초기엔 다소 노이즈).

실측 검증재시작 후 데이터가 쌓이자 크레인 18대에서 정상적으로 move가 집계되고 QC별·함대 처리량이 실시간 갱신됨을 확인. TOS K_MPH(정시버킷)보다 "지금 이 순간"을 더 잘 반영합니다.

Part 5

주의 · 한계

  • TOS가 최종 권위. websocket 값은 실시간 추정·교차검증이고, 야간 TOS 확정값이 최종 KPI입니다. 둘을 혼동하지 않게 화면에서 "⚡실시간"으로 명확히 구분.
  • 트윈/탠덤 과소. 한 번의 스프레더 사이클로 2개를 드는 트윈리프트는 PLC 전이가 1회라 move 1로 셈 → 실제보다 약간 과소(약 20% 트윈). TOS는 정확.
  • PLC는 C/M/Z만. RTG(야드크레인)는 PLC가 없어 야드측 처리량/대기는 이 방식으로 못 잼.
  • 콜드스타트. API 재시작 후 ~10여 분은 외삽이라 노이즈. 1시간 지나면 정확.
  • 경계 떨림. 하중 임계는 hysteresis(1.0/0.5t)로 떨림을 막았으나, 비정상 하중 신호가 있으면 오집계 가능 → 모니터링.
