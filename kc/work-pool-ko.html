라이브 작업 풀

TT 배차 화면의 두 축은 차량 풀(어떤 트럭이 노는가)과 작업 풀(어떤 크레인 작업이 트럭을 기다리는가)입니다. 차량 풀은 GPS로 바로 보였지만, 작업 풀은 "미래의 계획"이라 더 어려웠습니다. 이 문서는 그 작업 풀을 TOS 라이브 작업지시에서 끌어와 실시간 화면으로 만든 방법을, 처음 보는 사람도 이해하도록 설명합니다.

기준 2026-06-09 · 원천 TOS JOB_ORDER_LIST + JOB_QUEUE_SCHEDULE(90초) + websocket PLC/GPS 융합 · 화면 = 대시보드 TT 페이지
한눈에

30초 요약

무엇

지금 각 안벽크레인(QC)이 처리 중·예정인 컨테이너 작업과, 거기에 배정된 트럭·준비 시각(ETW)을 실시간으로 보여주는 화면.

어디서

TOS의 라이브 작업지시 두 테이블을 90초마다 추출 → Postgres → API → TT 페이지. websocket PLC/GPS와 융합.

왜 의미

차량 풀(공급)의 짝인 수요 쪽 그림. "다음 노는 트럭을 어느 작업에 줄까"가 한 화면에서 보임.

핵심 통찰TT 배차 예측 연구가 "학습으로 맞혀야 한다"던 ETW(크레인 준비 시각)를 TOS가 이미 라이브로 제공합니다. 그래서 1단계 화면은 예측 없이 TOS ETW를 그대로 씁니다. 그리고 QC 번호(C##)가 TOS 계획 ↔ 실시간 PLC ↔ 트럭 GPS를 한데 묶어줍니다.

Part 1

왜 어려웠나 — 공급 vs 수요

차량 풀이 실시간으로 쉬웠던 이유와 작업 풀이 어려웠던 이유는 데이터의 본질이 다르기 때문입니다.

차량 풀 = 공급

직접 관측됨 → 쉬움 ✓

각 트럭(TT)이 websocket으로 자기 현재 상태(적재 여부·목적지·도착·속도)를 직접 송신합니다.

"누가 지금 노는가"는 그냥 보입니다. 그래서 차량 풀은 standalone으로 실시간이 됐습니다.

작업 풀 = 수요

일부는 미래 → 어려움

"앞으로 처리할 작업 순서(NOW/NEXT/+N)"는 크레인 PLC도 트럭 GPS도 알려주지 않습니다 — 그건 TOS의 작업 계획입니다.

websocket은 지금 일어나는 1건만 드러냅니다. 미래 계획은 TOS DB에서 가져와야 합니다.

해결TOS에 라이브 작업지시 테이블이 실제로 존재한다는 것을 확인(읽기전용 탐사)했습니다. 거기엔 작업 순서·배정 트럭·ETW·컨테이너가 다 들어있습니다. 이걸 90초마다 끌어와 Postgres에 두고, 화면에서 실시간 PLC/GPS와 합쳤습니다. 대시보드/API는 여전히 Oracle을 건드리지 않습니다 — 추출기만 접근(원칙 유지).

Part 2

데이터 원천 — TOS 라이브 작업지시 2테이블

둘 다 계속 갱신되는 운영 테이블(UPD_DT=현재). 자세한 컬럼은 TOS DB 레퍼런스 §3.5에도 정리돼 있습니다.

JOB_ORDER_LIST — 개별 컨테이너 move

한 컨테이너 작업 1건 = 1행. 완료 이력 테이블(JOB_ORDER_HISTORY)의 라이브 판이지만, 완료행도 함께 보관(2018년~ 약 20만 행)되므로 반드시 라이브만 필터합니다.

JOB_ODR_JOBSTATUS — 작업 상태 코드 (열쇠) A 활성(배차됨·진행중)   Q 대기(백로그)   P 계획   B 차단   C 완료.
화면에 카드로 띄우는 건 A(활성) move — 이것만 ETW·배정 트럭이 채워져 있습니다.

  • JOB_ODR_YTNO — 배정된 트럭(예 TT1153). GPS의 장비 id와 같아 라이브 위치/상태를 머지.
  • JOB_ODR_ETW_DT — ETW = 크레인이 이 작업을 받을 준비가 되는 시각. ETW − 현재가 작업의 시급도.
  • JOB_ODR_QUEUENAME — 큐 이름(아래 큐 테이블과 잇는 조인 키). JOB_ODR_ARMGC RTG · JOB_ODR_CONTNO 컨테이너 · YT_TOPOS 야드 블록-베이.

JOB_QUEUE_SCHEDULE — QC별 작업 큐 계획

각 QC가 어떤 큐를 어떤 순서로 작업하는지. 작고 깨끗(한 큐당 1행).

  • JOB_QUE_CRANENO — QC(안벽크레인) 번호(C11..C55·M·Z·DC##). C##는 websocket PLC/GPS 크레인 id와 직접 일치.
  • JOB_QUE_SEQ 작업 순서 · JOB_QUE_DISLOAD 양하(D)/적하(L) · JOB_QUE_QUEUENAME 큐 이름(예 34H-D = 베이34 선창 양하).
  • TOTALQTY / COMPQTY — 총/완료 물량 → 잔여(백로그) = TOTAL − COMP.

둘을 잇기 = 작업 풀

JOB_ORDER_LIST (move: ETW·배정TT·컨테이너)
   └─ QUEUENAME + 선박 으로 조인 ─▶ JOB_QUEUE_SCHEDULE (QC·순서·진행률)
        = "이 QC가 / 이 순서로 / 이 컨테이너들을 / 이 트럭에 배정해 / 이 시각에 처리한다"

조인은 Postgres에서 (Oracle 아님)큐 이름(예 02D-L)은 항차마다 재사용되어, Oracle에서 바로 조인하면 과거 항차들과 fan-out(중복 폭증)이 납니다. 그래서 활성 move만 평범하게 뽑고, QC는 깨끗한 현재 큐 스냅샷으로 Postgres에서 붙입니다.

Part 4

화면 읽는 법 — QC 작업 시퀀스

TT 페이지의 "QC 작업 시퀀스 & 배차(라이브)" 패널은 QC당 한 컬럼입니다. 각 컬럼은 세 신호의 융합입니다.

TOS 계획 — 무엇을(컨테이너), 어느 순서로, 어느 트럭에, 언제(ETW). 90초 스냅샷.
PLC 실측 — 그 크레인이 지금 물리적으로 도는 중이면 컬럼 헤더에 PLC live 배지. (websocket ctab, 초단위)
트럭 GPS — 각 작업 카드의 배정 트럭(YT) 옆에 그 차의 실시간 상태 점(유휴/곧유휴/적재이동/RTG대기). (websocket wpt_gps)

컬럼 구성

요소의미
헤더QC 번호 + 점(가동 여부) + 선박명 + 잔여 N(이 QC의 남은 총 move = Σ(큐 TOTAL−COMP)). 가동 중이면 PLC live.
진행바이 QC 전체 큐의 완료/총 물량(예 54 / 838) + N개 활성.
작업 카드활성 move 1건. DSC양하/LOD적하 · 컨테이너 번호 · 경로(블록↔QC) · ETW 카운트다운 · 배정 트럭(점=실시간 상태).
+N 작업 더 보기 ▼그 QC의 나머지 활성 작업을 펼침/접기.

ETW가 "overdue"로 많이 뜨는 건 정상활성(A) 작업은 이미 진행 중인 핸드오버라 크레인 준비 시각(ETW)이 보통 직전 과거입니다. 즉 "overdue"는 오류가 아니라 "지금 처리 중"이라는 뜻입니다.

아래 "임박 작업 프론트"는 전체 활성 move를 ETW 시급순으로 모은 목록이고, "TT 상태 분포"는 차량 풀의 실시간 상태 도넛입니다.

Part 5

이 데이터는 진짜인가? — 그렇다

화면의 트럭 배정·컨테이너·ETW는 전부 TOS 실데이터이고, 트럭 옆 상태 점은 실시간 GPS입니다. 목업(가짜) 데이터는 없습니다.

화면 요소출처실/가
배정 트럭(YT 번호)TOS JOB_ORDER_LIST.JOB_ODR_YTNO진짜
컨테이너 번호 · ETW · RTG · 경로TOS JOB_ORDER_LIST진짜
QC 순서 · 진행률(완료/총)TOS JOB_QUEUE_SCHEDULE진짜
트럭 상태 점(유휴/곧유휴/…)websocket GPS 실시간 분류진짜
PLC live 배지websocket 크레인 PLC(가동 중)진짜

실측 교차검증 (2026-06-09)활성 작업에 배정된 트럭 94대 중 67대가 그 순간 라이브 GPS 피드에도 잡혔습니다(나머지는 그 순간 위치 fix 미송신뿐). 즉 TOS 배정과 실제 차량이 일치합니다.

한계 · 운영 노트

항목내용
활성 작업은 전부 배정됨TOS에서 A(활성)는 "트럭이 이미 배차된 진행 작업". 따라서 "미배차 수요"는 활성이 아니라 대기(Q) 백로그이고, 그 깊이는 큐의 잔여 = TOTAL−COMP로 표시합니다.
대기(Q) 백로그는 개별 카드 없음대기 move는 ETW·순서·배정이 없는 묶음이라 개별 카드로 못 띄웁니다 → 큐 단위 잔여 수치로만 표현. (추출도 활성만 가져옴)
ETW "overdue"진행 중 핸드오버라 정상(§4).
stale 고아 행드물게 큐가 사라진 활성 move(과거 ETW)는 CRE_DT ≥ SYSDATE−2 필터 + QC 없는 행 제외로 정리.
Oracle 부하90초마다 bounded 2쿼리. "Oracle 부하 최소화" 원칙을 한 단계 넘는 새 반복 쿼리라 도입 시 명시 승인을 받았습니다(2026-06-09, 90초).
다음 단계 (2단계)지금은 TOS의 ETW·배정을 그대로 표시. 2단계에선 이 라이브 작업 풀 + 차량 풀을 입력으로 AI 전역 매칭(헝가리안 등)이 더 나은 배차를 제안 — 예측 모형 연구 참조.
