처음 오신 분을 위한 안내

Westports(말레이시아) 컨테이너 터미널의 야드 트럭(TT) 운영을 데이터로 풀어가는 TT AiOps Platform의 지식센터입니다. 이 페이지 하나만 읽으면 어디로 가야 할지 알 수 있습니다.

▶
바쁘신가요? — 5분 요약
플랫폼 전체(문제 → 데이터 → 지표 → 운영 → 다음 단계)를 한 페이지로 압축했습니다.

환영합니다 — 이 프로젝트는 무엇인가요

컨테이너 터미널에서 안벽 크레인(QC)이 멈추지 않으려면 야드 트럭(TT)이 제때 도착해야 합니다. 이 플랫폼은 TT 운영을 측정하고(KPI), 실시간으로 보고(라이브 맵·배차 풀), 궁극적으로 AI 배차로 개선하기 위해 만들어졌습니다.

구조는 단순합니다: 운영 TOS(Oracle)는 절대 직접 건드리지 않고 추출기가 주기적으로 PostgreSQL로 복제하며, 터미널 차량의 실시간 GPS/PLC websocket을 함께 수신해 Rust API + React 대시보드로 서빙합니다. 모든 지표는 "TOS가 말하는 것"과 "차량이 실제로 움직인 것"을 교차 검증하며 만들어집니다.

여정 따라가기 — 5개 챕터

순서대로 읽으면 문제 정의부터 향후 연구까지 한 흐름으로 이어집니다.

01
문제 — 스마트 TT 배차란 무엇인가≈ 20분
고객사와의 워킹 세션 기록. 왜 TT 배차가 어려운 문제인지, 현재 운영 방식과 개선 기회가 어디에 있는지를 정의합니다. 모든 챕터의 출발점.
02
데이터 — TOS DB 레퍼런스≈ 14분
작업 지시(JOB_ORDER), 작업 큐, 장비 운영 테이블까지 — 우리가 읽는 TOS 테이블의 구조와 의미, 그리고 "운영 DB 무접촉" 원칙(추출기 → Postgres)을 다룹니다.
같은 챕터 → 실시간 websocket 데이터 (≈ 8분) — GPS·PLC 피드의 구조와 함정
03
지표 — KPI 산출 방법≈ 16분
사이클타임·공차거리·가동률·QC 대기 등 핵심 지표를 어떤 소스에서 어떤 수식으로 계산하는지. 지표마다 "왜 이 정의를 선택했는가"를 함께 기록했습니다.
같은 챕터 → websocket로 정확도 보강 (≈ 11분) — 실측 GPS 교차 검증
04
운영 — 배차 풀 (라이브)≈ 8분
대시보드 TT 페이지의 심장: 작업 풀(수요)과 차량 풀(공급)이 90초/실시간으로 어떻게 갱신되고 융합되는지. idle→staging 교정, 스왑 가능 공차 판정 로직 포함.
같은 챕터 → 사이클 감지 로직 (≈ 12분) — 이동필터·4단계 분해·tt_cycle_log
05
다음 — 2단계 예측 모형 연구≈ 35분
AI 배차로 가는 길: 작업 소요시간 예측 + 배차 최적화의 2단계 모형 설계와 학습 데이터 전략. 지금 쌓고 있는 사이클 로그(tt_cycle_log)가 이 연구의 연료입니다.

다른 방식으로 들어가기

구체적인 질문이 있다면
"ETW가 왜 DB와 다른가요?", "idle이 왜 줄었나요?" — FAQ에서 바로 답을 찾으세요.
모르는 단어가 있다면
ETW, c2c, 양하/적하, work pool, JOBSTATUS… 용어집이 한 줄씩 정리합니다.
시간순 원문이 궁금하다면
무엇을 시도했고 무엇을 발견했는지 — 연구 일지가 날짜별로 기록합니다.
인프라 구축을 계획한다면
CPU·메모리 등 클라우드 용량 산정 근거 자료.

이 플랫폼의 정직한 현황

운영 중 KPI 대시보드(7개 지표·일/주/월 이력) · 라이브 맵 · 배차 풀 · 사이클 이력(CYCLES) 페이지가 실데이터로 가동 중입니다. TT 사이클 로그는 ML 학습용으로 시간당 수백 건씩 적재되고 있습니다.
검증됨 "idle 과다 집계" 원인(배정 무시)과 "container1=배정 필드" 의미론을 실측으로 규명하고 교정했습니다. 관측하지 못한 값은 추정하지 않고 NULL로 남기는 원칙을 따릅니다.
주의 정확한 ETW는 Azure 게이트웨이(SSH 터널) 의존 — 터널이 죽으면 DB 폴백으로 정밀도가 떨어집니다. 사이클 4단계 분해는 ARRIVED 신호 의존이라 일부 사이클은 단계가 부분 관측됩니다.
미완 AI 배차(예측 모형)는 아직 연구 단계 — 현재는 학습 데이터 축적과 베이스라인 측정에 집중하고 있습니다.
