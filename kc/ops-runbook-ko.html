1운영 원칙

  • prod Oracle 직접 조회 절대 금지 — 추출기(게이트웨이)만 사용.
  • 관측 못 한 값은 NULL — 추정으로 채우지 않는다(추정은 ML의 일이지 적재의 일이 아님).
  • 라이브 핫패스 변경은 그림자(병렬) 검증 후 승격 — 즉흥 수정 금지. 페어드 비교로 무회귀 확인.
  • 비교는 페어드로만 — 같은 (ytno, dropped_at) 행에서 v1 vs v2. 운영 변동(시간대별 도착율 출렁임)이 양쪽에 동일 작용.
  • 재시작 워밍업 ~15분 — 인메모리 상태 초기화로 직후 완료 건은 앞 단계 누락. 측정은 opened_at > 재시작+15분 필터.

3빌드 · 배포 · 재시작

API는 release 빌드를 실행하므로, 코드 수정 후 반드시 release 빌드 → 서비스 재시작해야 반영된다.

# Rust API 빌드 후 재시작 (livemap/사이클 로직 변경 시)
cargo build --release -p wp-api
systemctl --user restart wp-api.service
systemctl --user status wp-api.service | head -5

# 지식센터(kc/) 페이지 렌더 체크 (playwright)
cd web && node kc-render-check.mjs <page.html>
재시작 = 워밍업 리셋재시작하면 인메모리 트럭 상태가 초기화된다. 직후 ~15분간 완료된 사이클은 앞 단계가 누락되므로 측정에서 제외(opened_at > 재시작 + 15분).

새 KC 문서 추가 시

  • kc/에 *-ko.html 생성 (기존 페이지의 head boilerplate + 셸 패턴 복제).
  • kc/kc-shell.js의 NAV 배열에 등록 — 사이드바 단일 소스.
  • NEWS에 한 줄 추가 · node kc-render-check.mjs로 렌더 확인.

4빠른 검증 명령

# 서비스 상태 + 사이클 적재 (v1 / v2 그림자)
systemctl --user status wp-api.service | head -5
psql "postgresql://wp:wp@127.0.0.1:5433/wp_tt" -c \
  "SELECT count(*), max(dropped_at) FROM tt_cycle_log WHERE dropped_at > now()-interval '1 hour';"
psql "postgresql://wp:wp@127.0.0.1:5433/wp_tt" -c \
  "SELECT count(*), max(dropped_at) FROM tt_cycle_v2  WHERE dropped_at > now()-interval '1 hour';"

  • CyclesPage의 phasesOf · 사이클 감지 로직(v1) 문서 갱신, 승격 기록.
  • 받기/주기(체류) + 공차이동시작을 CYCLES 막대에 노출 — ✅ 완료: CYCLES 페이지에 v2 6이벤트 5세그먼트 막대 + v1/v2 토글(미관측=빈칸). detail API가 tt_cycle_v2 LEFT JOIN.

T4. 백로그 (급하지 않음)

  • topos2 latch 활용 — 트윈 레그 타깃 예고.
  • lapse(체류 분) 활용 — QC 대기 교차검증.
  • 재시작 내구성 — 진행 중 사이클 스냅샷(tt_cycle_open).
  • PLC tpos(트롤리)로 육측/해측 구분 — 크레인별 캘리브레이션 필요.
  • ML 모델 착수 — tt_cycle_log/tt_cycle_v2가 연료. 예측 모형 연구의 라벨 ①②③.
