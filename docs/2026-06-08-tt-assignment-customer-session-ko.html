AS-IS · WP TOS

현행 배차 방식

bolt 현재 규칙

  • 차량이 유휴 상태가 되면 TOS가 즉시 작업을 할당합니다 — 1대당 1건.
  • 해당 차량 한 대 기준으로 보통 가장 가까운 작업을 선택합니다.
  • 차량 위치는 DGPS 또는 직전 작업 완료 지점에서 가져옵니다.
  • 전체 차량을 하나의 통합 풀로 관리하며 크레인 밸런스 규칙을 적용합니다.

swap_horiz 사후 보정 — 재배차 스왑

  • TOS는 공차 거리가 과도하게 긴 배차를 기억해 둡니다.
  • 이후 더 가까운 유휴 차량이 생기면, 그 작업을 더 나은 차량으로 교체(스왑)해 상황을 개선합니다.
  • 즉, 처음엔 어쩔 수 없이 멀리 배차하더라도 나중에 기회가 생기면 고치는 방식입니다.

이를 탐욕(greedy) 방식이라 합니다: 한 번에 한 대씩 결정하고, 부족한 부분은 사후 스왑으로 보정합니다.

The Gap

왜 "한 대씩"은 최선을 놓치는가

warning 각 차량의 최선 작업을 따로 고르면 터미널 전체의 최선 결과를 놓칩니다. 같은 예시(차량 2대·작업 2건)를 두 방식으로 비교합니다.

↔ 선 위 숫자 = 공차 이동 거리 (차량이 빈 채로 이동하는 거리)

A가 자기 기준 가장 가까운 작업1(거리 100)을 먼저 가져가면, B는 멀리 떨어진 작업2(거리 900)로 밀려 총 1000. 둘을 함께 보면 A→작업2, B→작업1로 총 700 — 약 30% 감소. 같은 차량·같은 작업인데도 결과가 달라집니다.

이 격차는 교대 시점에 가장 커집니다 — 400~500대가 동시에 배차를 필요로 할 때입니다.

차량 풀

현재 유휴 현재 작업이 없는 차량.

곧 유휴 마지막 핸드오버를 마무리 중인 차량 — 반응이 아니라 사전에 계획.

스왑 가능 공차 작업을 향해 공차 이동 중이나 아직 미상차 — 재매칭(스왑) 대상.

작업 풀

각 크레인의 ETW(작업 준비 예상 시각) 기준, 시급한 순서대로 후보 작업 추출.

차량 풀 크기만큼 채워서 모든 차량이 매칭 후보를 갖도록 구성.

매칭은 검증된 알고리즘(헝가리안 / Kuhn–Munkres)을 사용합니다. 500×500 규모에서 약 35 ms에 결과를 반환합니다.

Assignment Rule

계산은 전체, 실제 할당은 유휴 차량만

가장 중요한 원칙: 매칭 계산은 차량 풀 전체에 대해 수행하지만, 실제 배차 명령은 지금 유휴인 차량에게만 내립니다. 곧 유휴·스왑 가능 차량까지 계산에 포함하는 이유는, 유휴 차량의 작업 선택이 터미널 전체 관점에서 왜곡되지 않게 하기 위해서입니다.

차량 유형계산실제 동작
현재 유휴포함매칭 결과를 즉시 실제 배차
곧 유휴포함미리 할당하지 않음 — 유휴가 되는 순간 로직을 다시 수행해 그때 실제 할당 (가장 최신 상황 기준)
스왑 가능 공차포함기존 작업과 다른 작업이 매칭되면 스왑 수행

schedule 유휴 차량이 바로 작업을 못 받을 수도 있습니다

  • 작업 풀 크기가 차량 풀 크기보다 작으면(대기 작업 부족), 일부 유휴 차량은 그 라운드에 작업을 받지 못할 수 있습니다.
  • 이 경우 N초마다 반복 실행해 재할당하도록 할 수 있습니다 협의 필요
  • 현장 병목 회피를 위해 유휴 차량의 임시 이동이 필요하다면, 관련 차량 운영 정책 지원이 필요합니다 협의 필요

Cost Function

"비용"의 의미 — 쉽게 풀어서

엔진은 각 (차량 → 작업) 짝마다 '비용' 점수를 매기고, 비용이 가장 낮은 조합을 고릅니다. 비용이 낮다 = 더 좋은 배차.

1단계 — 거리만 본다

거리가 짧을수록 비용 ↓ = 더 좋은 배차

2단계 — 시간도 더한다

공차로 달린 시간

UAT 약 2주간 AI는 TOS에 실연동된 상태로 동작하되 배차 결정은 반영하지 않고 기록만 합니다. 목적은 알고리즘 성능 평가가 아니라 "실운영에 넣어도 안전한가"의 확인이며, Westports와 CLT가 공동으로 점검·판정합니다. 정확한 수치는 실측 데이터 첫 주를 바탕으로 합의합니다.

B. 알고리즘 건전성AI 배차 결과의 TOS 제약 충족률 100% (불가능 배차 0건) · 이상 배차 패턴 없음
C. 운영 준비대시보드 정상 동작 · 알림·폴백 runbook 검토 · 장애 대응 절차 리허설 · 공동 체크리스트 서명
