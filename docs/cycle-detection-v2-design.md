# TT 사이클 감지 v2 — 레그(Leg) 기반 상태머신 설계

2026-06-10 · wp-tt-dashboard · 설계 문서 (구현 전 리뷰용)

## 0. 목표와 비목표

**목표**
1. 4단계(공차이동·받기·부하이동·주기) **포착률과 시각 정확도 최대화** — 특히 현재 약한 크레인 측 경계(DS 픽업 29%, LD 드롭 42%).
2. 검증된 것은 불변: **사이클 완료 판정(이동필터)과 라이브 KPI는 그대로** — v2는 사이클 *내부* 단계만 재설계.
3. **그림자 우선**: v2는 별도 테이블에 병렬 기록 → 같은 사이클에서 v1과 페어드 비교 → 합격 게이트 통과 시에만 승격. (지난 핫패스 즉흥 수정 실패의 교훈.)
4. 관측 못 한 단계는 **NULL** (추정 금지 원칙 유지).

**비목표**: 인도(drop) 검증 로직 변경, 라이브 dispatch 분류 변경, KPI 산식 변경.

## 1. 신호 모델 — 조사로 확정된 사실 (2026-06-10)

| 신호 | 거동 | 신뢰도/특성 | v2에서의 역할 |
|---|---|---|---|
| `topos1` | **할당 이벤트에만 실림** (하트비트엔 빈값) | 값 자체는 정확. DS: 크레인(픽업)→블록(드롭), LD: 블록(픽업)→크레인(드롭) 전이 실측 확인 | **레그 시작 신호** (= 작업/구간 할당 시점) |
| `topos2` | 트윈/탠덤의 2번째 핸드오버 지점 (프레임의 8.3%) | container2와 짝 | 트윈 다중 레그(블록1→블록2) 추적 |
| `container1/2` | **TOS 배정 필드** (물리 적재 아님), 직전 drop에 사전배정(c2c) | 슬롯 스왑 잦음(가드 적용 중) | **완료 엣지**(이동필터) + 트윈 식별. 단계 경계론 사용 안 함 |
| `arrival` | `"ARRIVED"`, 도착 **상태로 지속** | 블록 ~78% / **크레인 ~23%** (정차 트럭 기준) | 도착 신호 #2 |
| `arr_dtime` ★신규 | **도착 시각** (HH:MM:SS), 도착 상태 동안 지속 | ARRIVED와 동행. **늦게 봐도 정확한 도착 시각을 소급 취득** → rising-edge 타이밍 불필요, **재시작에도 견고** | 도착 신호 #1 (시각의 원천) |
| `lapse/1/2` ★신규 | 도착 후 경과 **분** (서버 계산, `dtime−arr_dtime` 검증됨) | 체류 타이머 | 교차검증·QC 대기 보강 |
| `cur_loc`(=`addr`) ★신규 발견 | **현재 위치 라벨, 매 프레임 상시** (`02G`, `WHARF_12`…) | 미션 필드가 아니라 위치 필드 — 항상 실림 | 도착 신호 #3 (**상시·지역 단위**): 블록=prefix 매치, 크레인=`WHARF_*` 진입 |
| 크레인 GPS/centroid | 크레인 GPS 송출 57%, ARRIVED 트럭으로 학습한 centroid 보강 | 40m 근접은 시간 오차 큼(실측 ~133s) | 도착 신호 #4 (어느 크레인인지 정밀화 보조) |
| 크레인 PLC `load` | empty→laden 엣지 = 픽업, ~1초 | 크레인 단위(트럭 귀속은 모호) | **받기 구간의 보조 스탬프** (`pickup_plc_at`), 경계 1차 신호로는 안 씀 |

핵심 통찰 두 가지:
- (사용자) **topos1 전이가 곧 할당** → 사이클·레그를 할당 시점에 열면 공차 접근 구간이 *사이클 안*에 들어와, 지난 시도에서 DS=0이었던 구조적 공백이 사라진다.
- (필드 조사) **arr_dtime + cur_loc**이 있으면 약한 크레인 ARRIVED에 기대지 않아도 된다 — arr_dtime은 시각을 소급 제공하고, cur_loc은 매 프레임 위치를 준다.

**topos1 플립 시점 실측 (2026-06-10, raw 이중 구독: GPS topos1 플립 × 크레인 PLC 훅로드 엣지)**

질문: DS에서 topos1(크레인→블록) 플립이 핸드오버 **시작**(lift, 선박에서 집음)인가 **완료**(land, 트럭에 안착)인가?

| 측정 | n | 중앙값 (Δ = PLC엣지 − 플립) | IQR |
|---|---|---|---|
| DS 플립 vs **LAND**(완료) | 36 | **−19s** (land가 플립보다 19초 먼저) | −26 ~ −1s |
| DS 플립 vs LIFT(다음 무브 시작) | 37 | +10s | −0 ~ +16s |
| LD 해방 vs LIFT (캘리브레이션) | 39 | **−4s** — 선행 연구(±수초) 재현, 방법론 검증 | −14 ~ +2s |

타임라인: `land(박스가 트럭에 안착) → ~19초(트위스트락 해제·TOS 확인 추정) → topos1 플립 → ~10초 → 크레인 다음 lift`.
**결론: topos1은 핸드오버 완료 후(중앙값 +19초)에 바뀐다.** 측정 시 주의: 피드 서버가 ws ping에 무응답이라 python 클라이언트는 `ping_interval=None` 필수(아니면 ~40초에 조용히 끊김).

## 2. 핵심 개념 — 레그(Leg) 상태머신

트럭의 삶 = **레그의 연속**. 레그 = "한 목적지로의 이동+체류" 단위.

```
레그 시작   : latched_topos1이 새 비-빈 값으로 변경 (= 그 구간 할당)
레그 도착   : leg_arrived_at — 아래 도착 판정 (target 종류별)
레그 출발   : leg_left_at — 도착 후 speed ≥ 3km/h 회복, 또는 다음 레그 할당
레그 종료   : 다음 레그 시작 또는 사이클 완료
```

**도착 판정 (우선순위 합성, 출처를 `arr_src`로 기록)**

| target | 판정 (먼저 잡히는 신호 사용) |
|---|---|
| 블록 | ① `arr_dtime` 존재 → 그 시각 (소급) ② `arrival==ARRIVED` 관측 시각 ③ `block_prefix(cur_loc) == block_prefix(target)` 진입 시각 |
| 크레인 | ① `arr_dtime` ② `ARRIVED` ③ **`cur_loc`이 `WHARF_*` 진입** + (가능하면) 해당 크레인 GPS/centroid ≤ 60m로 정밀화 ④ 크레인 PLC 활동 + 정차 (보조) |

`arr_dtime`은 HH:MM:SS만 오므로 터미널 날짜(MYT)를 붙이고 자정 롤오버(현재시각보다 미래면 −1일)를 처리한다. lapse(분)와 일관성 검사.

## 3. 사이클 수명 (v2)

```
COMPLETE(v1, 불변)            OPEN(v2)
   직전 인도 ──────────────▶ 첫 할당 신호 = topos1 변경 ∪ container1 세팅 중 먼저 관측
                              (c2c는 인도와 동시, 공차 갭이 있으면 공차 중 할당)
```

**레그 → 단계 매핑 (jobtype별)**

| | 레그1 (공차) | 받기 | 레그2 (적재) | 주기 |
|---|---|---|---|---|
| **DS** | target=**크레인** · 공차이동 = open→레그1 도착 | 레그1 도착→**topos1 플립(=핸드오버 완료+~19s, 실측)**. 이 창의 QC PLC **land** → `pickup_plc_at`(정밀 물리 완료 스탬프) | target=**블록** · 부하이동 = 플립→레그2 도착 | 레그2 도착→인도 |
| **LD** | target=**블록** (트윈이면 블록1→블록2 두 레그: topos2) | 동일 (블록 도착→출발) | target=**크레인** | 동일 |
| **MI/MO/LC** | 블록→블록, 순서 기반 (첫 레그=픽업) | | | |

- 트윈(LD 수령형): 레그1a/1b 각각 도착·출발 기록 → "받기"는 1a 도착→1b 출발로 확장. (DS 분산 하역은 드묾 0.3% — 동일 구조로 자연 처리)
- 슬롯 스왑 가드, 이동필터, c2c 플래그 전부 v1 그대로 (완료 판정 공유).

**핸드오버 시작 추정 (`handover_start_est_at`) — soon_idle 관여 로직의 일반화 (사용자 결정)**

soon_idle 판정의 핵심 조건("크레인/RTG가 이 트럭에 관여 시작")을 레그 도착 후 첫 충족 시점으로 일반화해
픽업·드롭 양쪽에 스탬프한다. 받기/주기가 `도착 →[순수 대기]→ 시작 →[서비스]→ 완료`로 3분할되어
"장비를 기다린 시간"이 분리된다 (QC/RTG 대기 KPI·ML 피처 직결).

| 레그 타깃 | 시작 판정 | `handover_src` |
|---|---|---|
| 블록 | 도착(정차) 후 최근접 RTG ≤ 30m(같은 bay) 첫 진입 — `wait_rtg→soon_idle` 전이와 동일, 예측연구 검증(관여 ~25m vs 대기 100m+) | `rtg_bay` |
| 크레인 (정밀) | 플립-앵커 PLC 페어링: 서빙 land(플립 직전 0~60s 윈도우)의 **직전** lift. "최근접 엣지"가 아님에 주의 — 다음 무브 lift 오매칭 방지. 윈도우·순서(lift<land<플립) 검증 실패 시 폐기 | `plc_paired` |
| 크레인 (폴백 추정) | 도착 후 그 QC PLC 활성(최근 무브 ≤120s) 첫 충족 — soon_idle의 "PLC 확인"과 동일 | `plc_active` |

근거 실측: ① topos1 플립 = 핸드오버 완료 + ~19s (n=36) → 플립 직전 land가 서빙 무브로 귀속 가능
② PLC에는 트럭/컨테이너 식별자가 없음(plc_data 전수 열거: crane·load·lock·land·hpos·tpos 6키가 전부,
QC/RTG의 wpt_gps 미션 필드는 항상 빈값) → 귀속은 트럭측 앵커(플립)로만 가능, 폴백은 추정으로 표기.

## 4. 저장 — 그림자 테이블 `tt_cycle_v2`

v1 행과 `(ytno, dropped_at)`으로 1:1 조인. v1 컬럼은 일절 건드리지 않음.

```sql
CREATE TABLE tt_cycle_v2 (
  ytno            TEXT NOT NULL,
  dropped_at      TIMESTAMPTZ NOT NULL,      -- = v1 행의 dropped_at (조인키)
  opened_at       TIMESTAMPTZ,               -- 첫 할당 신호
  -- 평탄화된 4단계 경계 (비교·승격 대상)
  empty_arrived_at  TIMESTAMPTZ,             -- 픽업측 도착 (레그1)
  pickup_left_at    TIMESTAMPTZ,             -- 픽업측 출발 (레그1/1b)
  laden_arrived_at  TIMESTAMPTZ,             -- 드롭측 도착 (레그2)
  -- 핸드오버 분해 (도착→[대기]→시작→[서비스]→완료)
  pickup_hand_start_at TIMESTAMPTZ,          -- 픽업측 핸드오버 시작 (est/paired)
  drop_hand_start_at   TIMESTAMPTZ,          -- 드롭측 핸드오버 시작 (est/paired)
  pickup_plc_at     TIMESTAMPTZ,             -- 크레인측 물리 완료 = 서빙 land (플립-앵커)
  handover_src_pickup TEXT,                  -- 'plc_paired'|'plc_active'|'rtg_bay'
  handover_src_drop   TEXT,
  arr_src_pickup  TEXT,                      -- 'arr_dtime'|'arrived'|'cur_loc'|'gps'|'plc'
  arr_src_drop    TEXT,
  legs            JSONB,                     -- 레그 원자료 [{target,kind,assigned,arrived,src,left}…] (검증·디버그)
  PRIMARY KEY (ytno, dropped_at)
);
```

코드: 기존 사이클 머신과 **분리된 LegTracker**(Pos에 v2 전용 필드 — 기존 로직은 읽지 않음) + flusher가 v1 INSERT 직후 v2 행 INSERT. Pos 파싱에 `arr_dtime`·`lapse`·`topos2`·`cur_loc`(이미 있음) 추가.

## 5. 검증 계획 — 워밍업·운영변동 내성 (지난 실패의 교훈 반영)

- **페어드 비교**: 같은 사이클 행에서 v1 vs v2 → 운영 변동이 양쪽에 동일하게 작용 (A/B 시간창 비교의 함정 제거).
- **워밍업 필터**: `opened_at > 배포 + 15분` 사이클만 평가.
- **관찰 기간**: 최소 24시간 (블록 ARRIVED율이 시간대별로 78↔63% 출렁이는 것 확인됨).

**승격 게이트 (전부 충족 시에만 메인 컬럼 로직 교체)**

| # | 기준 | 합격선 |
|---|---|---|
| G1 | 포착률 (약했던 경계) | DS 픽업도착 ≥ 60% (v1 29%) · LD 드롭도착 ≥ 70% (v1 42%) |
| G2 | 포착률 (강했던 경계) | LD 픽업도착·DS 드롭도착 v1 동등 이상 (−5%p 허용) |
| G3 | 일치성 | v1·v2 둘 다 있는 경계의 \|Δ\| 중앙값 ≤ 30초 |
| G4 | 건전성 | 순서 위반 0 · 단계합 ≤ cycle_s(+5%) · `arr_src` 분포에 단일 신호 의존 없음 |

불합격 시 `arr_src`·`legs` JSONB로 신호별 원인 분해 → 수정 → 재검증 (그림자라 안전).

## 6. 리스크와 한계 (정직)

1. **topos1도 이벤트성** — 할당 메시지를 유실하면 레그 시작이 늦게 잡힘. arr_dtime 소급이 일부 만회하지만 완전하진 않음 → NULL 허용.
2. **cur_loc의 지역 해상도** — `WHARF_12`는 "안벽 어딘가"이지 어느 크레인인지 모름. 크레인 식별은 topos1(타깃)이 제공하므로 "그 타깃에 도착했는가"엔 충분하나, topos1이 틀렸/늦었다면 같이 틀림.
3. **PLC 귀속 모호** — 같은 크레인에 트럭 여러 대 대기 시 어떤 move가 어느 트럭인지 불확실 → 1차 경계로 안 쓰고 `pickup_plc_at` 보조 스탬프로만.
4. **arr_dtime 파싱** — 날짜 미포함, 자정 롤오버·시계 오차 가능 → lapse와 교차검증, 비정상이면 폐기(NULL).
5. **재시작 워밍업** — v2도 동일하게 겪음. 검증 필터로 통제하고, 승격 후에도 운영 지표 해석 시 주의 문서화.
