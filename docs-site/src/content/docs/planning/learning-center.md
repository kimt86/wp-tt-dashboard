## 한 줄 요약

**학습 센터**는 누적되는 작업 사이클 데이터로 **예측 모형을 학습→검증→활용**하는 일을 한곳에서 굴리는 거점이다. 무엇을·어떻게의 *기술 상세*는 이미 [예측 모형 연구](/kc/research/tt-prediction/)에 있다 — **이 문서는 그 학습을 어떻게 조직·운영·활용할지의 프로그램 차터**다.

## 왜 지금

- 운영(1단계)이 도는 것만으로 **학습데이터가 자동 생산**된다 — record-only 파이프라인이 사이클을 라벨로 바꾼다([연구 §5·§6](/kc/research/tt-prediction/)).
- 우리는 이미 [`tt_cycle_log`/`tt_cycle_v2`(6-이벤트)](/kc/experiments/cycle-v2-shadow/)를 검증된 단위로 쌓고 있다 — 학습의 연료가 매일 쌓인다.
- 흩어지면 또 까먹는다. 데이터셋·모델·실험·교훈을 [지식센터](/kc/start/overview/) 컨벤션 위에 **단일 출처로** 모은다.

## 1. 무엇을 학습할지

비용 함수가 예측 대상을 정해준다(상세·라벨 원천은 [연구 §3·§7](/kc/research/tt-prediction/)). 프로그램 우선순위순:

| # | 예측 대상 | 출력 | 우선순위 |
|---|---|---|---|
| ① | TT 이동 시간 (구간) | 소요 초 | 핵심 (먼저) |
| ② | QC ETW (크레인 준비 시각) | 준비까지 초 | 핵심 (가중치 최대) |
| ③ | 블록 핸드오버 시간 (선적) | 수령까지 초 | 핵심 |
| ④ | 핸드오버 소요 (QC↔TT) | 초 | 보조(정밀화) |
| ⑤ | "곧 유휴" 잔여 시간 | 초 | 보조(풀 편입 판정) |

- **데이터 자산**: 사이클 로그(6-이벤트)가 ①③⑤의 라벨을, QC PLC 상관이 ②④의 정밀 라벨을 준다.
- **약점 인지**: 블록측(RTG)은 PLC가 없어 ③·양하 곧유휴가 가장 어렵다 — RTG 관여(GPS 근접)로 보완([연구 §7b](/kc/research/tt-prediction/)).
- **레이아웃 인지**: 이동시간(①)·혼잡은 단순 거리가 아니라 **도로·차선 레이아웃**에 좌우된다 → 아래 §4에서 레이아웃 기반으로 별도 학습.

## 2. 어떻게 학습할지

- **자가 라벨(record-only)**: 운영 중 파이프라인을 돌려 *예측 vs 실제*를 매 배차 기록 → 라벨이 자동 누적. 별도 라벨링 없음.
- **점진적 복잡도**: `v0 베이스라인`(구간별 중앙값) → `v1` 그래디언트 부스팅 → 필요 시 `v2`. **항상 v0 대비 개선분을 입증**한다.
- **정직한 검증**: **시간 분할**(미래 교대로 검증, 누설 차단), 지표 **MAE(초)·P90**, 주/야·물량·작업종류 층화. 배차 품질은 보조 시뮬레이터로 교차.
- **데이터 품질 규율**(이 프로젝트의 핵심 자산):
  - 관측 못 한 값은 **NULL**, 추측으로 채우지 않는다.
  - 세 신호(GPS·PLC·TOS) 교차 일치 표본만 학습에 쓴다(노이즈 제거).
  - **누설 방지**: 피처는 "결정 시점 t₀ 이전" 정보만.

## 3. 학습한 것을 어떻게 활용할지

- **배차 비용 함수에 투입**: ①②③ 예측 → 후보 (차량→작업)의 ETA·ETW·대기 → 헝가리안 매칭으로 총비용 최소 조합([연구 §2·§3](/kc/research/tt-prediction/)).
- **그림자 → 측정 → 승격** (사이클 v2와 **같은 규율**, [실험](/kc/experiments/cycle-v2-shadow/)): record-only로 예측·추천을 그림자로 돌려 *예측 vs 실제* lift를 측정하고, 게이트 통과 시에만 실제 배차에 반영.
- **대시보드 노출**: 예측 사이클타임·추천 배차·혼잡/크레인대기 예보를 surfacing(FEED/HEALTH 연계).
- **폐루프**: 예측↔결과를 다시 학습데이터로 — 정확도를 상시 모니터.
- **폴백 안전**: 예측 신뢰 낮음·모형 장애 시 **1단계 거리비용으로 즉시 복귀**. 2단계는 1단계 위에 얹는 가중치라 투자가 보호된다.

## 4. 레이아웃 데이터 학습 (layout-aware)

레이아웃 — **영역**(바다·안벽·블록·도로)·**차선**(도로별 개수·주행 방향)·**시설**(게이트·주차) — 을 공간 substrate로 삼아, 이동·혼잡·경로 예측을 직선/arc 거리 근사에서 **실제 도로·차선 네트워크 기반**으로 끌어올린다. 레이아웃 정의·빌더·라우터는 별도 이니셔티브이고, 학습 센터는 **그 산출물을 입력으로 받아 학습**한다.

### 무엇을 (레이아웃에서 학습)

| 학습 대상 | 출력 | 라벨 원천 |
|---|---|---|
| 도로·차선 구간별 **유효 속도** | 구간 통행 초 | GPS 트레이스 → 차선 네트워크 map-match |
| 구간별 **혼잡 패턴**(시간대) | 구간 지연 초 | GPS 집계 |
| 교차로·회전·게이트 **노드 지연** | 노드 통과 초 | 트레이스 |
| **차량 스펙별** 통행 특성 | 속도·지연 보정 | 트레이스 × 차량 제원 |

### 어떻게

- GPS 트레이스를 레이아웃의 **차선 네트워크(edge·node)에 map-matching** → 구간·노드별 라벨. (현행 파이프라인의 블록/포인트/arc 매칭을 **차선 단위로 확장**.)
- 이동시간(①)의 거리 피처를 직선/arc → **레이아웃 경로 거리**(라우터 산출, 일방통행·차선 수 반영)로 교체·보강.
- 차량 크기/스펙을 피처로(좁은 구간·회전 영향). record-only 자가 라벨 규율은 동일.

### 어떻게 활용

- **라우터**(레이아웃 기반 경로) + 학습된 구간 시간 → 더 정확한 **ETA** → 배차 비용함수(위 §3).
- **구간별 혼잡 예보** → 라우터의 **예측형 데드락·병목 방지**에 직접 입력.
- 빌더로 레이아웃이 바뀌어도 **동일 파이프라인이 재학습**(레이아웃 버전 = 학습 컨텍스트로 기록).

:::caution[의존성]
(미검증) 레이아웃 정의(영역/차선/시설)·빌더·라우터가 선행/병행해야 한다. 정의 전엔 현행 arc/거리 근사를 유지하고, **차선 단위 map-matching은 GPS 정확도 임계**(예 ≤30m)에서만 채택한다(GPS는 장비당 ~3초로 성김).
:::

## 5. 학습 센터 운영 — "센터"의 형태

학습 산출물을 지식센터 어디에 둘지(단일 출처 유지):

| 산출물 | 위치 |
|---|---|
| 예측 대상·데이터 설계 연구 | [리서치](/kc/research/) (예: [예측 모형 연구](/kc/research/tt-prediction/)) |
| 모델 학습·게이트·lift 측정 | [실험](/kc/experiments/) (실험 로그 템플릿) |
| 큰 선택(모델·피처·승격) | [의사결정 ADR](/kc/decisions/) |
| 작업 일지·삽질·정정 | [구현 기록](/kc/journal/) |
| **데이터셋·모델 레지스트리**(신규) | 제안: 전용 `학습(learning)` 섹션 신설 — 데이터셋 카드·모델 카드·지표 대시보드 |

- **카덴스**(제안): 주간 모델 리뷰(지표·lift·드리프트), 실험은 로그로 즉시 기록.
- **합의 필요**(TOS팀): "곧 유휴" 정의·호라이즌·신뢰 임계([연구 §7b.4](/kc/research/tt-prediction/)).

## 6. 단계별 로드맵

연구의 일정과 정합([연구 §11](/kc/research/tt-prediction/)) — *(미검증)* 날짜는 연구 문서 기준, 함께 확정 필요:

1. **지금**: record-only 파이프라인 가동, `train_*` 테이블 축적, **v0 베이스라인으로 데이터 건전성 조기 점검**.
2. **학습·검증**: v1 학습, 시간분할 검증, 지표 합의.
3. **전환 검증**: record-only로 예측 vs 실제 lift 공동 판정.
4. **활용 전환**: 게이트 통과 시 시간비용(2단계)으로 전환. 선적 곧유휴 먼저, 양하는 점진 확대.

## 원칙

- **그림자 → 검증 → 승격.** 검증 안 된 예측을 핫패스에 바로 쓰지 않는다.
- **관측 못 한 값은 NULL**, 추측 금지. 베이스라인 먼저, 복잡도는 입증된 만큼만.
- 모든 학습 문서는 [글쓰기 규칙](/kc/start/contributing/)의 상태표시·`(미검증)` 원칙을 따른다.

## 열린 질문

- (미검증) 프로그램 **1차 목표 지표**는? (공차 절감 / 크레인 대기 절감 / 회전율 중 무엇을 우선)
- (미검증) "곧 유휴" 정의·호라이즌·신뢰 임계 — TOS팀 합의 필요.
- **데이터셋·모델 레지스트리**를 전용 `학습` 섹션으로 신설할지, [지식베이스](/kc/knowledge/)에 둘지.
- v0 베이스라인의 합격 기준(어느 MAE면 v1로 갈 가치가 있나).
- (미검증) **레이아웃 정의 완료 시점**, 차선 단위 map-matching 정밀도(성긴 GPS 대비), 라우터와 학습의 책임 경계(§4).
