이 프로젝트에서 가장 비싼 지식이다 — **재발견하지 말 것.** 필드 전수 레퍼런스는 [websocket 원본 필드](/kc/architecture/websocket-fields/) 참고.

## 핵심 결론

1. **container1/2 = TOS 배정 필드** (물리 적재 아님). 직전 drop에 다음 박스를 사전배정(c2c). 검증: 적하(LD) 34대가 container1이 세팅된 채 픽업 블록에 ARRIVED.
2. **topos1 = 작업/구간 할당 신호** (실측 확정). 양하(DS): 크레인(픽업)→블록(드롭), 적하(LD): 블록→크레인. **topos1 플립 = 핸드오버 완료 + 중앙값 19초** (PLC land 대비, n=36; LD 캘리브레이션 −4초로 방법론 검증).
3. **arrival(ARRIVED)**: 블록 ~78% / **크레인 ~23%**(정차 트럭) — 크레인 측 경계가 약한 근본 원인.
4. **arr_dtime** = 도착 시각(HH:MM:SS, 도착 상태 동안 지속 → 늦게 봐도 소급 취득 가능). **lapse** = 도착 후 경과 분(서버 계산). **cur_loc** = 매 프레임 상시 위치 라벨(미션 필드 아님).
5. **QC PLC(ctab) = 정확히 6키**(crane/load/lock/land/hpos/tpos) — **트럭·컨테이너 식별자 없음**(전수 열거로 확정). QC/RTG는 위치만 송출(미션 필드 항상 빈값). → 크레인-트럭 귀속은 **플립-앵커 페어링**이 유일했다(이후 핸드오버 수집은 포기, [사이클 v2 실험](/kc/experiments/cycle-v2-shadow/) 참고).

## 함정 (검증됨)

- **raw ws 구독 시 서버가 ping 무응답** → python `websockets`는 `ping_interval=None` 필수. 아니면 ~40초 무음 단절.
- **재시작 워밍업 ~15–20분**: 인메모리 상태 초기화로 직후 완료 사이클은 앞 단계 누락 → 측정은 `opened_at > 재시작+15분` 필터. 운영 변동(블록 ARRIVED 78↔63%)도 크다 → **비교는 페어드(같은 행 v1 vs v2)로만.**

## 근거 / 방법

라이브 피드를 구독해 필드별 등장 패턴·전이를 트럭·시간대별로 교차 관찰하고, PLC 엣지(lift/land)와 topos1 플립의 시차를 페어드로 측정했다(캘리브레이션으로 방법론 자체 검증).

## 열린 질문

- (미검증) PLC `tpos`(트롤리)로 육측/해측 구분 가능성 — 크레인별 캘리브레이션 필요.
- (미검증) `topos2`의 트윈 레그 타깃 예고 활용도.
