여정 05 · 다음 — 1단계(거리 기반)는 예측이 필요 없습니다. 2단계는 비용을 **시간**으로 바꾸므로 "TT가 언제 도착하는가 · 크레인이 언제 준비되는가"를 **예측**해야 합니다. 본 보고서는 우리가 가진 두 raw 데이터(TOS DB · 실시간 websocket)에서 **무엇을 학습 대상으로 삼고**, 그 학습을 위해 **어떤 학습데이터를 어떻게 가공**할지 연구한 결과입니다.

WP AI Service 2 · Phase 2 Prediction Research · 대상 단계 **Phase 2 (목표 전환 2026-10-30)** · 학습데이터 축적 구간 **Phase 1 운영 8/17–10/23** · 근거 데이터 **실측 18일치 + 실시간 GPS/PLC**

## 한 장 요약

2단계 비용 함수는 세 가지 시간을 더합니다 — **이동 시간 + 차량 대기 + 크레인 대기**(크레인 대기에 최대 가중치). 이 세 값을 후보 (차량→작업)마다 **배차 시점에 미리 알 수 없으므로 예측**해야 합니다. 예측의 뿌리는 세 가지입니다.

- **① TT 이동 시간** — 임의의 (출발지점 → 도착지점)을 TT가 이동하는 데 걸리는 시간. v→안벽, v→블록, 블록→안벽 모든 구간의 기본 단위.
- **② QC ETW (크레인 준비 시각)** — 안벽크레인이 현재 작업을 끝내고 **다음 TT와 핸드오버를 시작**할 수 있는 시각. 크레인 대기/차량 대기를 가르는 핵심.
- **③ 블록 핸드오버 시간(선적)** — 선적 시 TT가 **블록에 도착해서 RTG로부터 컨테이너를 받기까지** 걸리는 시간. 선적 경로에만 추가되는 지연.

:::note[핵심 결론]
이 세 예측의 **정답(라벨)** 은 새로 만들 필요가 없습니다. 실시간 GPS를 **지도-매칭 → 상태/이벤트 → 트립/작업사이클**로 가공하면 각 구간의 **실제 소요 시간**이 그대로 나오고, 크레인 PLC의 **적재 전이(PICKUP/DROP)** 및 TOS의 **작업 이력**과 시간으로 상관(±10초)시키면 "도착 ↔ 핸드오버" 쌍이 완성됩니다. 즉 **1단계 운영 중 record-only로 이 파이프라인을 돌리는 것 자체가 2단계 학습데이터를 자동 생산**합니다.
:::

실측 데이터가 동기를 뒷받침합니다: 현재 **TT 이동의 약 50%가 공차**(빈 차 이동)이고, **크레인이 TT를 기다리는 시간의 중앙값이 시간대에 따라 7~11분**으로 출렁입니다(§9). 이 둘이 2단계가 직접 줄이려는 손실입니다.

## 배경 — 왜 2단계는 예측이 필요한가

AI 배차 엔진은 매 라운드 **모든 차량 × 모든 후보 작업**의 비용 행렬을 만들고, 총비용이 최소가 되는 조합을 헝가리안 알고리즘으로 찾습니다. 핵심은 행렬의 각 칸 `cost(v, j)`를 무엇으로 채우는가입니다.

| 단계 | 비용 `cost(v,j)` | 필요 입력 | 예측 |
|---|---|---|---|
| 1단계 | 공차 이동 **거리** | 차량 위치(DGPS) · 작업 지점 · 현재 시각 — 모두 **확정 사실** | 불필요 |
| 2단계 | **시간**: 이동시간 + 차량대기 + 크레인대기 (가중합) | "언제 도착·언제 준비"는 **미래** — 관측 불가 | 필수 |

:::note
1단계가 거리를 줄이는 것은 좋은 **대리지표(proxy)** 지만, 실제 운영 손실은 "거리"가 아니라 "**크레인이 노는 시간**"입니다. 가까운 차량이라도 그 차량이 다른 작업을 마무리하느라 **늦게** 도착하면 크레인이 기다립니다. 거리는 같아도 길 막힘·경로 차이로 **시간**이 다릅니다. 2단계는 비용의 단위를 거리→시간으로 바꿔 이 격차를 직접 줄입니다.
:::

따라서 2단계의 숙제는 두 가지입니다 — (a) 후보마다 세 시간 성분을 **예측**하는 모형, (b) 그 모형을 학습시킬 **라벨링된 학습데이터**. 본 보고서는 (b)에 집중하되, (a)가 요구하는 형태에 맞춰 데이터를 설계합니다.

## 비용 함수 분해 — 예측이 들어가는 정확한 자리

2단계 비용은 한 후보 (차량 v → 작업 j)에 대해 다음과 같이 분해됩니다.

```
cost(v,j) = w_cw · 크레인대기 + w_vw · 차량대기 + w_tt · 이동시간   (w_cw 최대)
```

각 성분은 두 개의 "예상 시각"에서 파생됩니다 — 차량이 핸드오버 지점에 닿는 **ETA_v**와 크레인이 그 핸드오버를 받을 준비가 되는 **ETW_j**.

```
ETA_v = 현재시각 + 이동시간(v의 현재위치 → j의 지점)   ← 예측①
ETW_j = 크레인 준비 시각(작업 j 크레인)               ← 예측②
크레인대기 = max(0, ETA_v − ETW_j)    차량대기 = max(0, ETW_j − ETA_v)
```

### 선적(LD)은 한 단계가 더 있다 — 블록 경유

양하(DS)는 TT가 안벽으로 가서 크레인에게서 컨테이너를 받지만, **선적(LD)은 TT가 먼저 야드 블록에 가서 RTG로부터 컨테이너를 받아** 안벽으로 가져가야 합니다. 따라서 선적 후보의 ETA는 두 이동과 한 핸드오버의 합입니다.

```
ETA_v(선적) = 현재 + 이동(v→블록) + 블록 핸드오버(RTG 수령) + 이동(블록→안벽)   ← 예측①+③+①
```

**그림 2-1. 선적(LD) TT 여정 타임라인 — 색칠된 구간 = 예측 대상.** 양하(DS)는 ①(v→안벽)·②만으로 단순화됨.

| 단계 | 구간 | 예측 |
|---|---|---|
| 배차 시점(현재 위치) → 블록 도착 | ① 이동 v→블록 | 예측 ① |
| 블록 도착 → 컨테이너 수령 | ③ RTG 핸드오버 | 예측 ③ |
| 컨테이너 수령 → 안벽 도착(= ETA) | ① 이동 블록→안벽 | 예측 ① |
| 안벽 도착 → QC 핸드오버 | 대기/동기화 + QC 핸드오버 | 예측 ②(ETW vs ETA 차이 = 크레인대기 / 차량대기) + 보조 예측 |

:::tip[설계 원칙]
복잡한 단일 모형 하나가 아니라, **재사용 가능한 작은 예측기 3종**(이동시간·ETW·블록핸드오버)을 만들어 경로에 따라 조합합니다. 각각 독립적으로 학습·검증·교체할 수 있어 리스크가 분리됩니다.
:::

## 무엇을 학습하나 — 예측 대상 정의

핵심 예측기 3종과, 정확도를 높이는 보조 예측 2종입니다. 각각은 "입력(피처) → 출력(예측값)"이 명확한 회귀 문제입니다.

| # | 예측 대상 | 출력(예측값) | 비용함수에서의 역할 | 정답 원천 |
|---|---|---|---|---|
| ① | **TT 이동 시간** (핵심) | (출발 지점 → 도착 지점) 이동 소요 **초** | ETA 산출의 기본 단위(모든 경로 구간) | GPS 트립 |
| ② | **QC ETW**(크레인 준비 시각) (핵심) | 크레인이 다음 TT 핸드오버를 시작 가능한 시각까지 **초** | 크레인대기 / 차량대기 산출 | 크레인 PLC + MCH_OPERATION |
| ③ | **블록 핸드오버 시간**(선적, RTG 수령) (핵심) | 블록 도착 → 컨테이너 수령 완료까지 **초** | 선적 ETA에 추가되는 지연 | GPS 작업사이클(블록 dwell) |
| ④ | 핸드오버 소요 시간(QC↔TT 교환) | 크레인-차량 교환 자체에 걸리는 **초** | ETA·ETW 보정(정밀화) | PLC PICKUP/DROP 간격 |
| ⑤ | "곧 유휴" 잔여 시간(현재 작업 완료 ETA) | 작업 중인 TT가 유휴가 되기까지 **초** | "곧 유휴" 차량을 풀에 넣는 판정·시점 | GPS 작업사이클 |

:::note
①②③은 **비용 행렬을 채우는 데 직접** 쓰이고, ④⑤는 **동기화 정밀도**를 높입니다. 모두 동일한 가공 파이프라인(§5)의 산출물에서 라벨이 나오므로, 데이터 가공을 한 번 잘 해두면 다섯 가지를 모두 학습할 수 있습니다.
:::

## 우리의 raw 데이터 — 두 원천

학습데이터의 모든 재료는 두 곳에서 나옵니다. 성격이 상호보완적입니다 — TOS DB는 "무슨 작업이 있었나(이벤트·결과)"를, websocket은 "그게 공간·시간에서 어떻게 일어났나(연속 관측)"를 줍니다.

- **TOS DB — Oracle (tos-db-research)** — 완료된 작업의 권위적 기록. 컨테이너 단위 이벤트, 거리, 크레인 작업, 선석/항차 계획. 단위 timestamp는 **초** 단위, 보존 기간 제한(아래).
- **Websocket — 실시간 스트림(2 zone)** — 장비 약 **450대** 관측. `wpt_gps`(GPS+미션, 장비당 **~3초 간격·불규칙**) · `ctab`(크레인 PLC, **~1초 규칙적**). 연속 관측이라 "소요 시간"을 직접 측정 가능.

### 4.1 TOS DB 핵심 테이블

| 테이블 | 핵심 컬럼 | 학습에 주는 것 |
|---|---|---|
| `TOSADM.JOB_ORDER_HISTORY` (컨테이너 작업 이력) | `JOB_HIST_JOBTYPE`(LD/DS/MO/MI…), `CONTNO`, `POINT`, `SEQNO`, `DATE`/`TIME`, `ARMGC`(크레인), `YT_DIS_DT`, `ACTV_DT`, `CRNT_PSN_IDX_NO1`(블록), `LNDN_TRV_RNG`(적재거리), `UN_LNDN_TRV_RNG`(공차거리) | 작업 종류·컨테이너·출발/도착 지점·**크레인 큐 대기(YT_DIS→ACTV)**·적재/공차 거리·사이클 시간. **이동시간·크레인대기 라벨의 검증 기준**. |
| `TOSADM.MCH_OPERATION` (크레인 작업 기록) | `MCH_OPER_MACHNO`(C##=QC), `ST_DT`(시작), `COMPDATE`+`COMPTIME`(완료), `JOBTYPE`, `VESSEL`/`VOYAGE`, `CONTNO`, `TRK_ID` | 크레인별 **move 단위 시작/완료** → **ETW(다음 move 준비) 라벨**·move 소요·QC 유휴 갭. |
| `TOSADM.VSS_STATISTICS` (항차 계획·실적) | `VAN`/`MOVES`(계획 물량), `TEU`, `SIN/TWN/TND_MOV`(단/트윈/탠덤), `GROSSTIME/NETTIME`, `GQCR/NQCR`(QC rate) | 항차 컨텍스트 피처(남은 물량·규격 혼합·평균 rate) — ETW 예측 보강. |
| `MCH_WORKTIME · MCH_WORKSTOP · CDY_MACHINE` (TT 가동·정지) | TT 작업 세션 시작/종료, 정지 구간, 장비 마스터(YT/QC 타입) | 차량 가용성·교대 경계·정지 구간 제거 — 트립 정제와 가용 차량 판정. |

:::caution[보존 기간 — 즉시 행동 필요]
`JOB_ORDER_HISTORY`는 약 **15일**, `MCH_OPERATION/VSS`는 ≥35일만 보존됩니다. 2단계 학습엔 수개월치가 필요하므로, **1단계 운영 시작과 동시에(또는 그 전부터) 매일 스냅샷을 Postgres로 적재**해 영구 축적해야 합니다. 이미 우리 extractor가 일/교대 단위로 raw_* 테이블에 적재 중이며, 이를 trip/cycle 단위로 확장하면 됩니다.
:::

### 4.2 Websocket — `wpt_gps` (gps_update) 필드

차량/크레인이 보내는 위치 + **현재 미션** 패킷. 위치뿐 아니라 그 장비가 지금 무슨 컨테이너를 어디로 나르는지가 들어 있어, GPS만으로도 작업 맥락을 복원할 수 있습니다.

:::caution[실측 송신 주기 — 1Hz 아님(2026-06-08 70초 캡처)]
`wpt_gps`는 정기 1Hz 스트림이 아니라 **장비별 평균 ~3초 간격(중앙값 3.2s · p90 10s · 최대 55s)의 불규칙 보고**입니다(이동 시 더 잦고 정지 시 수십 초까지 벌어짐, 변경 시 추가 push). 각 `gps_update` 뒤에 `disconnect`가 1:1로 따라오는 "접속→fix 1건→끊김" 구조. 전체 합산 ~40건/초. **이 성김은 트립 재구성·이동시간 라벨의 시간 해상도에 영향**(§10). 반면 `ctab` PLC는 크레인당 거의 **1초 규칙적**(중앙값 1.04s)이라 핸드오버 순간 포착에 충분합니다.
:::

```js
// 실제 TT 패킷 샘플 (gps_update) — 위치 + 미션
{ "device":"TT1045", "lat":"2.9179205", "lon":"101.28711433",
  "speed":"8kmh", "engine_on":"ON", "accuracy":"10", "dtime":"11:50:27",
  "cur_loc":"01U", // 현재 위치 라벨(블록/안벽)
  "topos1":"03U-21", // 목적지 코드(다음 핸드오버 지점)
  "jobtype":"LD", "vslname":"ESL SHEKOU", // 작업 종류·선박
  "container1":"EGSU2064058", "container2":"EGSU3909308", // 트윈
  "arrival":"ARRIVED", "fuel_level":"38.8", "nett":"TELE" }
```

| 필드 | 의미 · 학습 활용 |
|---|---|
| `device, lat, lon, speed, engine_on, accuracy` | 장비ID·위치·속도·시동·GPS정확도 → 지도매칭·상태분류·트립 재구성의 기본. |
| `dtime, arr_dtime` | 장비 보고 시각(MYT) · 도착 시각 → 이벤트 시각 보정. |
| `cur_loc` / `topos1, topos2` | **현재 위치 라벨** / **목적지 코드**(예 `03U-21`=블록, `C35`=크레인). 트립의 의도된 도착지·작업 상대 식별. |
| `jobtype, vslname, container1/2` | 작업종류(LD/DS/MO/MI)·선박·컨테이너(트윈/탠덤 판별) → 작업사이클 분류·피처. |
| `arrival, lapse/lapse1/2` | 도착 플래그·체류 카운터 → 도착/체류 이벤트 보강. |
| `fuel_level, batt, nett, userid` | 연료·배터리·통신·운전자 → 보조 피처/품질. |

### 4.3 Websocket — `ctab` (plc_data) 크레인 PLC

크레인의 PLC 상태 — 실측 **거의 1초 규칙적**(중앙값 1.04s · p90 1.25s). **적재 하중(load)의 전이**가 곧 핸드오버 순간(집기/내리기)이라, 크레인-TT 동기화의 가장 정밀한 정답 신호입니다.

```js
// 실제 크레인 PLC 샘플 (plc_data)
{ "crane":"C51", "load":1.9, // 하중(톤): 0=빈후크, ≥1.0=적재
  "lock":"False", "land":"False", // 트위스트락 · 안착
  "hpos":"31.74", "tpos":"27.68" } // 호이스트·트롤리 위치(m)
```

적재 전이로 **PICKUP**(load 0.5↓→1.0↑)·**DROP**(load 1.0↑→0.5↓) 이벤트를 만듭니다. 이 이벤트 시각이 TT의 도착 시각과 ±10초 내로 맞으면 "그 TT가 그 크레인과 핸드오버했다"가 확정됩니다(§6).

:::note[중요: 어떤 크레인이 PLC를 주나]
현재 `ctab` PLC는 **C##(안벽), M##, Z#** 계열(동적 크레인)에서 옵니다. **RTG(야드크레인)는 PLC 미수신**일 수 있으므로, 블록 핸드오버(③)의 정답은 RTG PLC가 아니라 **TT 자신의 블록 체류 시간**(GPS로 측정한 도착→적재완료)에서 얻습니다. 이 점이 ③의 라벨 설계를 좌우합니다(§7-③).
:::

## 학습데이터 가공 파이프라인 — raw에서 라벨까지

raw GPS/PLC는 그 자체로 학습데이터가 아닙니다. "초당 좌표 점"을 "**한 번의 이동이 몇 초 걸렸다**"는 라벨로 바꾸려면 가공이 필요합니다. 다행히 참조 프로젝트(wp-tt-data-center)가 이 가공 파이프라인을 **이미 구현·검증**해 두었습니다. 우리는 이를 학습데이터 생산 라인으로 재사용합니다.

**그림 5-1. 세 raw 스트림을 가공·상관시켜 라벨링된 학습 테이블을 생산하는 파이프라인.** GPS 레인이 "구간 소요시간"을, PLC 레인이 "핸드오버 순간"을, TOS가 "검증·맥락"을 제공.

```
wpt_gps (위치+미션, 장비당 ~3s·불규칙):
  ① 투영(lat/lon→x,y mm) → ② 지도매칭(블록·포인트·arc) → ③ 상태분류(IDLE/이동/적·양하)
  → ④ 단계이벤트(도착/출발/적재…) → ⑤ 트립/작업사이클(구간 소요시간)

ctab (크레인 PLC, ~1초 규칙적):
  PLC 파싱(load·lock·land·pos) → PICKUP/DROP 검출(load 0.5↔1.0 전이)

TOS DB (작업 이벤트, 일/교대 적재):
  JOB_ORDER_HISTORY · MCH_OPERATION · VSS (거리·크레인 move·계획 — 검증/보강 피처)

세 레인 → [상관 + 라벨 저장소] → 학습 테이블 (§7)
```

### 가공 단계별 산출과 핵심 규칙

| 단계 | 산출 | 핵심 규칙(참조 구현 기준) |
|---|---|---|
| ① 투영 | x_mm, y_mm | 등거리 투영 + 6-파라미터 아핀(14k+ 쌍 OLS, RMS ~70m). |
| ② 지도매칭 | block_id · point_id · point_type · nearest_arc | STRtree+KD트리. point_type: 1=안벽, 2/3=블록작업, 5/6=게이트, 4/7=버퍼/배터리…. |
| ③ 상태분류 | IDLE / TRAVELING / LOADING / UNLOADING / BUFFER | 속도<0.3m/s·체류≥30s→IDLE, 안벽 12m·블록 8m 반경 + 저속→적/양하. 5초 flap 디바운스. |
| ④ 단계이벤트 | arrive_quay/depart_quay/arrive_block/depart_block/start·end_loading·unloading | 상태 전이 + 블록 진입/이탈(3샘플 디바운스)에서 8종 이벤트 발생. |
| ⑤ 트립/사이클 | `vehicle_trip` · `work_cycle` | 30s 갭/5분 IDLE로 트립 분할. 사이클: 도착·출발 타임스탬프 + dwell + cycle_type(DS/LD/MO·MI) + arc 경로/거리. |
| PLC | `crane_plc_state` · `crane_pickup_event` | PICKUP: prev<0.5t & cur≥1.0t / DROP: prev≥1.0t & cur<0.5t. |

이 산출물 중 **`work_cycle`** 한 행이 곧 "TT의 한 번의 컨테이너 운반"이며, 그 안에 출발/도착/적재완료 시각이 다 들어 있어 §7의 학습 테이블이 거의 그대로 만들어집니다.

## 라벨(정답) 생성 로직 — 도착 ↔ 핸드오버 상관

예측 모형의 정답은 "실제로 몇 초 걸렸나"입니다. 두 가지 정답을 두 방식으로 만듭니다.

### 6.1 이동·체류 라벨 — 단계 이벤트 사이의 시간차

한 `work_cycle`의 타임스탬프만 빼면 됩니다. 별도 데이터가 필요 없습니다.

| 라벨 | 계산식 (work_cycle 타임스탬프) | 대상 |
|---|---|---|
| 이동시간(구간) | `dest_arrive_at − origin_depart_at` | 예측 ① |
| 블록 핸드오버(선적) | `dest_work_end_at − dest_arrive_at` (cycle_type=loading, 블록) | 예측 ③ |
| 안벽 대기+핸드오버(TT측) | arrive_quay → depart_quay | 예측 ④ 보조 |
| "곧 유휴" 잔여 | 현재시각 → 예상 `next_depart_at` | 예측 ⑤ |

### 6.2 크레인 핸드오버 라벨 — 시간 상관(±10초)

"이 TT가 도착한 순간, 그 상대 크레인이 정말 집었나?"를 확정해야 ETW(②)의 정답이 깨끗해집니다. TT의 `dest_arrive_at`과 크레인 `crane_pickup_event.time`을 ±10초 창에서 가장 가까운 것으로 매칭합니다(참조 구현의 `crane_correlation_dt_s`).

**그림 6-1. GPS 도착, PLC 적재 전이, TOS move 기록을 컨테이너번호와 시각으로 교차 검증.** 세 신호가 맞을 때만 라벨로 채택 → 노이즈 제거.

| 신호 레인 | 한 핸드오버에서의 관측 |
|---|---|
| TT GPS | arrive(안벽/블록) → UNLOADING/LOADING 체류 → depart |
| 크레인 PLC | PICKUP/DROP. TT arrive와 `|Δt| ≤ 10s`이면 동일 핸드오버 확정 |
| TOS DB | MCH_OPERATION move (ST_DT→COMP) · CONTNO 일치 |

:::tip[왜 강력한가]
세 독립 신호(연속 GPS · 물리 PLC · 권위 TOS)가 같은 사건을 가리킬 때만 학습 표본으로 쓰므로, 잘못 매칭된 표본이 모형을 망치는 일을 막습니다. 컨테이너번호(`container1`↔`CONTNO`)가 강한 조인키가 됩니다.
:::

### 6.3 성긴 GPS를 PLC·TOS로 정밀화 — 시각 스냅

GPS가 1Hz가 아니라 **장비당 ~3초(정지 시 30~55초)** 이므로(§4.2), GPS만으로 잡은 "도착/출발 시각"은 **±수 초~수십 초** 흔들립니다. 이동시간(①)은 분 단위라 이 정도 오차가 작지만, **핸드오버 순간(②④)과 블록 수령(③)** 은 초 단위 정밀도가 중요하므로 더 정밀한 신호로 시각을 **스냅(보정)** 합니다.

| 라벨 시각 | 1차(성김) | 정밀화(스냅) | 최종 정밀도 |
|---|---|---|---|
| 이동 출발/도착 | GPS arrive/depart 이벤트 | 그대로 사용(분 단위라 영향 작음) | ±수 초 |
| **QC 핸드오버 순간**(②④) | GPS 안벽 도착 | **QC PLC PICKUP/DROP**(~1초)에 ±10s 스냅 → 라벨은 PLC 시각 | **±1초** |
| **블록 수령 완료**(③, 양하/선적 블록측) | GPS 블록 dwell 종료 | RTG PLC 없음 → **TT 상태전이** + **TOS** `YT_DIS_DT`/`ACTV_DT`로 보강 | ±수 초(PLC보다 약함) |

:::caution[비대칭 — 블록측이 약점]
안벽(QC)측 핸드오버는 PLC가 1초 단위로 찍어줘 라벨이 깨끗하지만, **블록측(RTG)은 PLC가 없어** 시각 정밀도가 낮습니다. 그래서 ③(블록 핸드오버)과, 뒤의 **양하 차량의 "곧 유휴" 판별**이 가장 어렵습니다(§7b). 이 약점은 **TOS의 권위적 작업완료 시각**과 **인접 RTG의 GPS 관여 단서**로 보완합니다.
:::

## 예측 대상별 학습 테이블 — 피처 + 라벨 + 샘플

각 예측기는 "한 행 = 한 사건"인 표를 학습합니다. 아래는 우리 raw에서 바로 채울 수 있는 구체 스키마입니다.

### ① TT 이동 시간 — `train_travel_time`

한 행 = TT의 완료된 한 이동 구간(트립). 출발→도착 지점쌍과 맥락으로 소요 초를 예측.

| 컬럼 | 원천 | 예시 |
|---|---|---|
| `origin_block, origin_point, origin_point_type` | 매칭(⑤ 트립) | `Y_B01, 01U, 1` |
| `dest_block, dest_point, dest_point_type` | 매칭 / `topos1` | `—, 03U-21, 1` |
| `straight_dist_m` (직선) | 투영 좌표차 | `742` |
| `arc_path_dist_m` (경로) | arc 경로 합 | `980` |
| `hour, shift, is_night` | 이벤트 시각 | `11, Day, 0` |
| `active_tt_count` (혼잡) | 윈도우 내 이동 TT 수 | `231` |
| `route_density` (경로 혼잡) | 경로 arc상 TT 밀도 | `0.34` |
| `jobtype, laden` | GPS 미션 | `LD, 1` |
| **▶ label_travel_sec** | **dest_arrive − origin_depart** | `312` |

```js
// train_travel_time 샘플 1행
{ origin_point:"01U", dest_point:"03U-21", straight_dist_m:742, arc_path_dist_m:980,
  hour:11, shift:"Day", active_tt_count:231, jobtype:"LD", laden:1,
  label_travel_sec:312 }
```

### ② QC ETW(크레인 준비 시각) — `train_qc_etw`

한 행 = 크레인의 한 "결정 시점"에서 본 상태. 다음 핸드오버 준비까지 남은 초를 예측. 라벨은 다음 PICKUP/DROP(또는 다음 move ST_DT)까지의 시간.

| 컬럼 | 원천 | 예시 |
|---|---|---|
| `crane_id, jobtype` | PLC / MCH_OPER | `C46, DS` |
| `cur_move_elapsed_s` | MCH_OPER ST_DT / PLC | `48` |
| `move_dur_ewma_s` (최근 move 평균) | MCH_OPER 직전 N건 | `94` |
| `spreader_phase` (hpos/tpos 추세) | PLC | `"hoisting"` |
| `twin_tandem` | container2 / VSS | `twin` |
| `moves_done, planned_remaining` | MCH_OPER 누적 / VSS | `512 / 188` |
| `vessel_nqcr` (항차 평균 rate) | VSS_STATISTICS | `29.4` |
| `hour, queue_tt_count` | 시각 / GPS | `11, 2` |
| **▶ label_sec_to_ready** | **다음 PICKUP/DROP.time − 결정시각** | `73` |

### ③ 블록 핸드오버 시간(선적) — `train_block_handover`

한 행 = 선적 작업사이클에서 TT가 블록에 머문 사건. RTG PLC가 없으므로 **라벨은 TT 자신의 블록 체류 시간**(도착→적재완료)에서 직접 측정.

| 컬럼 | 원천 | 예시 |
|---|---|---|
| `block_id, point_type` | 매칭 | `Y_EB02, 2` |
| `rtg_id` (담당 RTG, 가능시) | 근접 RTG GPS / TOS ARMGC | `RTG58` |
| `rtg_busy, block_queue_tt` | RTG 상태 / 블록 내 TT 수 | `1, 2` |
| `container_size, twin` | container1/2 | `40ft, 0` |
| `hour, shift` | 시각 | `14, Day` |
| **▶ label_handover_sec** | **dest_work_end − dest_arrive** (블록) | `221` |

:::caution[③ 핵심 — 대기(③a)가 지배적이고 어렵다]
블록 체류시간(라벨 = `dest_work_end − dest_arrive`)은 **③a RTG를 기다린 시간 + ③b 실제 수령 시간**의 합인데, **③a가 크고 가변적**입니다. RTG는 여러 블록을 공유하며 PLC도 없어, 도착한 차를 **언제 받아줄지가 TT 측 신호만으론 예측되지 않습니다**(§7b의 양하 곧유휴 어려움의 근본 원인). 그래서 ③ 예측·양하 곧유휴는 **RTG 관여(GPS)·블록 큐 길이(`block_queue_tt`)·학습된 RTG별 서비스 간격**으로 ③a를 추정하는 데 달려 있고, 정확도 한계도 여기서 옵니다. 1차는 합산값, 2차로 ③a/③b 분해.
:::

## "곧 유휴" 차량 판별 — 선적은 가능, 양하가 문제

(TOS팀 합의 항목)

전역 매칭이 효과를 내려면(특히 교대 시점), 지금 작업 중이지만 **곧 비게 될 차량(곧 유휴)** 을 미리 풀에 넣어야 합니다(고객 자료 슬라이드 6·7: "마지막 핸드오버를 마무리 중인 차량 — 반응이 아니라 사전에 계획"). 그런데 **"곧 유휴"를 무엇으로·언제 판정하는지**는 TOS팀과 합의가 필요합니다. 결론부터: **선적(LD)은 우리 데이터로 판별 가능하지만, 양하(DS)는 구조적으로 어렵습니다.**

### 7b.1 왜 작업종류가 갈리나 — "차를 비우는 핸드오버"의 위치

TT를 **유휴로 만드는 것은 마지막 핸드오버**(차에서 컨테이너가 내려가는 순간)입니다. 그 핸드오버가 어디서 일어나는지가 관건입니다.

**그림 7b-1.** 선적은 유휴 전환이 **QC(PLC 보임)** 에서, 양하는 **블록 RTG(PLC 없음)** 에서 일어남. 같은 "곧 유휴"라도 관측 가능성이 정반대.

- **선적(LD)**: 차를 비우는 핸드오버 = 안벽 QC → **PLC 보임 ✓**. 야드 블록(RTG 적재, PLC 없음) → 이동(적재 상태) → 안벽 도착 → **QC가 차에서 픽오프** → ★ 유휴됨. 이 순간이 QC PLC PICKUP(차 측 들어올림)으로 1초 단위로 보임.
- **양하(DS)**: 차를 비우는 핸드오버 = 야드 블록 RTG → **PLC 없음 ✗**. 안벽 QC(QC 적재, PLC ✓·시작만) → 이동(적재 상태) → 블록 도착 → **RTG가 차에서 내림** → ★ 유휴됨. 이 순간은 RTG PLC가 없어 직접 신호 없음. GPS·예측만.

### 7b.2 신호 가용성 — 무엇으로 판별할 수 있나

| 신호 | 선적(LD) 유휴=QC | 양하(DS) 유휴=블록 RTG |
|---|---|---|
| **QC PLC** (~1초, ctab) | ✓ 직접 — 차에서 픽오프(PICKUP) + 스프레더 하강(hpos↓)으로 임박 예고 | 해당없음 — QC는 **적재 시작**만, 유휴 전환점 아님 |
| **RTG PLC** | — | ✗ 없음 — RTG는 PLC 미수신 → 직접 신호 부재 |
| TT GPS (~3초·성김) | △ 안벽 도착·상태(UNLOADING) | △ 블록 도착·상태 — 성겨서 늦게 감지 가능 |
| TT 미션 (`jobtype·topos1`) | ✓ 어느 QC로 가는지(상대 식별) | ✓ 어느 블록으로 가는지 |
| RTG GPS (~3초) | — | △ **인접 RTG 관여 단서**(같은 레인 위치) — 약하지만 유일한 블록측 실시간 단서 |
| TOS 작업이벤트 | 사후(검증/학습) | 사후 `YT_DIS_DT·ACTV_DT`(검증/학습) |

:::tip[선적은 풀린다]
선적 TT는 **QC PLC + TT GPS 도착 + ETW②** 로 "이 차가 N초 뒤 안벽에서 픽오프되어 유휴"를 **사전에·높은 신뢰로** 판정할 수 있습니다. 사용자 직관대로입니다.
:::

### 7b.3 양하 문제와 해법 후보

:::caution[진짜 어려움 — POW 도착 ≠ 곧 유휴 (현장 핵심)]
양하 TT가 **POW(작업위치)에 도착(`ARRIVED`)했다고 해서 곧 유휴가 되는 게 아닙니다.** 그 차를 비워주는 주체는 **RTG**인데, RTG는 **여러 블록·레인을 공유**하며 다른 작업 중일 수 있고, 도착한 이 차를 **바로 받아주지 않을 수 있습니다**. 게다가 RTG는 **PLC 신호가 없어** "언제 이 차를 받을지(=대기 시간)"가 도착 시점엔 알 수 없습니다. 즉 **도착~유휴 사이의 대기가 크고 가변적**(크레인 큐 대기가 시간대별 7~11분으로 출렁이는 것과 같은 성격 §9.2)이라, **도착만으로 곧유휴 판정하면 오판**입니다. 따라서 양하의 관건은 "도착 감지"가 아니라 **RTG가 실제로 이 차를 받으러 왔는가(RTG 관여)와 그 대기 시간 예측**입니다.
:::

그래서 양하 곧유휴는 **"RTG 서비스 시각"을 예측**하는 문제(QC ETW②의 RTG판, 단 PLC 없음·다중블록 공유로 더 어려움)로 귀결됩니다. 네 가지 보완책:

| 안 | 방법 | 장점 | 약점 |
|---|---|---|---|
| **B** (핵심) | **RTG 관여 확인(GPS)**: 배정 RTG(TOS `ARMGC`)/인접 RTG의 GPS가 **이 차의 레인에 와서 작업 중**일 때만 곧유휴로 본다. RTG가 멀거나 다른 블록이면 **아직 아님**. | RTG 대기 병목을 직접 반영, 오판 방지 | RTG GPS 성김·단일점 → 짧은 호라이즌(관여 후에만) |
| **A** | **RTG 서비스대기 예측**: 블록 큐 길이(앞선 TT 수)·RTG 위치/가동·학습된 블록·RTG별 서비스간격으로 `대기③a + 핸드오버③b`를 예측 | 관여 전에도 추정 시도 가능 | **③a(RTG 대기)가 크고 불확실** — 정확도 한계의 핵심 |
| **C** | **근실시간 TOS 확인**: `JOB_ORDER_HISTORY` 완료(`ACTV_DT`) 저지연 감지 → "방금 유휴" 확정 | 권위적·정확 | 예측 아닌 **확인**(호라이즌 0), Oracle 폴링 부하·지연 |
| **D** | **TT-GPS 단독**: 블록 도착만으로 판정 | 추가 데이터 불필요 | **부적합 — 도착≠임박(RTG 대기 무시).** 반드시 B와 결합해야 |

:::caution[권장 조합]
**선적** = QC PLC 기반 — 높은 신뢰·긴 호라이즌으로 **지금 바로** 채택(§7b.5 실증). **양하** = **도착(필요조건) + B(RTG 관여 확인)이 충족될 때만** 곧유휴로, **짧은 호라이즌·낮은 신뢰**. RTG가 아직 관여 안 했으면 대기가 불확실하므로 **풀에 넣지 않음**(오포함 방지). 대기시간 예측(A의 ③a)과 호라이즌·임계는 **1단계 record-only**로 보정·합의.
:::

### 7b.4 TOS팀과 합의할 정의안 (제시)

| 합의 항목 | 제시안 |
|---|---|
| 유휴 전환 정의(종류별) | 선적=**QC가 차에서 픽오프** · 양하/이적=**블록에서 컨테이너 내림**. 이 시점에 TT를 "유휴"로 본다. |
| "곧"의 호라이즌 H | 선적 **90~120초**(QC 진행으로 길게 예측) · 양하 **RTG 관여 확인 후에만 ~30초**(도착 시점이 아니라 RTG가 이 차를 받기 시작한 뒤). 풀에 넣는 시점 = 예측 유휴까지 ≤ H. |
| 신뢰 임계 τ | 예측 신뢰도 ≥ τ인 차량만 풀 포함(오포함 시 매칭 왜곡). 양하는 τ를 높게. |
| 초기 적용 범위 | **1단계/초기 2단계: 선적만 곧유휴** 적용, 양하는 현재 유휴만(또는 낮은 가중). 데이터 축적 후 양하 확대. |
| 측정·검증 지표 | record-only로 **예측-유휴 vs 실제-유휴**의 시각 MAE, precision/recall(조기·과다 포함률) 합의. KPI 세션 §18 변수와 동일 층화. |
| 안전장치 | 곧유휴는 **계산에만** 포함, 실제 배차는 **유휴 순간 재실행**해 그때 확정(고객 자료 슬라이드 7). 오판정해도 미리 명령 안 내리므로 피해 없음. |

### 7b.5 실증 — 지금 데이터로 판별되는가 (120초 라이브 캡처, 2026-06-08)

두 zone(GPS+PLC)을 동시에 120초 캡처해, 위 논리가 **실제 신호로 성립하는지** 측정했습니다. 결론: **핵심 신호가 모두 살아 있고, 선적 유휴는 QC PLC로 즉시 포착됩니다.**

- **205 / 109** — 적재 / 유휴 TT (총 314). `container1` 차있음=작업중 / 빔=유휴 — **유휴 여부 직접 지표 성립**.
- **13 건** — 120초 내 실제 **유휴 전환**(container1 차→빔) 관측. 그중 **7건이 안벽(QC), 6건이 블록**.
- **±1~4 초** — QC 유휴전환 ↔ **QC PLC PICKUP** 시각차(3건 매칭: −4.4·+1.9·−2.3s). **크레인 신호로 정확 포착**.

#### 관측된 유휴 전환 — 안벽(QC)은 PLC로 잡히고, 블록은 신호 없음

| TT | 작업 | 목적지 topos1 | 유형 | QC PLC 매칭 |
|---|---|---|---|---|
| TT1214 | LD | `C55` (안벽) | QC | ✓ C55 PICKUP Δ−4.4s |
| TT984 | LD | `M4` (안벽) | QC | ✓ M4 PICKUP Δ+1.9s |
| TT1177 | LD | `M2` (안벽) | QC | ✓ M2 PICKUP Δ−2.3s |
| TT1154 · TT1156 · TT1314 | LD | `M2 · Z6 · Z4` (안벽) | QC | 해당 크레인 PLC 캡처 구간 밖(원리는 동일) |
| TT921 · TT914 · TT952 · TT1331 | DS | `01WW… 01LW… 08W… 08L…` (블록) | 블록 | RTG PLC 없음 → GPS만(`arrival=ARRIVED`) |

:::tip[실증 결론 — 선적은 지금 가능]
① `container1`이 **유휴 여부의 직접 지표**로 성립(205 vs 109). ② 적재 TT가 **안벽 도착(`arrival=ARRIVED`) → 유휴**까지 리드타임 **중앙값 ~60초(범위 30~96초)** — 즉 **약 1분 전에 "곧 유휴" 사전 판정 가능**. ③ 그 유휴 순간이 **QC PLC PICKUP과 ±수 초로 일치**(3건 실증) → 선적/하역계는 **크레인 신호로 정확·사전 포착**. **이 로직은 우리 API가 이미 파싱하는 필드(container1·jobtype·topos1·arrival + QC PLC)만으로 바로 구현 가능**합니다.
:::

:::caution[양하는 실증에서도 확인된 약점 — ARRIVED만으론 부족]
관측된 DS 유휴전환(TT921·914·952·1331)은 모두 **블록에서, RTG PLC 없이** 일어났습니다. 네 건 모두 직전 `arrival=ARRIVED`가 떠 있었지만, **도착했다고 곧 유휴인 것은 아닙니다** — RTG가 이 차를 **바로 안 받을 수 있고**, 그 대기는 크고 가변적입니다(§7b.3). 따라서 양하는 "도착 감지"가 아니라 **RTG가 이 차의 레인에 와서 받기 시작했는가(RTG 관여, 안 B)를 확인한 뒤에만** 짧은 호라이즌으로 곧유휴 판정해야 합니다. 도착~RTG관여 사이의 대기시간 예측 정확도는 1단계 record-only로 보정.
:::

#### 지금 구현 가능한 판별 로직 (의사코드)

```js
// 실시간 gps_update 스트림에서 TT 상태 유지 → 매 틱 판정
busy(tt)      = tt.container1 != ""              // 적재중 = 작업중
at_handover   = tt.arrival == "ARRIVED" && near(tt.cur_loc, tt.topos1)
dest_is_crane = /^[CMZ][0-9]+$/.test(tt.topos1)  // 안벽 QC/MHC/SC

if busy(tt) && at_handover:
    if dest_is_crane:                            // 선적/하역계 — 높은 신뢰
        soon_idle(tt, horizon=90, conf=HIGH,
                  confirm = QC_PLC[tt.topos1].imminent_PICKUP)  // 스프레더 하강/곧 픽오프
    else:                                        // 양하 블록 — 도착만으론 부족!
        rtg = assigned_or_nearest_RTG(tt)            // TOS ARMGC / 인접 RTG GPS
        if rtg.at_lane(tt) and rtg.active:        // RTG가 와서 받기 시작해야
            soon_idle(tt, horizon=30, conf=MED)  // 관여 후에만 짧게
        else:
            pass                                 // RTG 미관여 → 대기 불확실 → 풀에 안 넣음
// 실제 배차는 container1=="" 되는 순간(=확정 유휴) 재실행해 확정 — 오판정 피해 없음
```

### 7b.6 양하 해법 실증 — RTG·TT GPS 근접(같은 bay)으로 "RTG 관여" 판별

양하의 관건은 "RTG가 이 차를 받으러 왔는가"였습니다(§7b.3-B). RTG도 GPS를 보내므로, **핸드오버는 두 장비가 같은 bay에 있어야 한다**는 물리 제약을 GPS 거리로 직접 확인할 수 있습니다. 같은 120초 캡처(RTG 68대·TT 271대)로 측정했습니다.

- **2 m** — RTG GPS 정확도 중앙값(p90 12m·max 31m) → **bay 수준 판별에 충분**.
- **133 m** — 도착한 양하 TT의 **최근접 RTG 거리 중앙값**. 6건 중 4건이 100~172m — **도착해도 RTG가 멀리 있어 대기 중**.
- **0 / 6** — 도착 양하 TT 중 RTG가 **25m 이내(관여)** 인 건. 대부분 RTG 부재 = "ARRIVED≠곧유휴" 실증.

#### 도착한 양하 TT — 최근접 RTG 거리 (RTG 멀면 = 아직 대기)

| TT | 목적지 bay(topos1) | 최근접 RTG | 거리 | 판정 |
|---|---|---|---|---|
| TT1084 | `02N-09` | RTG191 | 28 m | 중간(접근 중?) |
| TT1298 | `07Q-0304` | RTG109 | 32 m | 중간 |
| TT1250 | `08E-0809` | RTG196 | 129 m | RTG 멀다 → 대기 |
| TT1129 | `10Y-2324` | RTG188 | 134 m | RTG 멀다 → 대기 |
| TT1150 · TT1197 | `07Q-1920 · 01GW-04` | RTG182 · RTG102 | 157·172 m | RTG 멀다 → 대기 |

:::tip[실증 결론 — RTG 근접도가 답이다]
RTG GPS는 정확(중앙값 2m)하고, **RTG–TT 거리가 "관여(≈같은 bay, ~25m 이내)" vs "대기(100m+)"를 또렷이 가릅니다**. 스냅샷에서 도착 양하 TT의 **대부분이 RTG 100m+ 떨어진 "대기" 상태**였습니다 — 이것이 "도착=곧유휴"가 틀리는 이유의 직접 증거이자, 동시에 **해법**입니다: **양하 곧유휴 = 도착 + (인접/배정 RTG가 같은 bay에 근접)** 일 때만 판정. **정밀화**: 거리 대신 RTG GPS를 **bay로 매핑**해 TT의 `topos1` bay와 일치하는지 보면 측방 오프셋까지 정확.
:::

:::caution[주의 — 비는 시점(container1)은 정밀 시각이 아님]
유휴 전환(container1 차→빔) 순간의 RTG 거리는 31~255m로 들쭉날쭉했습니다. 즉 `container1`이 비는 시점은 물리적 핸드오버와 정확히 일치하지 않습니다(시스템 상태 갱신 지연 등). 따라서 **라이브 판정**엔 RTG 근접도를 쓰되, **학습용 정밀 라벨**(언제 정확히 비었나)은 TOS `ACTV_DT`·RTG 근접 구간으로 보강합니다.
:::

### 7b.7 작업 싸이클·현재 단계 실시간 감지 — "마지막 단계"인지 알 수 있나

핸드오버가 일어난다고 다 유휴가 아니라, 그것이 **싸이클의 마지막 단계**(이후 차가 비는)여야 곧유휴입니다. 그래서 "이 TT가 지금 어느 싸이클의 어느 단계인가"를 실시간으로 알아야 합니다. **실증 결과 — 가능합니다.** 한 싸이클은 **빈차→픽업→적재→드롭→빈차**로 단순하고, **적재 레그가 정확히 1개**라 그 드롭이 곧 마지막 핸드오버입니다.

**그림 7b-2. TT 작업 싸이클 상태기계 — 실시간 신호로 단계 식별 (적재 레그 1개 → 드롭=마지막).** 실시간 단계 분류(295대 전수 분류 성공): 대기 106 · 픽업行 36 · 드롭行 103 · 도착-핸드오버 50. "곧유휴 후보"는 드롭行(근접) + 도착-핸드오버.

```
대기 IDLE ─[배정]→ 픽업行(공차) ─[적재(픽업)]→ 드롭行(적재) ─[도착]→ 도착-핸드오버(최종) ─[크레인이 받음]→ ★ 유휴
(container1=빔·저속)  (빔·이동·작업배정)   (container1=참·이동)   (적재·ARRIVED·크레인 관여)
```

#### 실측 싸이클 추적 예시 (라이브에서 단계 전이가 보인다)

```
// TT1296 (양하) — 싸이클 시작 포착
t=  0s  대기 IDLE          빈차              job=DS →C40        @WHARF_20_A
t=134s  픽업行(공차)        빈차              job=DS →C40        (QC C40로 픽업하러)
t=137s  드롭行(적재)        적재[OOCU7743]     job=DS →09X-4041   // QC서 받음, 목적지=09X 블록

// TT1292 (양하) — 싸이클 종료(유휴 전환) 포착
t=  0s  도착-핸드오버(최종)  적재[ONEU0135]     job=DS →10C-0607   arr=ARRIVED  @09C
t= 64s  대기 IDLE          빈차              job=(빔) →(빔)            // 비는 순간 작업필드도 클리어 = 종료 확정
```

| 현재 단계 | 실시간 신호(gps_update) | 곧유휴? |
|---|---|---|
| 대기 IDLE | `container1` 빔 · 저속 | 이미 유휴 |
| 픽업行(공차) | 빔 · 이동 · `jobtype/topos1` 배정됨 | 아니오(싸이클 시작) |
| 드롭行(적재) | `container1` 참 · 이동 | 드롭지점 근접 시 후보 |
| **도착-핸드오버(최종)** | 참 · `arrival=ARRIVED` · 목적지 도달 | **예 — 크레인 관여 시** |
| 유휴 전환 | `container1`(트윈이면 +`container2`) 참→빔 | = 확정 유휴 |

:::caution[트윈/탠덤 — 마지막 판정 주의]
실측상 적재 TT의 **20%가 트윈/탠덤(컨테이너 2개)** 입니다(예 TT1066 `BSIU3476+FCIU31`). 한 번에 둘 다 내리면 한 핸드오버가 마지막이지만, 순차면 `container1`만 비고 `container2`가 남습니다. 따라서 "마지막 핸드오버=유휴"는 **모든 컨테이너 필드가 비는 것**으로 판정해야 합니다.
:::

:::tip[로직 종합 — "곧유휴 = 마지막 단계 + 크레인 관여"]
**곧유휴(tt)** = (현재 단계 ∈ {도착-핸드오버, 드롭行 막바지}) **AND** (모든 컨테이너가 곧 비움 = 단일적재 또는 트윈 동시하역) **AND** (크레인 관여: 선적→**QC PLC 임박 PICKUP** / 양하→**RTG 같은 bay 근접**). 단계는 `container1/2`+위치+`arrival`로, 크레인 관여는 PLC·RTG GPS로 — **전부 지금 수신 중인 신호**. 견고화: `topos1`은 픽업↔드롭에 따라 동적으로 바뀌어 보조로 쓰되, 단계 확정은 참조 프로젝트의 지도매칭+상태기계(WorkCycleBuilder)를 **온라인**으로 재사용.
:::

:::note[핵심]
"곧 유휴"는 **① 작업 싸이클 단계(드롭行/도착-핸드오버 = 마지막 적재 레그) + ② 모든 컨테이너 비움 임박 + ③ 크레인 관여(QC PLC / RTG bay 근접)** 의 결합 판정입니다. 세 가지 모두 **지금 수신하는 GPS/PLC로 실시간 감지 가능함을 실증**했고(단계 분류 295대 전수·싸이클 전이 추적·RTG 근접 분리·QC PLC 매칭), 남은 것은 **정의·호라이즌·신뢰 임계를 TOS팀과 합의**하고 1단계 record-only로 정확도를 보정하는 일입니다.
:::

## 피처 카탈로그

세 예측기가 공유하는 피처 풀입니다. 모두 우리 raw에서 파생 가능하며, 누설(미래 정보 사용)이 없도록 "결정 시점까지의 정보"만 씁니다.

| 그룹 | 피처 | 원천 |
|---|---|---|
| **공간** | 출발/도착 블록·포인트·point_type, 직선거리, arc 경로거리, 안벽/블록 구분 | 투영 + 매칭 + `topos1` |
| **시간** | 시각(hour), 교대(N/D/E), 주중/주말, 교대 경과시간 | 이벤트 시각 |
| **혼잡** | 활성 TT 수, 경로 arc상 TT 밀도, 블록/안벽 큐 길이 | 실시간 GPS 집계 |
| **크레인** | 현재 move 경과, 최근 move 소요 EWMA, 스프레더 단계(hpos/tpos), 트윈/탠덤 | PLC + MCH_OPER + VSS |
| **작업맥락** | jobtype, 적재여부, 선박/항차, 남은 계획물량, 항차 NQCR | GPS 미션 + VSS |
| **장비** | 장비 종류/세대, (옵션) 운전자·연료 | device + GPS |

:::caution[누설 방지]
라벨은 "미래"(도착·핸드오버)에서 오지만, 피처는 반드시 **배차 결정 시점 이전** 정보만 사용해야 합니다. 학습 표본 생성 시 각 행에 "결정 시각 t₀"를 명시하고, t₀ 이후 값(예: 실제 도착 후의 혼잡도)은 피처에서 배제합니다.
:::

## 실데이터로 본 현황 — 왜 이 예측이 가치 있나

실측 18일치(2026-05-21~06-07, 우리 extractor가 적재)로 2단계가 겨냥하는 손실을 확인합니다.

### 9.1 공차 비율 — TT 이동의 절반이 빈 차

**그림 9-1.** 작업종류별 공차 비율(공차거리 ÷ 총거리, 최신일). 평균 **공차 1180~1980m/작업**. 이동의 절반이 빈 차 → 1단계가 직접, 2단계가 시간가중으로 줄이는 대상.

| 작업종류 | 공차 비율 |
|---|---|
| DS 양하 | 51% |
| LD 선적 | 50% |
| MO 이적 | 53% |
| 전체 ≈ | ~50% |

### 9.2 크레인 대기 — 시간대별로 7~11분 출렁임

**그림 9-2. 시간대별 크레인 큐 대기(중앙값, 초) — 18일 평균.** 크레인이 차량을 기다리는 시간(YT 하차→크레인 활성). 정오·심야에 최고(예: 12h 641s, 16h 439s). **이 출렁임이 동기화 부족의 신호**이며, ETW 예측으로 줄이려는 핵심 손실(가중치 최대).

### 9.3 작업 사이클 — 종류별로 크게 다름(피처의 근거)

**그림 9-3.** 작업종류별 컨테이너 사이클 시간(중앙값, 분). 컨테이너 전체 사이클(작업 첫~마지막 이벤트). 예측 대상인 "이동 구간"은 이보다 짧지만, 종류별 편차가 큰 만큼 **jobtype·규격·선박**이 핵심 피처임을 보여줌.

| 작업종류 | 사이클 시간(중앙값, 분) |
|---|---|
| LD | 64 |
| DS | 51 |
| MO | 48 |
| GO | 29 |
| MI | 15 |

- **~450 대** — 실시간 추적 장비(TT~280·RTG~100·QC~28). GPS 장비당 ~3s·불규칙 / PLC ~1s.
- **~50 %** — TT 이동 중 공차 비율 — 2단계 1차 절감 대상.
- **7–11 분** — 시간대별 크레인 큐 대기 중앙값 — ETW 예측의 표적.

## 데이터 품질 · 함정 (반드시 처리)

| 이슈 | 영향 | 대응 |
|---|---|---|
| GPS 지터·튐 (RMS ~70m, accuracy 수십~수백 m) | 지도매칭 오류 → 이동시간 라벨 오염 | 칼만 보정 + accuracy 임계(≤30m)에서만 핸드오버 상관, 점프 필터. |
| **불규칙·성긴 GPS 주기** (장비당 ~3초, 정지 시 30~55초; 1Hz 아님) | 궤적 해상도 낮음 → 도착/출발 시각 ±수초 오차, 30초 갭 분할과 충돌 시 트립 과분할 | 분할 임계를 실측 분포에 맞춰 상향(예: 60s)·체류기반 보강, 라벨 시간해상도(±수초)를 모델 평가에 반영. PLC(~1s)는 핸드오버 순간 포착에 충분. |
| 시각 정합 — `dtime`은 MYT·초단위·날짜 없음 | 서버(KST)↔터미널(MYT) 8h, 자정 넘김 | 수신시각 기준 보정 + 자정-wrap 휴리스틱(참조 normalizer). 라벨 계산은 단조 시각으로. |
| TOS 보존 15일 | 장기 학습 불가 | **매일 스냅샷 적재**(이미 가동) — 1단계 전 구간 누적. |
| RTG PLC 미수신 가능 | 블록 핸드오버(③) 직접 신호 없음 | TT 블록 체류로 라벨링(§7-③) + 근접 RTG GPS로 대기 추정. |
| 스왑·취소·재배차 | "의도한 도착"과 실제 불일치 | 완료된 핸드오버(세 신호 일치)만 표본; 미완 트립 제외. |
| 트윈/탠덤(2컨테이너) | 핸드오버 2회 → 소요 다름 | `container2`로 분리 피처화, 라벨 별도. |
| 희소 경로(드문 지점쌍) | 과적합·표본 부족 | 블록 프리픽스 집계 폴백, 거리 기반 사전(prior)으로 평활. |
| 분포 이동(교대·물량·기상) | 일반화 저하 | 교대/시간/물량 층화 검증, 외부변수 기록(KPI 세션 §18과 동일 변수). |

## 모델링 · 검증 접근 (개요)

- **점진적 복잡도** — **v0 베이스라인**: (출발,도착)별 과거 중앙값 / 거리÷평균속도. **v1**: 그래디언트 부스팅(LightGBM류)으로 혼잡·시각·작업맥락 반영. **v2**: 필요 시 시퀀스/그래프 모형. 항상 v0 대비 개선분을 입증.
- **정직한 검증** — **시간 분할**(미래 교대로 검증, 누설 차단). 지표는 **MAE(초)**·P90 오차. 층화: 주/야·물량·작업종류. 배차 품질은 **보조 시뮬레이터**(KPI 세션 §18)로 TOS/AI 교차.
- **record-only 자가 라벨** — 1단계 운영 중 파이프라인을 돌려 **예측 vs 실제**를 매 배차마다 기록 → 라벨이 자동 누적되고 모형 정확도를 상시 모니터(대시보드 FEED/HEALTH 연계).
- **폴백 안전** — 예측 신뢰도 낮음(희소·고분산)·모형 장애 시 **1단계 거리비용으로 즉시 복귀**. 2단계는 1단계 위에 얹는 가중치이므로 투자 보호.

## 데이터 볼륨 · 일정 정합

1단계 운영(8/17–10/23, 약 10주)이 곧 2단계 학습데이터 생산 기간입니다. 일정이 맞물립니다.

| 항목 | 추정 | 근거 |
|---|---|---|
| 원시 GPS 표본/일 | 약 **250~350만** | 실측 ~40건/초(장비당 ~3초, 1Hz 아님) × 가동시간. PLC는 별도 ~140만/일. |
| 완료 작업사이클(라벨)/일 | 수 **만 건** | QC당 ~400–700 move(예: C46 719) × 가동 QC + 야드 |
| 10주 누적 학습표본 | **수십만~백만+** | 그래디언트 부스팅에 충분, 희소 경로도 일부 커버 |
| 크레인 핸드오버 이벤트/일 | 수 **만 건** | PLC PICKUP/DROP(C/M/Z) + TOS move |

:::tip[일정]
8월말~10월초 모형 학습·정확도 검증 → 10월 중순 record-only 전환검증(공동 판정) → **10/30 시간비용 전환**. 본 연구의 학습 테이블은 **1단계 가동 즉시 채워지기 시작**하도록 지금 파이프라인을 준비하는 것이 관건입니다.
:::
