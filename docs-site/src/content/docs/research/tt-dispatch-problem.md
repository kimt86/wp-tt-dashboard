# Part 01 — 현행 TOS 로직 vs. AI TT 로직

현재 시스템이 하는 일, 놓치고 있는 가치, 그리고 AI 엔진이 다르게 하는 점.

## 현행 배차 방식

### 현재 규칙

- 차량이 유휴 상태가 되면 TOS가 **즉시** 작업을 할당합니다 — 1대당 1건.
- 해당 차량 한 대 기준으로 보통 **가장 가까운 작업**을 선택합니다.
- 차량 위치는 **DGPS** 또는 직전 작업 완료 지점에서 가져옵니다.
- 전체 차량을 하나의 통합 풀로 관리하며 크레인 밸런스 규칙을 적용합니다.

### 사후 보정 — 재배차 스왑

- TOS는 **공차 거리가 과도하게 긴 배차**를 기억해 둡니다.
- 이후 **더 가까운 유휴 차량**이 생기면, 그 작업을 더 나은 차량으로 **교체(스왑)**해 상황을 개선합니다.
- 즉, 처음엔 어쩔 수 없이 멀리 배차하더라도 **나중에 기회가 생기면 고치는** 방식입니다.

:::note
이를 **탐욕(greedy)** 방식이라 합니다: **한 번에 한 대씩** 결정하고, 부족한 부분은 **사후 스왑으로 보정**합니다.
:::

## 왜 "한 대씩"은 최선을 놓치는가

:::caution
각 차량의 최선 작업을 **따로** 고르면 **터미널 전체**의 최선 결과를 놓칩니다. 같은 예시(차량 2대·작업 2건)를 두 방식으로 비교합니다.
:::

**예시 — 차량 A(유휴), 차량 B(곧 유휴), 작업 1·작업 2.** 선 위 숫자는 **공차 이동 거리**(차량이 빈 채로 이동하는 거리)입니다.

| 방식 | 차량 A | 차량 B | 총 공차 이동 |
| --- | --- | --- | --- |
| ① 탐욕 — 한 대씩 선택 | → 작업 1 (100) | → 작업 2 (900) | **1000** |
| ② 전역 매칭 — 함께 선택 | → 작업 2 (400) | → 작업 1 (300) | **700** |

A가 자기 기준 가장 가까운 **작업1(거리 100)**을 먼저 가져가면, B는 멀리 떨어진 **작업2(거리 900)**로 밀려 총 1000. 둘을 함께 보면 **A→작업2, B→작업1**로 총 **700 — 약 30% 감소**. 같은 차량·같은 작업인데도 결과가 달라집니다.

:::caution
이 격차는 **교대 시점**에 가장 커집니다 — 400~500대가 동시에 배차를 필요로 할 때입니다.
:::

## AI 접근: 전체를 함께 매칭

:::tip
**핵심 아이디어:** **모든 차량 × 모든 후보 작업**을 동시에 보고, 총 비용이 가장 낮은 조합을 선택합니다. 이 계산은 **차량 풀이 바뀔 때마다**(유휴·곧 유휴 차량이 추가될 때) 다시 실행됩니다.
:::

### 한 번의 배차 라운드 흐름

`차량 풀 변경(트리거)` → `차량 풀` → `작업 풀` → `비용 행렬` → **`전역 매칭`** → `적용`

**차량 풀**

- `현재 유휴` 현재 작업이 없는 차량.
- `곧 유휴` 마지막 핸드오버를 마무리 중인 차량 — 반응이 아니라 사전에 계획.
- `스왑 가능 공차` 작업을 향해 공차 이동 중이나 아직 미상차 — 재매칭(스왑) 대상.

**작업 풀**

- 각 크레인의 **ETW(작업 준비 예상 시각)** 기준, **시급한 순서**대로 후보 작업 추출.
- **차량 풀 크기만큼** 채워서 모든 차량이 매칭 후보를 갖도록 구성.

:::note
매칭은 검증된 알고리즘(헝가리안 / Kuhn–Munkres)을 사용합니다. 500×500 규모에서 약 **35 ms**에 결과를 반환합니다.
:::

## 계산은 전체, 실제 할당은 유휴 차량만

:::tip
**가장 중요한 원칙:** 매칭 계산은 **차량 풀 전체**에 대해 수행하지만, 실제 배차 명령은 **지금 유휴인 차량에게만** 내립니다. 곧 유휴·스왑 가능 차량까지 계산에 포함하는 이유는, 유휴 차량의 작업 선택이 터미널 전체 관점에서 왜곡되지 않게 하기 위해서입니다.
:::

| 차량 유형 | 계산 | 실제 동작 |
| --- | --- | --- |
| `현재 유휴` | 포함 | 매칭 결과를 **즉시 실제 배차** |
| `곧 유휴` | 포함 | **미리 할당하지 않음** — 유휴가 되는 순간 로직을 다시 수행해 그때 실제 할당 (가장 최신 상황 기준) |
| `스왑 가능 공차` | 포함 | 기존 작업과 **다른 작업**이 매칭되면 **스왑 수행** |

### 유휴 차량이 바로 작업을 못 받을 수도 있습니다

- **작업 풀 크기가 차량 풀 크기보다 작으면**(대기 작업 부족), 일부 유휴 차량은 그 라운드에 작업을 받지 못할 수 있습니다.
- 이 경우 **N초마다 반복 실행**해 재할당하도록 할 수 있습니다 `협의 필요`
- 현장 병목 회피를 위해 유휴 차량의 **임시 이동**이 필요하다면, 관련 **차량 운영 정책 지원**이 필요합니다 `협의 필요`

## "비용"의 의미 — 쉽게 풀어서

엔진은 각 **(차량 → 작업)** 짝마다 '비용' 점수를 매기고, **비용이 가장 낮은 조합**을 고릅니다. 비용이 낮다 = 더 좋은 배차.

**1단계 — 거리만 본다**

공차로 이동하는 거리 = 비용. **거리가 짧을수록 비용 ↓** = 더 좋은 배차. 예측 불필요 — 위치·시각만 사용.

**2단계 — 시간도 더한다**

**기다린 시간**이 길수록 비용 ↑. 가중치가 큰 순서:

| 대기 유형 | 상대 비중 |
| --- | --- |
| 크레인이 차량을 기다림 | 가장 비쌈 |
| 차량이 크레인을 기다림 | 중간 |
| 공차로 달린 시간 | 가장 낮음 |

크레인이 노는 것이 가장 큰 손해 → 가장 무겁게 반영.

:::tip
1단계는 **예측이 필요 없습니다** — 확정된 사실(차량 위치(DGPS), 작업 지점, 현재 시각)만 사용. 그래서 첫 단계의 리스크가 낮습니다.
:::

## TOS vs. AI TT 로직

| 항목 | 현행 TOS | AI TT 로직 |
| --- | --- | --- |
| 의사결정 범위 | 한 번에 한 대 (탐욕) | **모든 차량·작업을 함께** |
| 목표 | 차량별 가장 가까운 작업 | **터미널 전체의 최저 총비용** |
| 타 차량 고려 | 스왑으로 사후 보정 | **매칭에 내재** |
| 곧 유휴될 차량 | 미고려 | **풀에 포함** |
| 긴 공차 배차 보정 | 사후 스왑 — 긴 공차 배차를 기억했다가 더 가까운 유휴 차량이 나오면 교체 | **매 라운드 전역 재매칭에 처음부터 반영** |
| 교대 시점 | 가장 취약 | **가장 큰 효과** |
| 후보 작업 풀 | 각 크레인 시퀀스의 선두 작업만 | **크레인 ETW 기준 시급한 순서**로 차량 풀 크기만큼 확장 (크레인 내 작업 순서는 그대로 존중) |

:::tip
AI 로직은 TOS의 **잘 작동하는 부분은 그대로 유지**하고, 의사결정 방식만 한 대씩 → 터미널 전체로 교체합니다.
:::

## UAT — 문제 없이 동작하는지 공동 확인

UAT 약 2주간 AI는 TOS에 실연동된 상태로 동작하되 **배차 결정은 반영하지 않고 기록만** 합니다. 목적은 알고리즘 성능 평가가 아니라 **"실운영에 넣어도 안전한가"**의 확인이며, Westports와 CLT가 **공동으로** 점검·판정합니다. 정확한 수치는 실측 데이터 첫 주를 바탕으로 합의합니다.

| 범주 | 합격 기준 (예시) |
| --- | --- |
| A. 시스템 건전성 | 엔진 가용성 ≥ 99.5% · 배차 응답 p99 ≤ 100ms · 크래시·메모리 누수 0건 · 데이터 유실률 ≤ 0.1% |
| B. 알고리즘 건전성 | AI 배차 결과의 TOS 제약 충족률 100% (불가능 배차 0건) · 이상 배차 패턴 없음 |
| C. 운영 준비 | 대시보드 정상 동작 · 알림·폴백 runbook 검토 · 장애 대응 절차 리허설 · 공동 체크리스트 서명 |

:::note
**합격 = A · B · C 전 항목 달성.** 미달 항목은 UAT 기간 내 보완 후 재점검합니다. 알고리즘의 **효과**는 별도로 측정·판정합니다 (Part 03).
:::

**8/17부터** UAT 합격을 전제로, AI 배차가 **실제 운영에 반영**되기 시작합니다. 거리 기반 전역 매칭으로 운영하면서 효과를 실측하는 단계입니다.

| 항목 | 내용 |

| 운영 범위 | **본선 풀 한정**으로 AI 배차 활성화 — 구내이적·하우스키핑은 기존 TOS 정책 유지. 이상 시 **TOS 자동 폴백** |
| 효과 측정 | 베이스라인 대비 공차 이동 거리·크레인 대기·사이클 타임 실측 (Part 03 기준), 효과 중간 리포트 공유 |

:::tip
1단계는 그 자체로 가치(공차 이동 감소)를 내면서, 동시에 2단계 고도화에 필요한 데이터를 만듭니다.

# Part 03 — KPI & 개선 효과 측정

로직과 직결된 명확한 지표, 고객사 데이터로 합의하는 공정한 목표, 그리고 정직한 비교 방법.

## KPI 계층 구조

핵심 질문: *"크레인과 차량이 서로를 덜 기다리게 되어, TT 사이클이 빨라졌는가?"*

| 계층 | 지표 | 의미 |
| --- | --- | --- |
| **최상위 결과** | TT 사이클 타임 | 배차 → 다음 배차 준비까지 |
| **주 지표** (1단계) | 작업당 공차 이동 거리 / 비율 | 거리 기반 로직의 직접 효과 `주 가설` |
| 1차 | QC 크레인 대기 시간 | 크레인이 차량을 기다리는 시간 |
| 2차 | 차량 대기 / 배차 레이턴시 | 차량이 기다리는 시간 / 배차 지연 |
| 참고 | QC 시간당 처리량 (MPH) | 관측만 (판정 제외) |

:::note
1단계는 **거리**를 최소화하므로 직접 결과는 **작업당 공차 이동 거리** — 이것이 주 가설 지표입니다. **QC 크레인 대기 시간**은 핵심 운영·경제 가치 지표입니다.
:::

## 효과 비교 — 본 비교 + 보조 비교

### 본 비교: 도입 전/후 실측 (판정 기준)

`측정 설계 사전 확정` → `베이스라인 4주 (KPI + 외부 변수 기록)` → `도입 후 4주+ (동일 기준 기록)` → **`층화·보정 + 통계 검정`**

### 측정 설계 제시안 `샘플 — 베이스라인 시작 전 공동 합의`

| 항목 | 제시안 |
| --- | --- |
| 분석 단위 | **교대(shift) 단위 터미널 전체 집계** — 글로벌 풀링은 전체 차량을 하나의 시스템으로 운영하므로 측정도 터미널 전체 기준이 정확. 항차·QC별 분석은 드릴다운용 보조 |
| 제외 기준 | 크레인 고장·악천후·안전훈련 등 **비정상 이벤트 교대** — 현장의 사전·즉시 공유 협조 필요 |
| 기록 변수 | 교대별 물량(TEU) · 가동 QC 수 · 규격 혼합(20/40ft·twin) · 주/야간 · 고장/정비 시간 · 기상 · 외부 트럭 비율 |
| 표본 규모 | 4주당 교대 약 80개+ · 베이스라인의 평균·변동 폭으로 **필요 표본 수 사전 계산** (Power Analysis) |
| 판정 방법 | 주 가설 = **작업당 공차 이동 거리 감소** · 교대 평균 비교 · p<0.05 |

:::note
**보조 비교** — 실운영 이력으로 만든 **동일 시나리오**를 같은 시뮬레이터에서 TOS·AI 로직으로 각각 운영해 비교. 운영 전환 전 사전 추정과 본 비교의 교차 확인에 사용.
:::

## 성공 목표치 (잠정)

| 지표 | 최소 목표 | 우수 목표 |
| --- | --- | --- |
| 작업당 공차 이동 거리 `주 가설` | **−8%** | −15% |
| 공차 이동 비율 | −5%p | −10%p |
| QC 크레인 대기 시간 | −5% | −10% |
| TT 사이클 타임 | −5% | −10% |

잠정값은 터미널 배차 최적화 선행 사례를 참고한 출발점이며, 최종 수치는 베이스라인 수집 후 Westports와 공동 확정합니다.

:::tip
**운영 전환 판정:** 작업당 공차 이동 거리가 최소 목표를 달성하고, 결과가 통계적으로 유의할 것.
:::
