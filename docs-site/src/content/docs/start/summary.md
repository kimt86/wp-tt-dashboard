## ① 문제 — QC를 굶기지 않는 배차

컨테이너 터미널의 생산성은 안벽 크레인(QC)이 결정합니다. 컨테이너를 내릴 때(양하·DS)도 실을 때(적하·LD)도 야드 트럭(TT)이 제때 크레인 아래에 있어야 하며, TT가 늦으면 QC는 그대로 멈춥니다 — 우리는 이것을 "QC가 굶는다"고 부릅니다. 배차가 어려운 이유는 구조적입니다. TT는 특정 QC 전속이 아니라 공유 풀에서 배차되고, 작업 수요는 본선 작업 계획에 따라 시시각각 출렁이며, 다음 작업 예정시각(ETW) 같은 핵심 신호조차 TOS DB 값은 수 시간의 오차를 보였습니다.

그래서 이 플랫폼은 "개선" 이전에 "측정"부터 시작합니다. 2026-06-08 고객사 워킹 세션에서 문제를 정의했고, 측정(KPI) → 실시간 가시화(배차 풀·라이브 맵) → AI 배차의 순서로 접근합니다.

## ② 아키텍처 — 운영 DB 무접촉 원칙

운영 Oracle TOS는 API가 절대 직접 질의하지 않습니다(API 크레이트에는 Oracle 클라이언트 자체가 없습니다). 추출기(extractor)가 타이머로 필요한 테이블만 PostgreSQL에 복제하고, Rust(axum) API는 Postgres만 바라보며, React/Vite 대시보드가 이를 소비합니다.

**배치 경로**

`Oracle TOS`(운영 DB · 읽기만) → *extractor · 타이머 복제* → `PostgreSQL`(tos-db-research) → *Rust (axum) API* → `React 대시보드`(Vite · KPI/TT/MAP…)

**실시간 경로**

`GPS · 크레인 PLC`(websocket · wpt_gps · ctab 존) → *SSH 터널 · 실시간* → `라이브 경로`(같은 API로 합류 → 맵·피드·사이클)

위가 배치 경로, 아래가 실시간 경로입니다. 모든 지표는 TOS가 "말하는 것"과 차량이 "실제로 움직인 것"을 교차 검증하며 만들어집니다.

## ③ 핵심 지표 — 무엇을 어떻게 재는가

대시보드 KPI 페이지는 일/주/월 이력과 함께 다음 지표를 제공합니다.

| 지표 | 한 줄 정의 | 소스 |
| --- | --- | --- |
| 사이클타임 | 한 작업 사이클의 소요시간 중위값 — 양하(DS)/적하(LD)는 특성이 달라 분리 집계합니다. | 융합 |
| 공차거리 · 공차비율 | 빈 차로 주행한 거리와, 전체 주행 대비 그 비율. 실측 GPS로 교차 검증합니다. | TOS |
| TT 가동률 | 작업 풀 배정 기반 — A(작업중)·A/B/Q(투입)를 60초 샘플(util_tt_sample)로 시간 평균. | TOS |
| QC 대기 | QC가 트럭을 기다린 시간 — 크레인 가동 구간 사이의 유휴 갭 평균. | TOS |
| 시간당 처리량 | QC가 한 시간에 처리한 move 수(move/hr) — 크레인 PLC 훅로드 신호로 교차 검증. | 융합 |

## ④ 라이브 기능 — 대시보드 여섯 페이지

대시보드는 KPI · TT · CYCLES · MAP · HEALTH · FEED 여섯 페이지로 구성됩니다. 핵심 네 가지:

- **TT · 배차 현황** — 작업 풀(수요)과 차량 풀(공급)을 한 화면에. extractor가 약 90초마다 TOS 작업 큐를 복제하고, 정확한 ETW는 Azure 게이트웨이(SSH 터널 · 30분 TTL)에서 받아 보강합니다.
- **MAP · 라이브 맵** — GPS websocket 기반 — MapLibre + ESRI 지도 위에 터미널 차량의 실시간 위치를 그립니다.
- **CYCLES · 사이클 이력** — 사이클마다 4단계(공차이동·받기·부하이동·주기)를 막대로 분해해 보여줍니다. 관측하지 못한 단계는 추정하지 않고 NULL로 둡니다.
- **FEED · WS 데이터 헬스** — GPS·PLC websocket 수신 상태를 감시합니다 — 라이브 지표의 신뢰 전제 조건입니다.

차량 풀 분류는 한 차례 교정을 거쳤습니다. "미배정 유휴(idle)"로 집계되던 102대 중 51대가 실은 배정 상태였고, 지금은 공차+정지+배정을 **staging(배차·대기)**으로 분리해 idle은 진짜 미배정(약 21대)만 셉니다.

## ⑤ 지금과 다음 — 측정에서 예측으로

지금은 학습 데이터를 쌓는 단계입니다. 모든 인도 사이클은 **tt_cycle_log**에 기록됩니다 — 트럭(ytno)·작업 메타(작업유형/모선/항차/컨테이너/QC/트윈탠덤)·단계별 타임스탬프·구간(leg) 거리/시간을 멱등하게 적재합니다. 인도 판정은 엄격합니다: GPS의 container1이 비-빈 값에서 바뀌는 엣지에 보유 ≥30초·운반 ≥150m 이동필터를 걸어, TOS 재배정 같은 artifact를 기각합니다.

이 과정에서 websocket의 container1이 물리 적재가 아니라 TOS의 **"배정" 필드**임을 실측으로 규명했고(LD 트럭 34대가 container1이 세팅된 채 픽업 블록에 도착), 한때 "미리 가서 대기"로 오독했던 패턴도 데이터로 정정했습니다. 관측하지 못한 값은 NULL로 남기는 것이 원칙입니다.

다음 단계는 **2단계 예측 모형**입니다 — ① 작업 소요시간 예측, ② 그 위의 배차 최적화. 지금 쌓이는 사이클 로그가 이 연구의 연료입니다.
