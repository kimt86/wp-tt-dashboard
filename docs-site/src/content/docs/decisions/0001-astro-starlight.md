**상태:** `채택됨` · **최종 검토:** 2026-06-12

## 맥락

초기 지식센터는 손으로 만든 정적 HTML 묶음(`kc/`)이었다 — 공통 셸(`kc-shell.js`)에 사이드바를 단일 소스로 두고, 다크/라이트 토글, 렌더 체크 스크립트까지 갖췄다. 그러나 문서가 16개를 넘기며 한계가 드러났다:

- **전문 검색이 없다** — "예전에 알아낸 것"을 다시 찾기 어렵다.
- 사이드바·prev/next를 **수작업**으로 유지해야 한다.
- 문서마다 head·스타일 boilerplate가 중복된다.
- 컨벤션(상태표시·프론트매터)이 강제되지 않는다.

지식센터의 목적은 **손실 없는 단일 출처**인데, 검색·구조·일관성이 약하면 그 목적이 흔들린다.

## 결정

지식센터를 **Astro Starlight**로 재구축한다. 마크다운/MDX로 작성하고, **Pagefind 전문 검색 내장**, 사이드바 자동 생성, 한국어 로케일을 쓴다. 정적 빌드 결과물을 기존 운영 API가 `/kc/`로 서빙한다(base `/kc`).

## 대안

- **손수 만든 HTML 유지** — 검색·자동화 부재가 핵심 문제라 기각.
- **Docusaurus / VitePress** — 기능은 충분하나 React/Vue 런타임·설정 부담. Starlight가 정적·경량·검색 내장으로 더 적합.
- **MkDocs (Material)** — 훌륭하나 Python 툴체인 추가. 프로젝트가 이미 Node 생태계(`web/`)라 Astro가 일관.

## 결과

- ✅ 전문 검색(Pagefind), 자동 사이드바, MDX 컴포넌트(asides/카드), 일관 컨벤션.
- ✅ 정적 빌드 → 기존 API가 그대로 `/kc/` 서빙(추가 프로세스 없음).
- ⚠️ 비용: 콘텐츠 변경 시 **빌드 단계** 필요(`npm run build`). 기존 16문서 **마이그레이션** 필요(별도 진행).
- ⚠️ base `/kc` 때문에 내부 링크는 `/kc/...` 절대경로를 써야 한다([글쓰기 규칙](/kc/start/contributing/)).
- 기존 `kc/`는 마이그레이션·검증 완료 후 제거한다.
