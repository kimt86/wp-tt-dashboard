websocket GPS 피드에서 모든 야드 트럭의 작업 사이클을 끊김 없이 감지하고, 검증된 사이클을 4단계(공차이동·받기·부하이동·주기)로 분해해 영구 적재하는 내부 메커니즘의 전체 설계. CYCLES 페이지와 향후 ML 학습 데이터의 기반이다.

## 1. 왜 내부적으로 사이클을 감지하는가

출발점은 운영 화면의 이상 징후였다 — "지금 유휴(idle)" 트럭이 비정상적으로 많았다.

WP-TT GPS websocket은 **이벤트 드리븐**이다. 메시지마다 차량 상태 전체가 오는 게 아니라 바뀐 필드 위주로 오고, 수신부는 메시지마다 차량 상태를 새로 만들기 때문에 빠진 작업 필드(`container1`, `jobtype`, `topos1` 등)는 비어 버린다. 순간 필드가 빈 트럭을 그대로 분류하면 **배정받아 일하던 트럭이 유휴로 둔갑**한다.

:::note[실측 (2026-06-10)]
dispatch=idle로 분류된 102대 중 **51대(50%)가 TOS 작업풀에 실제 배정(상태 A)**되어 있었고, 진짜 미배정은 약 21대뿐이었다. 분류기가 "공차 + 정지"만 보고 배정 여부를 무시한 것이 주원인.
:::

그래서 세 가지 목표를 세웠다:

1. **끊김 없는 추적** — 필드가 빠진 heartbeat에도 트럭의 작업 상태를 잃지 않는다 (latch + 작업풀 교차참조).
2. **사이클 축적** — 완료된 모든 작업 사이클을 PostgreSQL `tt_cycle_log`에 영구 적재한다 (ML 학습 데이터).
3. **라이브 교정** — 배정됐는데 대기 중인 트럭은 idle이 아니라 **staging**으로 분류한다.

## 2. 데이터의 진실 — container1은 '배정' 필드다

사이클 감지의 가장 중요한 발견. 이걸 모르면 단계 분해가 전부 어긋난다.

`container1`은 트럭이 **물리적으로 싣고 있는** 컨테이너가 아니라 **TOS가 배정한** 컨테이너다. TOS는 직전 인도(drop) 시점에 다음 작업을 미리 배정하므로(**c2c, 연속 사전배정**), 트럭이 빈 채로 다음 픽업지로 달리는 동안에도 container1은 차 있다.

:::tip[검증]
라이브 스냅샷에서 적하(LD) 작업 트럭 **34대가 container1이 세팅된 채 픽업 블록에 ARRIVED** 상태였다 — 아직 컨테이너를 받지도 않았는데. container1 변화 엣지만 보면 공차 구간은 구조적으로 보이지 않는다.
:::

따라서: `container1`이 세팅되는 순간 = **배정/사이클 시작**에 가깝고, 물리적 픽업·인도는 **ARRIVED 신호와 이동 거리**로 복원해야 한다. 3장·6장이 그 방법이다.

## 3. 인도(drop) 판정 — 이동필터

"사이클이 끝났다"의 단일 기준. TOS의 필드 재기록과 진짜 인도를 가르는 물리 증거는 **이동**이다.

인도 후보 엣지는 `container1`이 **비어있지 않은 값에서 다른 값으로** 바뀌는 순간이다(→빈값 또는 →다른 컨테이너. c2c가 많아 "비워짐"만 보면 인도의 약 3/4을 놓친다). 후보가 진짜 인도로 인정되려면 두 조건을 모두 통과해야 한다:

| 조건 | 상수 | 의미 |
| --- | --- | --- |
| 보유 시간 | `MIN_LOADED_MS = 30초` | 컨테이너를 30초 이상 갖고 있었다 (필드 깜빡임 제거) |
| 운반 거리 | `MIN_CARRY_TRIP_M = 150 m` | 갖고 있는 동안 실제로 150 m 이상 주행했다 (정지 상태의 TOS 재배정 기각) |

거리 누적에는 GPS 점프 가드가 있다 — 연속 fix 간 이동이 `MAX_FIX_STEP_M = 600 m`를 넘으면 그 스텝은 버린다 (텔레포트/정확도 스파이크가 "이동"을 위조하지 못하게).

- 두 조건을 통과 → **인도 확정**: 플릿 인도 카운트(λ), 사이클 간격 샘플(중위값, `[120, 1200]초` 밴드), 그리고 5장의 사이클 완료 트리거.
- 이동 미달로 기각 → **artifact**(TOS 재배정 의심)로 별도 집계. 그중 `[100, 150)m` 구간은 near-miss로 따로 추적해 필터가 진짜 초단거리 운송을 얼마나 버리는지 감사 가능하게 했다.

:::note[설계 원칙]
사이클 감지기는 이 검증 엣지를 **그대로 재사용**한다(감싸기). 별도의 두 번째 감지기를 만들지 않으므로 라이브 KPI(사이클 중위값·Little's law)와 적재되는 사이클 로그가 항상 같은 사건을 본다.
:::

## 4. 끊기지 않는 상태 — latch & 작업풀 교차참조

### 필드 latch

heartbeat가 작업 필드를 생략해도 직전 값을 유지한다. `latched_container`, `latched_jobtype`, `latched_vessel`, `latched_topos`는 **새 값이 실제로 도착했을 때만 갱신**되고, 빠지면 이전 값을 그대로 둔다. 검증된 인도가 완료되면 latched_container는 비운다.

### 작업풀 교차참조 (권위 소스)

GPS만으로는 "공차·정지인데 배정됨"과 "진짜 미배정"을 구분할 수 없다. 그래서 백그라운드 태스크가 **약 30초마다** `live_assigned_tt ⨝ live_workpool`(추출기가 ~90초마다 TOS에서 갱신)을 읽어 트럭별 배정 캐시(`assigned_pool`)를 유지한다 — 배정 여부 + 작업 메타(jobtype/vessel/voyage/contno/qc/twintandem).

- 이 캐시는 8장의 staging 분류와, 사이클 오픈 시 작업 메타 스냅샷에 쓰인다.
- 캐시는 30초마다 통째로 교체되며, 스테일 기준(5분)을 넘긴 배정은 빠진다. 별도의 해제 게이트는 없지만 효과는 보수적이다 — 배정 캐시를 참조하는 분기가 "공차·정지" 분기뿐이라, 주행·적재 중인 트럭은 풀 이탈/틱 지연(90초)의 영향을 받지 않는다.

## 5. 사이클의 생애 — 상태머신

메시지 하나가 들어올 때마다 트럭별 상태가 이 흐름을 따라 전이한다.

미배정 Unassigned → **container1 세팅 = 사이클 오픈** → 공차이동 (픽업지로) → 받기 (도착·수령) → 부하이동 (적재 주행) → 주기 (핸드오버) → **인도 = 사이클 완료** → 다음 배정

**핵심:** 사이클은 **물리적 픽업이 아니라 container1이 세팅되는 순간(= 배정 관측)에 열린다** — 2장에서 보았듯 container1은 배정 필드라, 오픈 시점에 트럭은 아직 빈 차로 픽업지를 향한다. 그 뒤 4단계(공차이동·받기·부하이동·주기)는 6장의 ARRIVED 신호로 복원된다. 연속적재(c2c)면 직전 인도와 **동시에** 다음 사이클이 열리고, 공차 구간이 있는 경우엔 빈 차로 달리는 도중 배정이 관측되어 열린다 — 어느 쪽이든 사이클의 논리적 시작(`assigned_at`)은 공차 leg 시작(직전 인도)으로 소급 기록된다.

| 이벤트 | 트리거 | 기록되는 것 |
| --- | --- | --- |
| 사이클 오픈 | `container1` 빈값→세팅 (= 배정 관측) | `OpenCycle` 생성: 작업 메타 스냅샷(latch + 작업풀), 공차 leg 시간·거리, `assigned_at`(공차 시작 = 직전 drop의 프록시; 공차 시작을 못 봤으면 현재 시각) |
| 픽업측 도착 | 적재 측(6장) ARRIVED 상승엣지 | `pickup_arrived_at` |
| 픽업 출발 | 도착 후 속도 ≥ 3 km/h 회복 | `pickup_left_at` (부하이동 시작) |
| 드롭측 도착 | 하역 측(6장) ARRIVED 상승엣지 | `arrived_at` |
| 사이클 완료 | 3장의 이동필터를 통과한 container1 변화 | `CompletedCycle` 확정 → 적재 버퍼로. laden leg 거리·시간, c2c 플래그 |

공차 leg도 대칭으로 측정한다: 트럭이 비는 순간부터 `empty_since`·`empty_trip_m`(공차 주행 경로 길이)을 누적해 사이클 오픈 시 함께 스냅샷한다.

## 6. 4단계 분해 — ARRIVED 측면 분류

container1이 배정 필드인 이상(2장), 물리적 픽업/인도는 ARRIVED 신호를 "어느 쪽에 도착했는가"로 갈라 복원한다.

4단계 비율(예시): 공차이동(가장 김) · 받기(짧음) · 부하이동(가장 김) · 주기(짧음).

| 단계 | 구간 | 의미 |
| --- | --- | --- |
| 공차이동 | `assigned_at → pickup_arrived_at` | 배정 후 빈 차로 픽업 지점까지 주행 |
| 받기 | `pickup_arrived_at → pickup_left_at` | 픽업 지점 대기 + 컨테이너 수령 |
| 부하이동 | `pickup_left_at → arrived_at` | 적재 상태로 하역 지점까지 주행 |
| 주기 | `arrived_at → dropped_at` | 하역 지점 핸드오버 (크레인/RTG 인계) |

### 측면 분류 규칙

작업유형별로 어느 쪽이 적재(픽업) 측이고 어느 쪽이 하역(드롭) 측인지가 정해져 있다. ARRIVED가 뜬 목적지(`topos1`)가 크레인 코드(C/M/Z+숫자)인지 블록 코드인지로 판정한다:

| 작업유형 | 픽업 측 | 드롭 측 |
| --- | --- | --- |
| LD (적하) | 야드 블록 (RTG) | 안벽 크레인 (QC) |
| DS (양하) | 안벽 크레인 (QC) | 야드 블록 (RTG) |
| MI / MO / LC (야드 이동) | 둘 다 블록 → 순서로 판정: 첫 도착 = 픽업, 픽업 출발 후 도착 = 드롭 | (좌동) |

- **상승엣지만** 인정한다 (직전 메시지가 ARRIVED가 아니었을 때 → ARRIVED). 직전 작업에서 남은 스테일 ARRIVED가 새 사이클의 픽업 도착으로 오인되는 것을 막는다.
- **출발 판정**: 픽업 도착 후 ARRIVED가 풀리고 속도가 `IDLE_SPEED_KMH = 3 km/h` 이상으로 회복되는 첫 순간 = `pickup_left_at`.
- **표시 클램프**: 화면(CYCLES 페이지)에서는 `배정 ≤ 픽업도착 ≤ 출발 ≤ 도착 ≤ drop`의 단조 순서로 클램프해 단계 합이 항상 사이클 시간과 일치하게 그린다. DB에는 관측값 그대로 저장.

## 7. 영속화 — tt_cycle_log

완료된 사이클은 메모리 버퍼에 쌓였다가 30초마다 PostgreSQL로 흘러간다. 재시작에도 안전하다.

### 스키마 (마이그레이션 0020 · 0021 · 0022)

| 컬럼군 | 컬럼 | 내용 |
| --- | --- | --- |
| 식별 | `ytno, business_date, shift` | 트럭 번호, 터미널 운영일·교대 |
| 작업 메타 | `jobtype, vessel, voyage, container, qc, twintandem` | 픽업 시점의 latch + 작업풀 스냅샷 (트윈 리프트는 ML에서 가중치 보정용으로 보존) |
| 단계 시각 | `assigned_at, pickup_arrived_at, pickup_left_at, pickup_at(=배정 관측), arrived_at, dropped_at` | 4단계 분해의 원천. **관측 못 한 단계는 NULL** (추정 금지) |
| leg 측정 | `idle_before_s, empty_leg_s/m, laden_leg_s/m, cycle_s` | 공차/적재 leg의 시간·경로거리. laden_leg_m ≥ 150 보장(이동필터) |
| 품질 플래그 | `movement_ok, incomplete, container_to_container` | 이동필터 통과 여부, (예약) 중도소멸, c2c 연속적재 |

### 적재 경로

- 완료 사이클 → 메모리 버퍼 (상한 5,000건, 초과 시 oldest drop + 경고 로그).
- flusher 태스크가 **30초마다** 배치 INSERT. `UNIQUE(ytno, dropped_at)` + `ON CONFLICT DO NOTHING`으로 **멱등** — API가 재시작해도 중복 행이 생기지 않는다.
- 재시작 시 손실 범위 = 트럭별 진행 중 사이클 1건 + 아직 flush되지 않은 버퍼(최대 ~30초분). 이미 flush된 완료분은 영속.
- 적재량 약 14,000건/일 규모. append-only, ML용으로 전량 보존.

## 8. 라이브 분류 교정 — idle vs staging

라이브 맵/배차 풀의 분류기는 이제 작업풀 배정 캐시(4장)를 함께 본다:

| 상태 | 조건 | 의미 |
| --- | --- | --- |
| **staging** (배차·대기) | 공차 + 정지(<3 km/h) + **작업풀에 배정됨** | 픽업 순서를 기다리는 중 — 가용 자원이 아니다 |
| **idle** (유휴) | 공차 + 정지 + **진짜 미배정** | 즉시 배차 가능 |

적용 직후 idle 수가 ~102 → ~21로 정상화됐고, 차이(~51)는 staging으로 잡힌다. staging은 가동 트럭(active)에는 포함되지만, Little's law(W = L/λ)의 순환 트럭 수 L에서는 제외한다 — 아직 laden 왕복 아크에 올라가지 않았기 때문이다. TOS 기반 가동률(K_UTIL)은 원래 배정 기반이라 영향이 없다 (두 정의가 일관됨).

## 9. 정직한 한계와 데이터 품질

:::caution[부분 관측]
단계 시각은 ARRIVED 신호 의존이라 모든 사이클이 4단계를 다 갖지는 않는다. 가동 15분 실측: 신규 219 사이클 중 픽업도착 44% · 픽업출발 29% · 드롭도착 59% · 4단계 완전체 20%. 관측 못 한 단계는 NULL로 남고 화면에서는 해당 구간이 접힌다. 완전체 비율은 가동이 길어질수록 올라간다.
:::

:::caution[assigned_at은 프록시다]
GPS 피드에 TOS 배차 시각이 직접 없으므로 assigned_at = 공차 leg 시작(≈ 직전 drop) 시각이다. c2c 사전배정 관행 덕에 실제 배차와 거의 일치하지만, "TOS 배차 기록"이 아님을 분석 시 기억할 것.
:::

:::tip[관측경계 아티팩트 — 발견과 정정의 기록]
초기 데이터에서 "픽업도착 < 배정" 역전 행이 나왔다. 처음엔 '미리 가서 대기'로 해석했으나, 전수 조사 결과 **역전 전부가 assigned==pickup(공차 시작 미관측 fallback)**이고 83%가 재시작 후 첫 사이클 — 운영 현상이 아니라 관측 경계 아티팩트였다. 조치: 공차 leg 시작을 실제로 관측한 경우에만 픽업도착을 시딩하도록 가드 추가 + 오염 행 NULL 정리(진단 시점 55행, 정리 시점 56행). 현재 역전 0건. "트럭은 배차 없이 작업지점을 알 수 없다"는 운영 상식이 데이터 검증의 잣대가 된 사례.
:::

- **idle_before**(배정 전 유휴 시간)는 적재 시점이 아니라 오프라인에서 도출한다 — 같은 트럭의 연속 행에서 `이번 pickup_at − 직전 dropped_at`.
- **twin/tandem (컨테이너 2개)**: 트윈 리프트는 `container1`·`container2`가 모두 차고, 적재 판정·픽업·인도는 `container1` 기준이라 사이클은 **물리 1트립 = 1건**으로 기록된다(container 컬럼엔 1번 박스만; `twintandem='W'`로 표시되어 무브 수 분석에서 ×2 보정 가능). 실측: 트윈 운반 트럭 전수에서 "container2만 차고 container1은 빈" 위험 케이스는 0건이라 적재를 놓치지 않는다. 단, 피드가 두 박스 번호를 `container1`↔`container2` **슬롯만 맞바꾸는** 현상이 잦아(실측 20분간 50회, 다수가 주행 중) 맨 container1 엣지만 보면 가짜 인도로 오인되거나 carry 측정이 리셋된다 — **비-빈 {c1,c2} 집합이 그대로면 슬롯 스왑**으로 판정해 인도·carry 리셋 모두 건너뛴다(집합이 줄면 진짜 부분 인도로 정상 처리). 적용 후 트윈 가짜 사이클(back-to-back <90초)은 1.35/시 → 0이 됐다.
- 150 m 미만 초단거리 야드 이동은 이동필터가 의도적으로 거른다 (near-miss 카운트로 규모 감사 가능).

## 10. 어디서 쓰이나

- **대시보드 CYCLES 페이지** — 플릿 요약(총 사이클·시간당·중위·적재거리), 트럭 리더보드, 트럭별 4단계 세그먼트 막대 타임라인. 기간 1h/4h/12h/24h/3d.
- **API** — `GET /api/tt-cycles/summary?hours=` (플릿 집계 + 처리량 버킷 + 트럭별 집계), `GET /api/tt-cycles/detail?ytno=&hours=&limit=` (사이클 행 + 단계 시각 전체).
- **라이브 KPI** — 같은 인도 엣지가 사이클 중위값·Little's law·인도율(λ)을 공급 (감싸기 설계라 항상 일치).
- **ML 학습 데이터** — [05 예측 모형 연구](/kc/research/tt-prediction/)의 ①이동시간 ②크레인 준비 ③핸드오버 예측에 필요한 per-job 라벨이 이 테이블에서 나온다.

용어가 낯설다면 [용어집 & FAQ](/kc/knowledge/glossary/)를, 이 로직이 나온 과정은 [연구 일지](/kc/journal/research-log/)를 참고. **차세대 설계**(topos1 레그 기반, 그림자 가동 중)는 [사이클 v2 설계](/kc/experiments/cycle-v2-shadow/)에 있다.
