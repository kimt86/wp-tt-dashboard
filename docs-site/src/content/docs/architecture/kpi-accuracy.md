KPI의 권위는 **TOS**지만, TOS는 몇 분마다 당겨오고 시간 단위로 버킷팅되며 — 때로는 **아예 다른 양을 잽니다**. 초당 GPS/PLC를 활용하면 실시간 구간의 정확도를 끌어올리고, 잘못 측정되던 지표를 바로잡을 수 있습니다. 이 문서는 "**실제로 어떻게 정확해지는가**"에 초점을 둡니다 — 특히 **TT 사이클 타임** 재검토를 중심으로.

기준 **2026-06-09** · 원칙: websocket은 실시간 보정·교차검증, 야간 TOS 확정값이 최종 권위 · **외부 평가 4회 반영**

## 30초 요약 — 무엇이 어떻게 정확해지나

| KPI | TOS의 문제 (실시간/측정) | websocket 보정 → 어떻게 정확해지나 | 상태 |
| --- | --- | --- | --- |
| **TT 사이클** | **아예 다른 양**을 잼 — 컨테이너 작업 생애 span(≈40분), 트럭 운반 사이클 아님 | GPS로 **실제 트럭 사이클**(배달→배달 간격) 측정 ≈ 10–15분(실측 중앙값 ~13분, n=264, 전체 1h 윈도우). "올바른 것"을 잼 | 완료 |
| **K_MPH** | 3분마다 당김 + 가동시간 정시버킷(거침) | 크레인 PLC 사이클을 초당 카운트 → 실시간 QC 처리량 | 완료 |
| **K_QC_Q** | 유휴갭 `HAVING≥10`로 교대 중 과소 | 크레인 유휴 + 무트럭을 직접 감지 → 기아 실시간 | 완료 |
| **K_UTIL** | 로그인 세션 기준(논다고 빠지지 않음) | 현재 비유휴(작업중) 트럭 비율 — 즉시 교차검증 | 완료 |

:::note[핵심 메시지]
정확도 향상은 두 방식입니다 — ① **더 신선하게**(K_MPH·K_QC_Q·K_UTIL: 초당 신호로 실시간), ② **올바른 양을 측정**(TT 사이클: TOS가 잘못된 것을 재고 있었음). 특히 ②가 본질적 향상입니다.
:::

## 정확도는 "어떻게" 올라가나 — 두 메커니즘

**① 신선도 (freshness)** — TOS는 교대 틱이 **3–15분**마다 당기고, 가동시간을 **정시 단위**로 셉니다. websocket은 **GPS ~3초·PLC ~1초**라, "지금 이 순간"의 처리량·대기·가동을 실시간으로 보여줍니다. → K_MPH·K_QC_Q·K_UTIL.

**② 올바른 양 (correctness)** — 더 본질적 — TOS의 "TT 사이클"이 사실 **컨테이너 처리 생애 span**을 재고 있었습니다(≈40분). 실제 트럭 운반 루프는 **GPS ≈13분**, 그리고 **TOS move이력(MCH_OPERATION)로도 ≈14분** — 독립된 두 방법이 일치. 신선함을 넘어 **맞는 것을 잼**.

:::note[원칙 — 대체가 아니라 보정]
websocket 값은 **실시간 추정·교차검증**이고, 과거/권위 보고는 야간 TOS 확정값을 씁니다. 화면에서 **⚡(실시간)**으로 명확히 구분 표시합니다. GPS는 정밀 타임스탬프·권위가 없는 대신, "지금 무엇이 일어나는가"를 직접 봅니다.
:::

## TT 사이클 재검토 — 잘못 측정되던 지표

대시보드의 "TT 사이클 타임"이 **≈40분**으로 비정상적으로 길었습니다. 조사 결과, TOS 측정값이 **트럭 사이클이 아니라 다른 것**을 재고 있었습니다.

**TOS가 재던 것 (잘못) — ≈ 40 분**: **컨테이너 작업의 전체 이벤트 span** — (CONTNO,POINT,SEQNO)별 첫~마지막 이벤트(작업당 ~9–12개). 야드 대기·다단계 처리를 모두 포함한 **생애주기**. 한 트럭의 운반 루프가 아님.

**실제 TT 사이클 (GPS) — ≈ 10–15 분**: **트럭이 배달→배달까지 한 바퀴** 도는 시간. GPS로 직접 관측. 픽업→운반→핸드오버→공차복귀→다음픽업.

### 조치 ① — TOS 지표 개명 → 내부 보관(미표출)

TOS 값 자체는 유효한 "**작업 처리 시간(Container Handling Span)**"이라 삭제하지 않고 개명했습니다(`crates/core/src/kpi.rs`). 다만 운영팀 요청으로 **지금은 화면에 표출하지 않고 내부에만 보관**합니다. 사이클 카드는 "**TT 사이클 타임**"으로 되돌리되 값은 실제 트럭 사이클(아래)로 채우고, **데이터가 없으면 "—"**로 둡니다.

### 조치 ② — GPS로 실제 사이클 측정

핵심은 **배달(delivery) 이벤트**를 옳게 잡는 것입니다. 처음엔 "container1이 비워질 때(loaded→empty)"로 셌는데 — **이게 실측 결과 틀렸습니다**:

```
// 2분 실측: 배달 이벤트 카운트 방식 비교
container1 비움(loaded→EMPTY)만 셈        →   ~90/시간   (❌ 너무 적음)
container1 변경(nonempty→다른값)을 셈     →  ~574/시간   (✓ 현실적)
                                                          └ 배달의 75%가 A→B 직접전환(빈 값 안 거침)
```

:::note[왜 정확해지나]
트럭은 컨테이너 A를 배달하고 **곧바로 B를 받습니다** — container1이 A→B로 바뀌며 빈 값을 거의 안 거칩니다(3초 GPS 샘플엔 더더욱). "비움"만 보면 배달의 3/4을 놓쳐 처리량이 폭망(90/h)하고 사이클이 폭주(수백 분)했습니다. **"비어있지 않은 값에서 다른 값으로의 변경"**을 배달로 세니 574/h로 현실화 — 이게 정확도의 분기점입니다.
:::

#### 측정 — 트럭별 배달 간격 중앙값

헤드라인은 **트럭별 연속 배달 간격의 중앙값**입니다 — 캡 `[2분, 20분]`(느슨한 물리 sanity), ≥5표본, **p25–p75 spread도 함께 노출**. 우편향 꼬리에 강건합니다.

:::caution[막다른 길 — 시간 하한으로 "예쁜 값" 만들기 (외부 평가 #3이 기각)]
처음엔 median이 **3.0분**으로 짧게 나오자 "사이클은 더 길어야 한다"는 **선입견**으로 캡 하한을 2→3분으로 올렸습니다. 외부 평가가 이를 **순환논리**로 지적: 「짧으니 잡음이다」→잡음을 빼니→「길어졌다, 거봐 잡음이었다」. 하한을 올리면 median은 **기계적으로** 올라가므로, 이는 측정이 아니라 **분석자의 기대를 숫자에 새겨넣는 것**. 독립 증거 없이 표본 하위꼬리만 손으로 잘라낸 것이라 기각했습니다.
:::

:::note[정답 — GPS 이동으로 배달을 직접 검증]
진짜 배달이면 트럭이 컨테이너를 **물리적으로 운반**해 움직였어야 합니다. 그래서 시간이 아니라 **이동거리**로 거릅니다: 적재 중(container1 보유) 누적 주행거리를 추적해, container1이 바뀔 때 트럭이 **≥150m** 움직였으면 진짜 배달, **안 움직였으면 TOS 재배정 아티팩트**로 기각(GPS 떨림은 600m 점프 가드로 제외). "사이클은 길다"는 가정을 전혀 안 쓰고, **관측 가능한 행동(움직였나?)**으로 판정 — median을 인위적으로 올리지 않습니다.
:::

:::caution[투명성 — 아티팩트 카운트 노출]
기각된 정지-상태 container1 변경 수를 **최근 1h 아티팩트 건수**로 함께 노출합니다(화면 tooltip). 필터가 무엇을 얼마나 빼는지 숨기지 않고 감사 가능하게 — 진짜 배달 대 아티팩트 비율을 운영팀이 직접 봅니다. **실측(약 1h 윈도우): 진짜 배달 236 vs 아티팩트 66 ≈ 78% 진짜** — 5건 중 1건이 정지-상태 재배정. 떨림으로 보기엔 너무 높고 누수로 보기엔 너무 낮은, 과잉공급·TOS 잦은 재배정 피드에 합당한 대역.
:::

:::caution[정직한 한계 — 150m 컷의 상방 편향 + 표시 가드 (외부 평가 #4 반영)]
150m는 떨림 vs 운반의 합리적 경계지만, **한 방향으로 편향**됩니다: 인접 스택↔QC 같은 진짜 초단거리 하울이 잘려 median이 **약간 위로**(이 지표가 "상한·방향성"인 이유). 그래서 **100–150m 근접 기각 건수**를 따로 노출해 얼마나 잘리는지 보이게 했습니다. 또한 **(1) n≥20부터만 median 표시**(이하 "수집중 n/20"), **(2) 숫자는 항상 n·IQR과 함께**(맨 숫자 단독 금지), **(3) 1h 윈도우가 안 차면 "윈도우 n/60분" 표기** — 부분 윈도우의 잠정값을 확정값처럼 인용하지 않도록.
:::

:::caution[정직한 한계 — 유휴 포함]
배달→배달 간격은 그 사이 **잠깐의 유휴**도 포함합니다(과잉공급으로 트럭이 노는 시간). 따라서 이 값은 **활성 사이클의 상한**입니다(완전한 순수 운반시간은 유휴 차감이 필요 — 차후). 그래도 TOS의 40분 생애 span보다 훨씬 트럭에 가깝습니다. 화면 라벨도 "사이클"이 아닌 **"배달간격"**으로 정직하게 표기.
:::

:::caution[Little's law는 여기선 부정확 (외부 평가 후 표시 제외)]
W=L/λ도 계산하지만, L(사이클 중 트럭)을 dispatch 상태로 셀 때 **empty_travel**에 회송·유휴성 트럭이 섞여 L이 과대 → W가 폭주(관측 ~53분). 그래서 **화면엔 표시하지 않고** 중앙값을 씁니다. (정밀한 L 정의는 차후 과제.)
:::

### 조치 ③ — TOS로도 트럭 사이클 근사 (GPS와 교차검증)

GPS(websocket)는 **실시간 전용**이라 피드가 끊기거나 이력이 필요하면 빈칸이 됩니다. 그래서 **TOS DB만으로도** 트럭 사이클을 근사하는 방법을 찾았습니다. 핵심은 **`TOSADM.MCH_OPERATION`**: 크레인 move마다 그 move를 처리한 **트럭 ID(`TRK_ID`)**와 완료시각이 남습니다(보존 **≥35일**). 핸들링 span을 재던 `JOB_ORDER_HISTORY`는 **컨테이너 단위**라 트럭 컬럼이 없어 사이클을 못 냈는데, MCH_OPERATION에는 있습니다.

**방법:** QC move(`MACHNO ^C##`, LD/DS)만 골라 **TRK_ID별로 완료시각을 정렬**해 연속 move 간격을 냅니다. 트럭은 한 사이클에 QC를 한 번 방문하므로, **같은 트럭의 연속 QC move 간격 = 한 사이클**(픽업→야드 드롭→복귀→다음 픽업). 캡 `[2분,20분]` 후 중앙값.

:::note[교차검증 — 독립된 두 방법이 일치]
실측(2026-06-09, 10–11시 창, n=160): **TOS MCH_OPERATION 중앙값 ≈ 14.2분**. 같은 시기 **GPS 중앙값 ≈ 13분**. **완전히 다른 출처**(TOS 작업이력 vs 실시간 GPS)가 ~13–14분으로 **수렴** — 둘 다 옳은 트럭 사이클을 재고 있다는 강한 증거이자, TOS의 **40분 핸들링 span과 명백히 다른 값**임을 재확인. (캡을 [2,30]분으로 넓히면 긴 꼬리가 들어와 ~19분.)
:::

:::note[상태 — 파이프라인 구현 완료(2026-06-10)]
추출기 파이프라인을 구축했습니다: `c10_k_tt_cycle.sql`(MCH_OPERATION) → `raw_k_tt_cycle` → 집계 → **K_CYCLE 카드**. 이제 사이클 카드는 **TOS 근사=메인(이력·항시), GPS=⚡실시간 보조**로 채워지고 둘 다 없을 때만 "—". 최근 10일 백필 결과 일별 **12–14분**으로 안정(40분 span과 명확히 구분). 부하: 야간/틱에 가벼운 MCH_OPERATION 쿼리 1개(K_MPH가 이미 스캔하는 같은 날짜 슬라이스). 핸들링 span은 `raw_k_cycle`에 그대로 보관(내부).
:::

## K_MPH 실시간 — PLC 사이클

크레인 PLC 하중의 **빈→적재 상승 에지(픽업) = move 1회**. 40초 최소간격으로 떨림을 흡수하고, 롤링 1시간 카운트 → 실시간 QC별 move/hr. TOS의 **정시버킷** 약점(한 시간에 10분 일해도 1시간)을 메웁니다. QC별·함대 평균을 TOS와 교차검증.

:::caution[정직한 한계]
하중 신호가 지저분합니다(같은 move 중 임계를 넘나듦, 일부 크레인은 낮게 읽힘) → 카운트가 시간대별로 흔들립니다. 그래서 PLC-MPH는 **방향성 교차검증**으로 쓰고, 더 깨끗한 신호는 §2의 **트럭 container1 변경**(함대 배달율)입니다. 둘은 서로 다른 처리량 프록시(크레인 move vs 트럭 배달)로, 정확히 일치하지 않습니다.
:::

## K_QC_Q 실시간 — QC 기아 직접 감지

가동 중인 안벽크레인이 마지막 move 후 **정상 간격(≈90–120초)을 넘겨 유휴**인데, 그 크레인에 **도착한 TT가 없으면**(GPS topos1==크레인 & ARRIVED 부재) ≈ "트럭 대기". TOS의 `HAVING≥10` 교대중 과소를 회피하고 직접 측정합니다.

:::caution[정직한 한계 (외부 평가 반영)]
크레인 유휴는 트럭만이 아니라 베이 이동·트윈 셋업·래싱·선박/해치 교체·휴식으로도 생깁니다. 그래서 임계를 **120초로 상향**(정상 사이클 갭 초과)했고, 이는 "확정"이 아니라 **후보 신호**입니다(안벽 QC 한정, RTG는 PLC 없음). 인과를 단정하지 않습니다.
:::

## K_UTIL 실시간 — 가동/유휴는 "작업 할당" 기준

정확한 정의: **가동 = 작업 할당~완료(크레인 전달) 사이**, **유휴 = 작업 할당을 못 받고 대기하는 시간**. 할당받은 트럭은 크레인 앞에 **멈춰 큐잉 중이어도 가동**입니다. TOS K_UTIL은 `로그인−정지`라 **할당 없이 노는 시간도 가동**으로 세어 ~95%로 과대평가 → 미표시.

:::caution[두 번의 오류 — 멈춤=유휴, 그리고 GPS 필드 신뢰]
① 처음엔 "이동·적재 중"만 가동으로 봐서 **할당받고 정차한 트럭을 유휴로 오인**. ② 그래서 GPS 작업필드(topos1/jobtype)로 할당을 판정했더니 **13%**로 폭락 — 알고보니 **이 필드는 할당 이벤트 때만 실리고 3초 하트비트엔 비어** 최신 fix 대부분이 빈 값(같은 트럭이 한때 topos1=C56이었다가 곧 None). 스냅샷으로 세면 대량 누락(작업풀엔 할당 160대인데 GPS 스냅샷은 18대).
:::

:::note[교정 — TOS 작업풀이 권위 출처]
할당의 권위 출처는 **TOS 작업풀**입니다(JOB_ORDER_LIST의 배정 YT, JOBSTATUS=A, ~90초 갱신). 진짜 가동률 = **할당 보유 트럭 / 가동 트럭**(가동 = 할당 ∪ 맨드[GPS 엔진ON]). 이 중 실제 이동·적재 중은 일부뿐 — 나머지는 **할당받고 큐잉/대기 = 여전히 가동**.
:::

:::caution[세 번째 오류 — 야드작업(MI/MO) 누락]
처음엔 작업풀을 **선박작업(DS/LD)으로만** 봐서 가동률이 낮게(≈57%) 나왔습니다. 실측: 활성 할당이 DS·LD 외에 **MO·MI·LC**도 있어 **야드 이동 중인 ~43대를 유휴로 오인**. 할당 집합을 **모든 활성 작업유형**으로 넓혔습니다(`live_assigned_tt`).
:::

:::note[분모도 100% TOS로 (GPS 제거)]
GPS 맨드(엔진ON)가 불안정해 분모도 TOS로 전환했습니다: **가동률 = 활성(A=디스패치 중) / 배치(A·블록 B·큐 Q, 작업 보유 트럭)**. 실측 **≈63%**(A 250 / 배치 ~397). 검증: TOS 분모(391~407)가 **GPS 맨드(~402)와 거의 일치** → 63%는 GPS 오류가 아니라 견고한 값. 의미: 작업이 걸린 ~400대 중 크레인이 동시에 굴리는 활성은 250대뿐, 나머지 ~147대는 **큐 대기 = 실제 과잉공급**. ("할당=ytno 배정"을 가동으로 더 넓게 보면 ~100%가 되어 무의미하므로, 활성 디스패치(A)를 가동으로 봄.)
:::

:::note[시간기반 + 이력 — 작업풀 표본 누적]
순간값을 넘어 **시간기반 가동률**로: API가 **60초마다** (할당, on-duty)를 `util_tt_sample`에 적재하고, 교대 평균이 곧 시간기반 가동률입니다(할당~완료=가동, 큐잉 포함, 유휴=미할당 시간). 카드 헤드라인=**교대 평균**, 보조=현재값. 트럭별 과거 "할당시각"이 TOS 이력에 없어(JOB_ORDER_HISTORY엔 YT id 없음) 소급은 불가하지만, 이 표본은 **지금부터 이력이 쌓입니다**. (~35일 소급 근사가 필요하면 MCH_OPERATION 간격법 별도 추가 가능.)
:::

## 한계 · 정직한 노트 (외부 평가 4회 반영)

| 항목 | 내용 |
| --- | --- |
| TOS가 최종 권위 | websocket은 실시간 추정·교차검증. 야간 TOS 확정값이 최종 KPI. 화면 ⚡로 구분. |
| 트윈/탠덤 과소 (사이클) | 한 번에 2개를 드는 트윈은 container1 변경이 1회라 배달을 1로 셈 → 처리량 λ가 트윈만큼(~20%) 과소 → Little's 사이클이 약간 길게 나옴. (container2까지 보면 개선 가능, 차후) |
| 배달 검증 | 배달은 **GPS 이동 ≥150m**로 검증(시간 아님). 정지 상태 container1 변경=재배정 아티팩트로 기각, 건수 노출. 캡 [2,20]분은 느슨한 sanity일 뿐. |
| K_QC_Q 귀인 | 유휴 크레인 = 트럭대기로 단정 못함(§4). 후보 신호로만. |
| K_UTIL 할당 기준 | 가동=할당~완료, 유휴=할당 없는 시간(§5). 할당받고 큐잉 중이면 멈춰도 가동. 할당 출처=TOS 작업풀(live_workpool.ytno) — GPS 작업필드는 간헐적이라 신뢰 불가. 카드는 할당/가동 ≈63%만(실시간 전용, 이력 없음). TOS 세션값은 미표시. |
| TOS 사이클 근사 | MCH_OPERATION.TRK_ID 연속 QC-move 간격 ≈12–14분(GPS 13분과 일치, §2 조치③). QC 방문 1회/사이클 가정. **파이프라인 구현 완료** — K_CYCLE 카드 메인값. |
| 핸들링 span | ≈40분(컨테이너 생애)은 내부 보관·미표출. 사이클 카드는 TT 사이클 타임, 데이터 없으면 "—". |
| PLC-MPH 노이즈 | 하중 신호 불안정 → 방향성 교차검증용(§3). |
| 콜드스타트 | API 재시작 후 윈도우가 찰 때까지(~10여 분) 표본 부족·외삽. 이후 안정. |

구현: `crates/api/src/livemap.rs`(container1 배달+GPS이동 검증·아티팩트 카운트·median/spread·기아·가동/작업률) · `crates/core/src/kpi.rs`(개명) · `web/src/App.tsx`(카드별 출처 배지 TOS/⚡WS·듀얼값). TOS 사이클 근사(MCH_OPERATION)는 검증 완료·파이프라인 차후. 외부 평가 4회(진단·코드리뷰·사이클 필터·최종 go/no-go)의 필수 수정 반영.
