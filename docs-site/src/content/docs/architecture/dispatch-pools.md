TT 배차의 두 축 — **차량 풀**(어떤 트럭이 비었는가 · 공급)과 **작업 풀**(어떤 크레인 작업이 트럭을 기다리는가 · 수요). 둘은 **갱신 방식이 완전히 다릅니다**: 차량 풀은 websocket으로 초단위 실시간, 작업 풀은 TOS에서 90초 스냅샷. 이 문서는 각 풀이 어떻게 갱신되는지, 그리고 둘이 화면에서 어떻게 합쳐지는지를 설명합니다.

기준 **2026-06-09** · 차량 풀 = **websocket GPS/PLC**(메모리·실시간) · 작업 풀 = **TOS JOB_ORDER_LIST + JOB_QUEUE_SCHEDULE**(90초) · 화면 = TT 페이지

## 두 풀 — 갱신 방식이 다르다

**차량 풀 = 공급 · websocket · 초단위 실시간**

- **출처** websocket GPS/PLC (TOS 아님)
- **갱신** API 메모리 스냅샷 → 화면 ~2.5초 폴링
- **방식** standalone — 각 TT를 5개 상태로 실시간 분류

**작업 풀 = 수요 · TOS · 90초 스냅샷**

- **출처** TOS 라이브 작업지시 2테이블
- **갱신** 추출기 90초 → Postgres → 화면 ~15초 폴링
- **방식** QC별 작업 시퀀스 + 배정 트럭 + ETW

:::note[왜 다른가]
차량 상태는 **직접 관측**됩니다(트럭이 GPS로 자기 상태를 송신) → 실시간이 쉬움. 작업 계획은 **TOS의 미래 의도**라 websocket에 없음 → DB에서 주기적으로 가져옴. 화면(Part 4)은 둘을 **융합**해 "계획 + 실측"을 한 번에 보여줍니다.
:::

## Part 1 · 차량 풀 갱신 — 실시간 (websocket)

API가 websocket 두 zone을 받아 **장비별 최신 위치 스냅샷을 메모리에 유지**하고, `GET /api/livemap/positions`로 노출합니다. 화면은 ~2.5초마다 폴링. (데이터 자체는 [websocket 문서](/kc/architecture/websocket-data/) 참조.)

### 신선도 — 메모리 스냅샷의 age-out

- **fresh** ≤ 15초 · **active(표시)** ≤ 120초 · **lost(제거)** 600초 초과 → 장비가 떠나도 자동으로 사라짐.
- DB에 안 쌓음 — 현재 순간만. 그래서 터널이 끊기면 즉시 빔(피드 헬스가 감시).

### 5개 배차 상태 (TT)

| 상태 | 뜻 · 화면 |
| --- | --- |
| `idle` 현재 유휴 | 공차 + 정지 → **즉시 배차 가능**. 배차 풀 1열. |
| `soon_idle` 곧 유휴 | 적재 + 마지막 핸드오버 진행 중 → 곧 풀림. 배차 풀 2열. |
| `empty_travel` 공차 주행 | 공차로 픽업 향함. 목적지 멀면(≥150m) **스왑 후보**. 배차 풀 3열. |
| `delivering` 적재 이동 | 적재하고 이동 중(아직 도착 전). |
| `wait_rtg` 도착·RTG 대기 | 블록 도착했으나 RTG 미관여 → 아직 대기. |

"실시간 배차 풀" 카드는 이 분류로 **현재 유휴 / 곧 유휴 / 스왑 가능 공차** 3열을 채웁니다. 분류 로직은 Part 2.

## Part 2 · 배차 상태 분류 상세

각 TT를 매 스냅샷마다 아래 결정 트리로 분류합니다. **websocket 신호만** 사용(standalone). 상수: 유휴속도 **3km/h** · RTG 같은bay **30m** · 스왑 최소거리 **150m**.

```
적재 여부 = container1 이 비었나?
공차(빈 차)
  ├ 속도 < 3km/h ───────────────▶ idle            (즉시 배차 가능)
  └ 이동 중 ─────────────────────▶ empty_travel    (목적지까지 거리로 스왑성 판정)
적재
  ├ arrival ≠ ARRIVED ──────────▶ delivering      (아직 운반 중)
  └ ARRIVED + 드롭 사이드 도착
      ├ 안벽(LD): QC PLC 신선 ──▶ soon_idle       (크레인 관여 확인)
      └ 블록(DS/MO/MI):
          ├ RTG GPS ≤ 30m ─────▶ soon_idle       (같은 bay = 핸드오버 중)
          └ RTG 멀음 ──────────▶ wait_rtg        (도착 ≠ 곧유휴)
```

### 드롭 사이드 — 어디서 풀리나

작업종류로 "이 차가 어디서 짐을 내려놓고 비는가"를 정합니다. **LD(적하)는 안벽(QC)에서** 내리고, **DS·MO·MI는 블록(RTG)에서** 내립니다. 적재 차가 *반대쪽*에 도착한 거면 방금 **픽업**한 것이라 아직 delivering.

:::caution[양하(DS)가 어려운 이유 — RTG엔 PLC가 없다]
적하는 QC PLC 하중 전이로 핸드오버를 ±1초로 잡습니다. 하지만 양하 블록측 RTG는 PLC가 없어, **RTG GPS와 TT GPS가 같은 bay(≈30m)인지**로 "곧 유휴"를 판정합니다. 블록에 *도착만* 했다고 곧유휴로 보면 오판 — RTG가 100m+ 떨어져 대기 중인 경우가 흔합니다(`wait_rtg`).
:::

### 블록 위치 학습 (centroid)

크레인이 항상 GPS를 쏘지 않으므로, **ARRIVED TT들의 GPS로 블록/베이 위치를 학습**합니다(코드별 이동평균, 상한 500). 그래서 공차 TT가 픽업 목적지까지 **몇 m 남았는지**를 TOS/레이아웃 의존 없이 추정 → 스왑성 판정에 사용.

### 스왑 가능 공차

공차 주행 중이라도 **목적지에 너무 가까우면(150m 미만) 스왑 부적합**(곧 픽업하니 재배정 무의미). 목적지까지 **150m 이상** 남은 공차만 스왑 후보로 띄웁니다. 목적지(topos1)가 없으면 회송으로 보고 제외.

## Part 3 · 작업 풀 갱신 — TOS 90초 스냅샷

작업 계획은 websocket에 없으므로 **TOS 라이브 작업지시**를 90초마다 끌어옵니다. 두 테이블(컬럼 상세는 [TOS 레퍼런스 §3.5](/kc/architecture/tos-db-reference/)):

- **`JOB_ORDER_LIST`** — 개별 컨테이너 move — 배정 트럭(YTNO)·ETW(크레인 준비시각)·컨테이너·RTG. **JOBSTATUS A(활성)만** 가져옴(완료행도 보관되므로 라이브 필터 필수).
- **`JOB_QUEUE_SCHEDULE`** — QC별 작업 큐 계획 — 크레인(C##)·순서(SEQ)·양/적·진행률(완료/총). **잔여 = TOTAL−COMP**.

:::note[JOBSTATUS 코드]
`A` 활성(배차됨·진행중) · `Q` 대기(백로그) · `P` 계획 · `B` 차단 · `C` 완료. 화면 카드 = **A**(ETW·배정 트럭이 채워진 진행 작업).
:::

### 파이프라인 (90초)

| 단계 | 내용 |
| --- | --- |
| ① TOS Oracle | 라이브 작업지시 2테이블. 추출기만 읽기. |
| ② 추출기 90s | `extractor workpool`. bounded 2쿼리, ~1–3s. |
| ③ Postgres | `live_workqueue` + `live_workpool`(통째 교체). |
| ④ API | `/api/workpool` — QC별로 묶어 반환. |
| ⑤ 화면 | QC 시퀀스 + 작업 프론트(~15s 폴링). |

:::note[조인은 Postgres에서 (Oracle 아님)]
큐 이름(예 `02D-L`)이 항차마다 재사용돼 Oracle에서 조인하면 과거 큐와 fan-out. 그래서 활성 move만 평범히 뽑고, QC는 **깨끗한 현재 큐 스냅샷**으로 Postgres에서 붙입니다. ETW는 MYT→UTC 파싱.
:::

## Part 4 · 두 풀의 융합 — QC 작업 시퀀스

화면의 "QC 작업 시퀀스 & 배차(라이브)" 패널은 작업 풀(TOS 계획)을 차량 풀/websocket(실측)과 합칩니다. **QC당 한 컬럼**, 세 신호의 융합:

| 신호 | 출처 | 화면 표현 |
| --- | --- | --- |
| TOS 계획 | 작업 풀 (90초) | 무엇을·어느 순서로·어느 트럭에·언제(ETW). 작업 카드. |
| PLC 실측 | websocket ctab | 그 크레인이 지금 도는 중이면 헤더에 `PLC live`. |
| 트럭 GPS | 차량 풀 | 배정 트럭(YTNO) 옆에 그 차의 **실시간 상태 점**(유휴/곧유휴/적재이동/RTG대기). |

연결 고리: TOS의 `YTNO`(예 TT1153) ↔ GPS device id, TOS의 QC(C##) ↔ PLC crane id. 그래서 "이 작업에 배정된 트럭이 지금 실제로 어디서 무엇을 하는지"가 한 줄에 보입니다.

:::tip[실데이터 검증]
화면의 트럭 배정·컨테이너·ETW는 **전부 TOS 실데이터**, 트럭 상태 점은 **실시간 GPS**. 가짜 없음. (실측: 활성 작업 배정 트럭 94대 중 67대가 그 순간 라이브 GPS에도 일치, 2026-06-09.)
:::

## Part 5 · 한계 · 노트

| 항목 | 내용 |
| --- | --- |
| 활성 작업은 전부 배정됨 | TOS의 `A`(활성)는 "트럭이 이미 배차된 진행 작업". "미배차 수요"는 대기(Q) 백로그이고, 깊이는 큐의 **잔여 = TOTAL−COMP**로 표시. |
| 대기(Q)는 개별 카드 없음 | 대기 move는 ETW·순서·배정이 없는 묶음 → 큐 단위 잔여 수치로만. (추출도 활성만) |
| ETW "overdue" | 활성=진행 중 핸드오버라 크레인 준비시각이 직전 과거 → 정상. |
| 차량 풀 standalone | 차량 분류는 websocket만으로(TOS 무관). 곧유휴 LD는 ~1분 전 사전 판정 가능, DS는 RTG 근접 확인 후. |
| 다음 단계 (2단계) | 지금은 TOS의 ETW·배정을 그대로 표시. 2단계엔 두 풀을 입력으로 **AI 전역 매칭**이 더 나은 배차를 제안 — [예측 모형 연구](/kc/research/tt-prediction/). |
