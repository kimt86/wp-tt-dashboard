WP-TT GPS/PLC 피드가 실제로 보내는 **모든 필드**를, 원본 프레임을 떠서 구현자 관점으로 정리합니다 — 각 필드가 언제 실리고, 무슨 의미이며, 구현하며 무엇을 관찰했는지. 상위 개요는 [실시간 websocket 데이터](/kc/architecture/websocket-data/)를 참고하세요. 이 문서는 그 아래의 "필드 사전"입니다.

## 1 · 봉투 & 두 개의 존

모든 프레임은 같은 봉투에 담겨 오고, `zone`으로 두 피드가 갈립니다.

```json
{"data":{"id":"TT1045","zone":"wpt_gps","datas":"<문자열로 인코딩된 JSON>"}}
```

- `data.id` — 장비 ID (우리 `id`). `data.zone` — 피드 종류. `data.datas`는 **문자열**이라 한 번 더 JSON 파싱한다.
- **`wpt_gps` 존** → 안에 `gps_update` (차량·장비 GPS, 장비당 ~3초). 본 문서 3~7장.
- **`ctab` 존** → 안에 `plc_data` (크레인 PLC, ~1초). 8장.

핸드셰이크: `{"command":{"identify":"…","zone":"wpt_gps"}}` → 2초 → `{"checkin":{"username":"…","user":"…"}}`. 존 구독형 브로드캐스트라, 체크인한 클라이언트는 그 존의 모든 장비 업데이트를 받습니다.

## 2 · 전체 26개 키 — 실제 프레임

한 프레임에 **항상 26개 키가 모두 포함**됩니다. 다만 해당 장비/상황에 없는 값은 빈 문자열(또는 0)로 옵니다. 우리가 파싱하는 것과 무시하는 것이 섞여 있습니다.

### 활성 LD 트럭 (도착 상태)

```json
{ "device":"TT1253", "lat":"2.93590967", "lon":"101.29527217",
  "speed":"1kmh", "engine_on":"ON", "accuracy":"4", "distance":"0.0",
  "userid":"P18884<br/>(0147150166)", "batt":"0%", "nett":"TELE", "dtime":"21:08:40",
  "cur_loc":"WHARF_12", "addr":"WHARF_12", "lapse":20,
  "arrival":"ARRIVED", "arr_dtime":"20:48:19",
  "container1":"CMAU8919605", "topos1":"C14", "lapse1":20,
  "container2":"", "topos2":"", "lapse2":20,
  "vslname":"CNC URANUS", "jobtype":"LD", "ins":"", "mds":"Y", "fuel_level":"32.00" }
```

### 유휴 크레인 장비 (미션 필드 전부 빈값)

```json
{ "device":"C54", … "cur_loc":"WHARF_23_A", "addr":"WHARF_23_A",
  "arrival":"", "container1":"", "topos1":"", "container2":"", "topos2":"",
  "vslname":"", "jobtype":"", "lapse":"", "ins":"", "mds":"", "fuel_level":"" }
```

:::note[관찰]
약 1,660 프레임을 28초간 떠보니 26개 키가 매 프레임 존재. 미션 필드(4장)는 대부분 빈값이고 소수에만 값이 실립니다(예: `topos2` 8.3%). 이게 "할당 이벤트성"의 직접 증거입니다(9장).
:::

## 3 · 정체성·위치·모션 — 항상 실림 (매 픽스)

| 필드 | 예시 | 의미 · 구현하며 관찰한 것 |
| --- | --- | --- |
| `device` | TT1253 | 장비 ID(불변). 앞 알파벳이 종류 — TT(트럭)/RTG/C·M·Z(크레인). 우리 `id`. |
| `lat / lon` | 2.9359 / 101.2952 | 위치. **문자열**로 옴. `0,0`이면 무픽스 → 버림. |
| `speed` | "1kmh" | 속도, 끝의 `kmh` 떼고 파싱. **&lt;3km/h = 정지** 판정. |
| `engine_on` | "ON" / "" | 시동. 켜졌는데 공차+정지 = 유휴 후보. |
| `accuracy` | "4" | GPS 정확도(m). 스파이크 시 위치 점프 → 거리 누적에 600m 점프가드. |
| `dtime` | "21:08:40" | 장비 로컬 시각. 우리는 수신 서버시각을 쓰고 이건 부가표시. |
| `distance` | "0.0" | ⚠️ **항상 0.0 — 무용**. 우리가 GPS로 직접 경로거리를 누적한다. |
| `userid` | P18884&lt;br/&gt;(…) | 운전자 ID+전화. **HTML `<br/>`이 박혀 옴** → `clean_driver`로 정제. |
| `batt` | "0%" / "100%" | 단말 배터리. 트럭은 0%로 자주 옴(센서 미연결 추정). |
| `nett` | "TELE" / "GLOG" | 통신망 라벨. 피드 헬스 참고. |
| `fuel_level` | "32.00" | 연료 %. 부가표시. |

## 4 · 미션/작업 필드 (할당 이벤트성) ⚠️ 핵심이자 함정

이 필드들은 **3초 하트비트에 안 실립니다. 작업 할당 *이벤트* 때만 한 번 실리고, 다음 픽스부턴 빈값**으로 옵니다. (메모리·문서에 스터디됨: "ride only on assignment events".) 그래서 빠진 값을 latch(직전 비-빈값 유지)해야 합니다.

| 필드 | 예시 | 의미 · 구현하며 관찰한 것 |
| --- | --- | --- |
| `topos1` | C14 / 03U-21 | **목적지(다음 핸드오버 지점)** — 블록 베이 또는 크레인(C/M/Z+숫자). **= 작업 할당 시점 신호.** 비었다가 잡히는 순간이 그 작업 배정. DS: 크레인(픽업)→블록(드롭), LD: 블록(픽업)→크레인(드롭). 할당 직후엔 아직 100m+ 떨어져 있기도(타깃이지 도착이 아님). |
| `container1` | CMAU8919605 | 1번 박스 번호. ⚠️ **물리 적재가 아니라 TOS 배정 필드** — 직전 drop 시점에 다음 박스가 미리 배정됨(c2c). 비-빈→다른값 + 이동≥150m = 진짜 인도. |
| `container2` | (트윈일 때) | 2번 슬롯 — **트윈/탠덤(20ft 2개)**일 때 채워짐. c1↔c2 슬롯 스왑이 잦음(가짜 인도 유발 → {c1,c2} 집합 동일하면 무시). c1==c2면 40ft 1개를 두 슬롯에 표시한 것. |
| `topos2` | M7 / 08Q-21 | **2번째 핸드오버 지점** — 트윈/탠덤의 두 번째 박스 목적지. 상세는 [5장](#5--topos2-심층-조사). |
| `jobtype` | LD / DS / MI·MO·LC | 적하(LD)/양하(DS)/야드이동(MI·MO·LC). **픽업·드롭이 어느 쪽인지 결정**: LD=블록픽업·크레인드롭, DS=반대. latch. |
| `vslname` | CNC URANUS | 작업 대상 선박명. latch. |
| `cur_loc` | WHARF_12 / 01U | **현재 위치 라벨**. topos1(목적지)과 같아지면 도착. (이건 비교적 자주 실림 — 위치성.) |

:::caution[함의 — 단계 감지의 방향]
topos1이 "비었다가 잡히는 전이" = 할당 이벤트이므로, 작업 사이클의 단계 경계를 약한 `arrival` 플래그가 아니라 **topos1 전이 + GPS 근접 + PLC**로 잡는 것이 더 정확합니다. 상세는 [사이클 감지 로직](/kc/architecture/cycle-detection-v1/).
:::

## 5 · topos2 심층 조사

"topos2에 값이 실제로 실리나? 언제?"를 원본 프레임으로 직접 조사한 결과입니다.

### 실리는 빈도

- 키는 **매 프레임 존재**하지만, 값이 차는 건 **1,660 프레임 중 138개 = 8.3%**. topos1처럼 **할당 이벤트성**(평소 빈값).
- 값의 형태: 크레인 코드(`C34`·`M7`·`Z7`)와 블록 베이(`08Q-21`·`04L-03`) 둘 다.

### 무슨 의미인가 — 트윈/탠덤의 2번째 목적지

값이 찰 때 `container2`도 거의 함께 차고, 작업유형/topos1과의 조합이 트윈 다중경유를 그대로 보여줍니다:

| jobtype | topos1 | topos2 | 해석 |
| --- | --- | --- | --- |
| LD | 02W-41 | 08Q-21 | 서로 **다른 두 블록**에서 수령 (트윈 컬렉션) |
| LD | 11X-20 | M7 | 블록 픽업 + 크레인(M7) — 박스1 블록, 그 다음 크레인 |
| LD | C34 | C34 | 둘 다 같은 크레인 — 두 박스 같은 QC로 |
| MO | 04L-15 | 04L-03 | 야드이동, 인접한 두 베이 |

:::tip[결론]
`topos2` = **두 컨테이너를 다루는 작업(트윈/탠덤)의 두 번째 핸드오버 지점.** topos1과 짝을 이뤄 트럭의 다중 경유(블록→블록→안벽, 또는 안벽→블록→블록)를 표현한다. 단일 컨테이너 작업에선 빈값. 현재 우리는 topos1만 파싱하지만, 트윈의 두 번째 leg를 정확히 잡으려면 topos2를 함께 보면 된다.
:::

## 6 · 도착·체류 — arrival · arr_dtime · lapse

"arrival(도착) 같은 상태도 데이터로 오나?" → 네. 도착 플래그 + 도착 시각 + 체류 타이머가 한 묶음으로 옵니다.

| 필드 | 예시 | 의미 · 구현하며 관찰한 것 |
| --- | --- | --- |
| `arrival` | "ARRIVED" / "" | 트럭이 목적지(topos1) 핸드오버 지점에 도달하면 `"ARRIVED"` 문자열, 아니면 빈값. ⚠️ **블록에선 신뢰성 있음(정차 트럭 ~78%), 크레인에선 약함(~23%)** — 크레인 레인이 길고 지오펜스가 느슨해 크레인 쪽 단계가 잘 안 잡힘. |
| `arr_dtime` | "20:48:19" | **도착(ARRIVED)된 시각.** arrival이 켜진 순간의 타임스탬프. |
| `lapse` | 20 (숫자) | **도착 후 경과 분 = 체류 타이머.** `dtime − arr_dtime`과 일치 검증됨(예: 20:48 도착, 21:08 현재 → lapse 20). **핸드오버 대기시간을 서버가 직접 계산해 보내준다** — QC 대기/체류 분석에 바로 쓸 수 있는 값. |
| `lapse1 / lapse2` | 20 / 20 | topos1/topos2 각각의 체류 분(짝). 관측상 `lapse`와 같은 값으로 옴. |

:::note[활용 여지]
`lapse`는 우리가 따로 계산하던 "도착 후 대기 시간"을 피드가 이미 분 단위로 주는 것입니다. 사이클의 핸드오버(받기/주기) 체류를 보강하거나 QC 대기 판정에 교차검증으로 쓸 수 있습니다.
:::

## 7 · 보조·미해독 필드 (현재 미사용)

| 필드 | 관측값 | 해석 |
| --- | --- | --- |
| `addr` | WHARF_12, 02G… | **현재 위치 주소 라벨.** 활성 샘플 전부에서 `cur_loc`과 동일하게 옴 → cur_loc의 다른 표기로 보임. 272종 관측. |
| `mds` | "Y" | 활성 트럭에서 항상 `"Y"`. 의미 미상(모드/유효 플래그 추정). 변별력 없어 미사용. |
| `ins` | "" (항상 빈값) | 관측 구간 내내 빈값. 의미 미상, 미사용. |

:::caution[정직한 한계]
`ins`·`mds`는 이번 조사 구간(야간)에서 변별값이 없어 의미를 확정하지 못했습니다. 다른 시간대/작업에서 값이 다르게 실릴 수 있으니, 추정으로 단정하지 않고 "미해독"으로 남깁니다.
:::

## 8 · ctab — 크레인 PLC (`plc_data`, ~1초)

두 번째 존. 크레인의 스프레더 상태를 1초 해상도로 보냅니다.

| 필드 | 예시 | 의미 · 구현하며 관찰한 것 |
| --- | --- | --- |
| `crane` | C46 | 크레인 ID — GPS device id·TOS QC번호와 일치 → 위치/계획 연결고리. |
| `load` | (톤, 숫자) | 훅 하중. 빈후크 ~0, 적재 ≥1t. **empty→laden 상승엣지 = PICKUP(무브 1건).** "크레인이 지금 작업 중"의 직접 신호 → 라이브 move/hr·QC대기에 사용. |
| `lock / land` | "True"/"False" | 트위스트락 / 안착(문자열 불리언). |
| `hpos / tpos` | (m) | 호이스트(들어올림) / 트롤리(횡행) 위치. |

## 9 · 피드 거동 규칙 & 구현 함의

1. **두 속도**: GPS ~3초/장비, PLC ~1초/크레인. 과거 없음(현재 순간만, DB 미적재) — 그래서 사이클은 우리가 직접 누적·영속화한다.
2. **미션 필드는 할당 이벤트성**(4장): 할당 때만 실리고 평소엔 빔. ⚠️ GPS 스냅샷으로 "배정 수"를 세면 안 됨(한때 160 배정인데 18로 폭락한 사고) → 배정은 작업풀(TOS)로 센다.
3. **수신부가 매 메시지 상태를 새로 만듦** → 빠진 필드는 None. 그래서 carry-forward/**latch**가 필수.
4. **잡 신호로 쓸 만한 것**: 위치/모션(항상), `arrival`+`arr_dtime`+`lapse`(도착·체류), PLC `load`(크레인 작업). 미션 필드는 전이(이벤트)로 해석.
5. **age 밴드**: ≤15s fresh · ≤120s active(맵 서빙) · &gt;600s 제거.

이 레퍼런스는 [사이클 감지 로직](/kc/architecture/cycle-detection-v1/)의 토대입니다 — 어떤 필드가 신뢰할 만하고(위치·PLC·블록 ARRIVED), 어떤 게 함정인지(container1=배정, 크레인 ARRIVED 약함)를 알아야 단계 경계를 옳게 설계할 수 있습니다.
